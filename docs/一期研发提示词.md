## 开发前必读卡

- **文件定位**：Codex 逐轮投喂执行手册，覆盖 Prompt 0–14 的目标、禁止事项、输出和验收。
- **实现约束**：每轮只做当前 Prompt 的目标；通用边界、统一响应、trace、审计和失败处理规则全轮次继承。
- **禁止越界点**：禁止跨 Prompt 偷跑后续功能；Prompt 中出现后续能力不代表当前轮可以实现。
- **下游影响**：约束每轮开发输入、修复流程、验收命令、PR 检查和开发日志更新。
- **自动化验收关注点**：代码能跑、测试通、没越界；每轮至少有可自动化 DoD。

**统一硬边界提醒**：人才服务区仅展示发布快照；对接接口仅 stub；不得输出原始候选人数据；同意、撤回、数据域、字段级权限必须在服务端收口。

## 执行摘要

本手册面向 PM/Tech Lead/Dev 团队，提供**从零到可交付**的一期（Phase‑1）Codex 逐轮投喂方法：共 15 轮（Prompt 0–14），每轮固定五段结构（主控说明 / 本轮目标 / 禁止事项 / 输出要求 / 验收要求），并将三条闸门固化为可自动化 DoD：**代码能跑、测试通、没越界**。

一期的工程风险重点是：**访问控制失效**（OWASP Top 10 A01）与**安全日志/监控失效**（A09），必须做到“服务端强制 + 可审计”。citeturn2search7turn1search5 个人信息合规方面，“**同意**与**撤回**”必须闭环：法律要求提供便捷撤回同意方式。citeturn3search7turn3search8

一期范围硬边界（所有轮次默认继承；任何违反即退回重做）

- **仅**实现“地方人才服务平台一期”；**不**实现 ATS/受控候选池；仅保留**对接接口** stub（不得直连外部原始主库）。
- **人才服务区**仅展示**发布快照**；不得展示或返回任何**原始候选人数据**（联系方式、简历正文、附件、聊天记录、证据材料等）。
- 权限必须服务端强制：**RBAC + 数据域 + 字段级权限**；导出为高风险动作必须审批。
- 必须实现**同意→发布快照→展示**与**撤回→下线**闭环，并提供“便捷撤回方式”。citeturn3search7

本手册采用的默认建议（你已确认全部采用）

- **认证体系（SSO/OIDC）**：**不接** SSO/OIDC；一期默认“本地账号 + JWT”。
- **容量指标（QPS/导出频次/快照规模）**：默认分页；导出异步；预留限流位；对象存储承载导出文件。
- **部署形态（Docker/K8s）**：一期默认 **docker-compose**；K8s 仅作为可选扩展（若上 K8s，按官方探针配置）。citeturn3search4
- **对象存储**：默认 MinIO；导出/下载使用预签名 URL（短期有效）。citeturn3search1
- **trace**：默认 X-Trace-Id；对接接口可兼容 W3C Trace Context（traceparent）。citeturn1search1

## 运营模型与一轮的标准闭环

仓库“运营文件”清单

- `docs/00_MASTER_BOUNDARY.md`：一期宪法（边界冻结 + 数据边界 + 权限审计强制）。
- `docs/90_DEV_LOG.md`：开发日志（每轮必须记录）。
- `docs/91_ACCEPTANCE_CHECKLIST.md`：验收清单（每轮至少新增/完善 1 条自动化 DoD）。
- `scripts/`：闸门脚本集合（越界扫描、字段黑名单、迁移校验、全量测试）。

一轮标准闭环（你按这个机械执行即可）

1. **准备**：新建分支 → 确认本轮只做一个目标。
2. **投喂执行**：把“本轮最小投喂片段”贴给 Codex，让它生成变更。
3. **投喂校验**：用“校验提示词模板”让 Codex 自检：有没有越界、漏测试、漏审计、漏字段裁剪。
4. **本地跑命令**：按本轮给出的 start/test/migration 命令跑一遍。
5. **失败处理**：按“失败情形→修复提示词”执行修复，再回到第 3/4 步。
6. **提交 PR**：填 PR 模板 + Checklist，通过后合并。
7. **复盘**：写 docs/90_DEV_LOG.md 复盘；更新验收清单。

校验提示词模板（每轮 Codex 写完代码后必投喂）

```text
请对比 docs/00_MASTER_BOUNDARY.md 与 Step1–Step4：
1) 本轮是否越界（ATS/受控候选池、人才服务区是否仅展示发布快照、是否输出原始候选人数据）？
2) 是否新增/修改了哪些文件（按文件树列出）？
3) 缺失的自动化验收项是什么（至少 3 条）？
4) 审计与字段级权限是否落实在服务端（列出落点）？
5) 给出“必须修复清单”（按阻断级别排序）。
```

修复提示词模板（当出现编译失败/测试失败/越界时使用）

```text
你现在是“修复工程师”。请只做最小修复：
- 目标：让本轮命令全部通过（编译、测试、迁移/脚本闸门）。
- 约束：不得引入新需求；不得越界（ATS/受控候选池；人才服务区仅展示发布快照；不输出原始候选人数据）。
- 输出：1) 修复的文件列表 2) 变更原因 3) 修复后需要我运行的命令清单。
```
