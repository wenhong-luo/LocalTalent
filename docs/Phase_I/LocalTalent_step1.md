## 一、一期冻结边界声明

### 必须执行的冻结边界

- 一期仅建设地方人才服务平台，不建设 ATS 系统，不建设受控候选池系统。
- 一期仅允许预留对接接口、字段兼容位、数据库结构兼容位，不得以任何形式提前落地 ATS 与受控候选池业务能力。
- 一期禁止直接接入 ATS 原始候选人数据。
- 一期禁止直接接入受控候选池原始主库。
- 一期人才服务区仅展示候选人发布快照，不展示原始简历、联系方式、聊天记录、附件、证据材料。

### 冻结边界的设计含义

- 平台先独立完成职位、企业、求职者、活动、签到、同意、发布控制等闭环。
- 后续若接入 ATS 或受控候选池，应通过对接接口、状态回调、发布快照同步等方式增强，而不是提前打穿原始主库。
- 任何试图把“一期预留端口”升级成“一期提前落地功能”的行为，均视为范围漂移。

本节为强约束，不属于建议项。后续步骤二、步骤三、步骤四的所有设计，均不得突破本节冻结边界。

## 二、项目目标 / 输入 / 输出 / 不做范围

项目目标

完成地方人才服务平台的一期建设

一期输出

人才网前台 + 企业后台 + 求职者中心 + 运营后台

必须预留

ATS 接口位 / 受控候选池接口位 / 审计链路

当前不做

ATS / 受控候选池正式实施与正式接入

后续基线

平台先独立可运行，再平滑增强

本项目应被定义为“一个可独立运营的人才服务平台”，而不是“ATS 前端皮肤”或“候选池展示页”。因此当前建设必须完成完整的人才网基础能力，同时为未来两系统预留标准接口与数据库兼容位。

## 三、主体与系统边界

| 层级 | 对象 | 边界定义 |
| --- | --- | --- |
| **主体 A** | 高风险主处理主体 | 未来负责 ATS 中台、受控候选池、候选人授权与更高风险的数据处理能力。当前阶段不实施系统建设，仅允许保留接口与字段兼容表达。 |
| **主体 B** | 地方人才服务平台主体 | 当前阶段唯一启动的建设主体。负责职位、企业、求职者、活动、投递、签到、同意、人才服务区、运营后台等能力。 |
| **系统 1** | 地方人才服务平台 | 当前正式建设对象，必须独立可运行。 |
| **系统 2** | ATS 系统 | 当前不实施，仅预留职位同步、投递同步、面试安排同步、授权回传接口位。 |
| **系统 3** | 受控候选池系统 | 当前不实施，仅预留发布快照同步、发布撤回联动、映射关系接口位。 |

方案 B 的核心不变：两个主体 + 三个系统。当前只启动系统 1；系统 2 和系统 3 一律视为“后续增强对象”，不得提前落地为一期功能。

## 四、关键术语统一说明

### 人才服务区

前台公开展示候选人发布快照的业务区，不是公共简历库。

### 发布快照

满足发布条件后进入公开展示链路的最小必要候选人展示副本。

### 原始候选人数据

原始简历、联系方式、聊天记录、附件、证据材料及其他未进入公开展示链路的数据。

### 对接接口

面向 ATS 与受控候选池预留的接口能力，一期只定义，不正式启用业务接入。

### 同意

候选人完成明确授权并处于有效状态的业务结果，是进入发布快照展示链路的前置条件之一。

### 撤回

候选人取消原同意状态的动作，发生后应触发停止展示、下架联动与审计留痕。

### 数据域

控制“谁能看哪一批数据、哪一个企业、哪一个候选人范围”的边界规则。

### 字段级权限

控制“在已允许查看的数据内，哪些字段可见、脱敏或隐藏”的边界规则。

人才服务区的展示边界在本稿中写死：人才服务区只展示发布快照，不展示原始候选人数据；同意、撤回、数据域、字段级权限的后续实现口径应与步骤二至步骤四保持一致。

## 五、平台产品定位与建设目标

### 平台定位

- 面向地方企业、求职者、活动组织方、运营方的地方人才服务平台。
- 既承接基础招聘信息服务，也承接活动、签到、人才服务等能力。
- 当前可独立运营，不依赖 ATS 与受控候选池即可落地。
- 未来可接入 ATS 提供职位、面试、授权等增强能力，可接入受控候选池提供发布快照增强能力。

### 建设目标

- 建设一个功能较完整、可独立运行、可审计、可扩展的人才服务平台。
- 具备职位、企业、求职者、投递、活动、签到、同意、发布控制等完整闭环。
- 具备审计留痕、可见范围控制、展示边界控制能力。
- 为后续接入 ATS 与受控候选池保留响应端口、字段位、数据库兼容位。

## 六、一期功能范围

### 一期必做

- 人才网前台：首页、职位列表、职位详情、企业列表、企业详情、活动列表、活动详情、政策资讯、人才服务区。
- 企业后台：企业入驻、认证、职位发布、职位管理、投递查看、活动报名管理。
- 求职者中心：注册登录、简历管理、投递记录、活动报名、面试通知、签到记录。
- 运营后台：企业审核、职位审核、活动管理、内容管理、统计看板、审计日志查看。
- 面试签到与同意中心：签到码、签到记录、同意状态展示、撤回入口预留。
- 权限中心、日志中心、审计中心。

### 一期只预留端口，不落地实现

- ATS：职位同步、投递同步、面试安排同步、授权回传。
- 受控候选池：候选人受控发布、候选人一键同步、发布撤回联动。
- 复杂画像、复杂推荐、复杂标签体系。
- 公共简历库深度运营能力。
- 跨平台复杂同步编排。

“只预留端口，不落地实现”在一期中属于冻结性范围控制，不得在开发过程中被理解成“可先做一部分”。

一期设计原则：平台自身先具备独立可运营的业务闭环；未来接入 ATS 与受控候选池时，采用“增强模式”接入，而不是“重构模式”接入。

## 九、业务规则与数据控制规则

| 规则主题 | 规则内容 |
| --- | --- |
| **职位公开规则** | 职位信息可公开展示，但须经过企业与运营规则校验；职位同步到平台后不涉及候选人数据。 |
| **候选人展示规则** | 候选人不得默认进入人才服务区；只有满足 consent_status=同意 且 publishable_flag=1 时，才允许生成并展示候选人发布快照。 |
| **人才服务区规则** | 人才服务区只展示候选人发布快照，不展示原始简历、联系方式、聊天记录、附件、证据材料。 |
| **撤回规则** | 候选人撤回同意后，平台必须停止继续展示，并记录 revoke_status 与 revoke_time；未来接入受控候选池时要能触发联动下架。 |
| **企业可见规则** | 企业默认仅可见本企业相关数据；未来接入候选池后，也必须遵循 visibility_scope、source_type、consent_status 的联合控制。 |
| **运营看板规则** | 默认展示聚合统计，不展示未经授权的候选人原始明细。 |

## 十一、数据库设计总原则

### 原则 1

平台数据库当前可独立支撑人才服务平台运行，不依赖未来系统存在。

### 原则 2

主表必须预留 source_type、legal_basis、consent_status、publishable_flag、visibility_scope 等未来集成字段。

### 原则 3

平台不设计成“直连 ATS 或受控候选池原始主库”，而设计成“发布快照表 + 同意回执表 + 同步日志表”的兼容模式。

## 十三、日志、审计与可追溯要求

### 日志中心

记录登录、发布、审核、投递、签到、同意、撤回、同步、导出等行为。

### 审计中心

重点保留候选人展示控制、同意变更、导出审批、字段访问日志。

### 追溯能力

要求任何一个候选人发布结果，均能追溯到来源、同意版本、操作人、时间和同步链路。

## 十四、建议研发交付顺序

| 阶段 | 交付内容 |
| --- | --- |
| **第一阶段** | 权限中心、日志中心、审计中心、基础字典、企业认证、职位主流程。 |
| **第二阶段** | 求职者中心、简历管理、投递流程、活动模块。 |
| **第三阶段** | 面试签到、同意中心、展示控制、撤回控制。 |
| **第四阶段** | 对接接口、同步任务、外部映射、运营看板。 |

建议先做“控制底座”和“主流程”，再做“同意展示链路”，最后做“外部系统预留接口”。这样能保证平台先独立跑起来，再平滑增强。

## 十五、法律与合规设计依据（用于方案解释）

| 规则依据 | 本方案中的设计含义 |
| --- | --- |
| **个人信息向其他处理者提供需告知并取得单独同意** | 因此“候选人同意后展示/同步”必须单独设计同意接口、同意范围、接收方与版本记录，而不能把展示视为默认动作。 |
| **向其他网络数据处理者提供、委托处理个人信息应通过合同约定并保存处理记录** | 因此未来平台与 ATS / 受控候选池对接时，不能只做技术接口，还要保留接口日志、审计记录和合同化管理的基础条件。 |
| **可靠电子签名与手写签名、盖章具有同等法律效力** | 第一期可采用“实名核验 + 勾选同意 + 二次确认 + 快照 + 审计”组合方案，后续如需更强证明力，可升级对接可靠电子签署。 |

参考口径：官方个人信息保护、网络数据安全及电子签名相关规则，以及公共招聘服务实践中的公开招聘与授权边界要求。

## 十六、第一步骤修订版结论

### 结论 1

方案 B：两个主体 + 三个系统继续成立，主体方向不变。

### 结论 2

当前只启动地方人才服务平台，ATS 与受控候选池一律视为后续增强对象。

### 结论 3

一期冻结边界已经写死：禁止直接接入 ATS 原始候选人数据，禁止直接接入受控候选池原始主库，人才服务区只展示候选人发布快照。

本稿形成后，步骤二、步骤三、步骤四的所有后续设计均应建立在本稿边界之上，不得再重新放宽一期范围。
