第一期禁止项与数据边界

硬禁止项
一期禁止直接接入 ATS 原始候选人数据；一期禁止直接接入受控候选池原始主库；一期仅允许预留对接接口位与结构兼容位，不得提前落地两系统业务能力。人才服务区仅展示候选人发布快照，不展示原始简历、联系方式、聊天记录、附件、证据材料。fileciteturn1file0
两类数据产品必须隔离
原始候选人数据（用于业务处理与留痕）与发布快照（用于人才服务区公开展示）是两类不同数据产品：发布快照必须经过同意、裁剪/脱敏与可见范围控制；人才服务区仅读发布快照。fileciteturn1file1 fileciteturn1file3

一期 DoD（Definition of Done）总则

  • 范围 DoD：无任何“直连 ATS/受控候选池原始主库”的代码路径、配置项或数据同步任务；对接接口仅 stub。fileciteturn1file0
  • 数据边界 DoD：人才服务区任一接口与页面均不返回原始候选人数据字段；发布快照字段清单可审计且可回归测试。fileciteturn1file3
  • 权限 DoD：RBAC + 数据域 + 字段级权限在服务端强制执行；任何对象 ID 查询均做对象级授权校验（对应 OWASP API1:2023 BOLA 风险）。citeturn4search16
  • 审计 DoD：关键动作写 audit_log；敏感字段读取写 field_access_log；日志不记录敏感明文（对应 OWASP A09 & Logging Cheat Sheet）。citeturn1search5turn1search17
  • 合规 DoD：提供便捷撤回同意方式并生效；遵循合法、正当、必要、最小范围收集等原则（PIPL）。citeturn3view0

按 Step4 的分层原则实施：先字典与权限附加表，再核心业务表，再对接/审计/证据表，降低依赖与联调成本。fileciteturn1file3

部署与运维要点

环境与发布流程

  • 环境：dev / staging / prod 至少三套；staging 具备与 prod 一致的 DB 迁移与对象存储策略。
  • 发布顺序：先执行 DB 迁移（DDL-0..DDL-4）→ 再滚动发布应用 → 冒烟测试（关键链路）→ 指标观察。
  • 健康检查：若采用 K8s，应配置就绪/存活/启动探针，避免未就绪流量进入并支持自愈。citeturn1search3
  • 回滚策略：业务代码可回滚；DB 结构优先 roll-forward（通过新增迁移修正），生产避免直接回滚结构。

备份、监控告警与日志采集

  • 备份：DB 全量 + 增量（binlog/WAL）；定期恢复演练；审计库与证据元数据同等级保护。
  • 监控指标：接口 P95/P99、5xx、导出审批堆积、重试队列长度、DB 连接池、对象存储 4xx/5xx。
  • 日志采集：统一采集应用日志 + 审计日志；按 trace_id 关联；禁止记录敏感明文（对应 OWASP A09 与日志最佳实践）。citeturn1search5turn1search17

对象存储分层与短期下载策略

  • 分桶建议：resume（简历附件）、snapshot（发布快照资源）、evidence（同意证据）、export（导出文件）。
  • 短期下载：导出文件与证据下载使用短期预签名 URL；导出必须先经 export_apply 审批通过。fileciteturn1file3
