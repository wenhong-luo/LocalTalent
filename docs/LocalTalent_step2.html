术语统一

一-A、关键术语统一说明

人才服务区

前台公开展示“候选人发布快照”的业务区，不是公共简历库。

发布快照

满足发布条件后进入公开展示链路的最小必要候选人展示副本。

原始候选人数据

联系方式、简历正文、附件、证据材料及其他未进入公开展示链路的数据。

对接接口

一期只预留对接约束，不落地 ATS 或候选池正式接入与正式运营。

数据域

控制“谁能看哪一批数据、哪一个企业、哪一个候选人范围”的边界规则。

字段级权限

控制“在已获准查看的数据内，还能看哪些字段、看原值还是脱敏值”的边界规则。

本稿进一步明确：“原始候选人数据”和“发布快照”是两类不同的数据产品。前者用于业务处理与留痕，后者用于人才服务区公开展示。

候选人进入人才服务区平台生成并展示候选人发布快照脱敏姓名、城市、技能、经验年限、岗位方向等发布快照涉及个人信息；属于公开展示链路基于明确授权 + 发布控制规则否是是只展示发布快照，不展示原始候选人数据；撤回后联动下架
候选人撤回同意候选人主动撤回revoke_status、revoke_time、当前发布状态涉及同意状态变更撤回同意权利否不适用是撤回后必须停止继续发布；保留历史证据与审计轨迹
数据导出 / 报表导出企业或运营导出数据投递记录、候选人卡片、活动名单、统计数据可能涉及个人信息内部管理必需 + 审批授权否通常不以同意为主，而以权限与审批为主中到高导出审批、脱敏模板、下载留痕、超范围禁止导出
审计日志 / 字段访问日志系统记录查看、导出、发布、撤回等动作operator_id、biz_id、field_name、access_type、trace_id可能包含个人信息引用，但不应复制原始内容安全与审计管理所必需否否中日志白名单；原始正文禁入日志；日志单独保护并设置留存期
合规场景清单的核心结论：一期风险最高的不是职位公开，而是“候选人同意发布 → 进入人才服务区 → 撤回联动下架”这条链路。后续数据库与权限设计必须优先围绕这条链路收口。

审计与访问日志operator_id、field_name、access_type、trace_id、create_time审计中心一般不流转按审计与追责周期管理日志正文禁入原始简历全文与聊天全文只记录行为与对象，不复制整份个人信息正文

原始候选人数据的定义

  • 手机号、邮箱、微信号等直接联系方式
  • 完整简历原文、附件全文
  • 聊天记录、内部评价、内部标签
  • 证件照、实名核验结果、同意证据材料
  • 任何未经过 publishable_flag 判断的字段

可发布快照的定义

  • 脱敏姓名或匿名标识
  • 城市、行业方向、岗位方向
  • 经验年限、技能标签摘要
  • 经规则裁剪后的教育/经历摘要
  • publishable_flag=1 且 consent_status=有效的最小必要内容
候选人数据地图的核心结论：平台必须把“原始候选人数据”和“发布快照”当成两类不同的数据产品管理。前者用于业务处理与留痕，后者仅用于人才服务区公开展示。前者用于业务处理与留痕，后者仅用于人才服务区公开展示。

六、字段分级 / 接口字段 / 权限边界

S1 公开级

可公开展示的职位、企业、活动、资讯内容，以及候选人发布快照中的最小必要展示字段。

S2 业务级

平台内部业务流转字段，不可公开，但一般不属于高度敏感。

S3 受限级

候选人个人信息、简历正文、投递与面试相关个人数据。

S4 高敏级

同意证据、实名核验结果、导出材料、审计关键证据。
