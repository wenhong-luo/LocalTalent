# LocalTalent 服务器部署与自动更新方案

> 本文用于把 LocalTalent 部署到普通 Linux VPS / 云主机，并说明“本地提交后服务器是否能立刻更新”。当前推荐先采用 **源码部署 + 服务器构建 + systemd 重启**。三期仍是小范围灰度试运营候选，不等同于公网正式上线。

## 1. 结论先行

- 可以部署到服务器上。
- 本地 `git commit` 不会自动影响服务器；必须至少 `git push` 到远程仓库。
- 服务器要更新，需要执行 `git pull` 或由 webhook / CI 触发部署脚本。
- 前端是 Next.js SSR，生产运行依赖 `.next` 构建产物，所以正式部署必须重新 `npm run build`。
- 后端 Java 代码变更后也必须重新 `./mvnw clean package` 并重启服务。
- 不建议用 `npm run dev` 给领导或灰度环境看；开发模式接近“代码一变立即更新”，但不稳定，不代表试运营形态。

## 2. 推荐架构

```text
用户浏览器
  |
  v
Nginx :80/:443
  |-- /api/*  -> Spring Boot backend :8080
  |-- /health -> Spring Boot backend :8080
  `-- /*      -> Next.js frontend :3000

服务器本机：
- localtalent-backend.service
- localtalent-frontend.service
- Docker Compose: MySQL / Redis / MinIO
```

当前仓库已有：

- `infra/docker-compose.yml`：MySQL / Redis / MinIO。
- `frontend/next.config.ts`：`output: 'standalone'`，并将 `/api/*` rewrite 到后端。
- `frontend/package.json`：`npm run build`、`npm start`。
- `backend/pom.xml`：Spring Boot 可打包为 `target/backend-0.1.0-SNAPSHOT.jar`。
- `scripts/deploy_server_source`：服务器源码部署脚本。
- `deploy/systemd/*.service.example`：systemd 服务模板。
- `deploy/nginx/localtalent.conf.example`：Nginx 反向代理模板。

## 3. 服务器准备

建议版本：

| 组件 | 建议 |
| --- | --- |
| OS | Ubuntu 22.04/24.04 或同类 Linux |
| Node | Node 24 LTS |
| Java | JDK 21 |
| Docker | Docker Engine + Compose plugin |
| Nginx | 稳定版 |
| Git | 能访问 GitHub 仓库 |

安装后建议创建应用用户：

```bash
sudo useradd -r -m -d /opt/localtalent -s /bin/bash localtalent
sudo mkdir -p /opt/localtalent /etc/localtalent /var/log/localtalent
sudo chown -R localtalent:localtalent /opt/localtalent /var/log/localtalent
```

## 4. 首次部署

克隆仓库：

```bash
sudo -iu localtalent
git clone https://github.com/wenhong-luo/LocalTalent.git /opt/localtalent/app
cd /opt/localtalent/app
```

准备依赖：

```bash
docker compose -f infra/docker-compose.yml up -d
```

建议先复制环境模板，再按服务器真实情况修改：

```bash
sudo cp .env.phase3.example /etc/localtalent/backend.env
sudo cp .env.phase3.example /etc/localtalent/frontend.env
sudo chown root:root /etc/localtalent/*.env
sudo chmod 600 /etc/localtalent/*.env
```

生产/灰度环境必须至少改掉：

- `LOCALTALENT_DB_URL`
- `LOCALTALENT_DB_USERNAME`
- `LOCALTALENT_DB_PASSWORD`
- `LOCALTALENT_JWT_SECRET`
- `LOCALTALENT_MINIO_ENDPOINT`
- `LOCALTALENT_MINIO_ACCESS_KEY`
- `LOCALTALENT_MINIO_SECRET_KEY`
- `LOCALTALENT_OIDC_*`（如启用 OIDC）
- `LOCALTALENT_API_BASE_URL`
- `NEXT_PUBLIC_LOCALTALENT_SITE_URL`

首次构建：

```bash
cd /opt/localtalent/app
./scripts/check_boundary
./scripts/check_portal_snapshot_fields
./scripts/check_phase3_boundary
./scripts/check_phase3_ops
./scripts/check_phase3_security_acceptance
./scripts/check_phase3_gray_acceptance

cd backend
./mvnw clean package

cd ../frontend
npm ci
npm run build
```

安装 systemd 服务：

```bash
sudo cp deploy/systemd/localtalent-backend.service.example /etc/systemd/system/localtalent-backend.service
sudo cp deploy/systemd/localtalent-frontend.service.example /etc/systemd/system/localtalent-frontend.service
sudo systemctl daemon-reload
sudo systemctl enable --now localtalent-backend localtalent-frontend
```

安装 Nginx：

```bash
sudo cp deploy/nginx/localtalent.conf.example /etc/nginx/sites-available/localtalent.conf
sudo ln -s /etc/nginx/sites-available/localtalent.conf /etc/nginx/sites-enabled/localtalent.conf
sudo nginx -t
sudo systemctl reload nginx
```

## 5. 手动更新流程

```bash
cd /opt/localtalent/app
git pull --ff-only

./scripts/check_boundary
./scripts/check_portal_snapshot_fields
./scripts/check_phase3_boundary
./scripts/check_phase3_ops
./scripts/check_phase3_security_acceptance
./scripts/check_phase3_gray_acceptance

cd backend
./mvnw clean package

cd ../frontend
npm ci
npm run build

sudo systemctl restart localtalent-backend
sudo systemctl restart localtalent-frontend
```

## 6. 推荐自动更新：push 后部署

标准链路：

```text
本地 commit -> git push origin main -> 服务器 webhook / CI 触发 -> git pull -> 检查 -> 构建 -> 重启服务
```

注意：

- 不能只靠本地 commit 自动更新服务器。
- 至少要 push 到 `origin/main`，服务器才有稳定来源。
- 自动部署建议只部署 `main` 或受保护分支。

服务器上可以使用：

```bash
cd /opt/localtalent/app
LOCALTALENT_DEPLOY_CONFIRM=deploy ./scripts/deploy_server_source
```

脚本默认行为：

- 确认当前仓库可部署。
- 检查工作区是否干净。
- `git fetch origin main`。
- `git reset --hard origin/main`。
- 执行边界和三期闸门脚本。
- 打包后端。
- 安装前端依赖并构建。
- 重启 `localtalent-backend` 和 `localtalent-frontend` systemd 服务。

如果要先演练，不重启服务：

```bash
LOCALTALENT_DEPLOY_CONFIRM=deploy \
LOCALTALENT_DEPLOY_RESTART_MODE=none \
./scripts/deploy_server_source
```

## 7. GitHub webhook / CI 触发建议

最小方案：

1. 服务器开放一个只允许 GitHub 访问的 webhook endpoint。
2. webhook 校验 secret。
3. webhook 在服务器上执行：

```bash
cd /opt/localtalent/app
LOCALTALENT_DEPLOY_CONFIRM=deploy ./scripts/deploy_server_source
```

更稳方案：

- GitHub Actions 先跑测试。
- 测试通过后再 SSH 到服务器执行部署脚本。
- 服务器脚本仍然本地再跑边界脚本和构建，作为最后一道门。

当前不建议：

- 直接在 webhook 中执行未校验来源的 shell 命令。
- 用 `git pull` 部署脏工作区。
- 用开发服务 `npm run dev` 作为长期服务。

## 8. 后续更正式：Docker 镜像部署

当前仓库没有 backend/frontend Dockerfile。后续若要更正式，可以补：

- 后端 Dockerfile：构建 Spring Boot jar 并运行。
- 前端 Dockerfile：构建 Next standalone 并运行。
- GitHub Actions：push 后构建镜像并推送。
- 服务器：只拉取镜像并重启容器。

这会比服务器本地构建更干净，但会增加镜像仓库、密钥和发布治理工作。

## 9. 部署后验收

```bash
curl -I http://你的域名/
curl -I http://你的域名/jobs
curl -I http://你的域名/companies
curl -I http://你的域名/portal/talent-service-area
curl -I http://你的域名/auth/login
curl http://你的域名/health
```

三期灰度验证：

```bash
LOCALTALENT_FRONTEND_BASE_URL=http://你的域名 \
LOCALTALENT_API_BASE_URL=http://你的域名 \
./scripts/smoke_phase3

LOCALTALENT_FRONTEND_BASE_URL=http://你的域名 \
./scripts/load_phase3_public
```

验收标准：

- 首页、找工作、找企业、人才服务区、招聘会、资讯、HR 工具箱可访问。
- 后端 `/health` 正常。
- 登录入口可打开。
- 私有中心未登录时不泄露数据。
- 人才服务区仍只展示发布快照。
- 风险池能力仍未开放。

## 10. 安全边界

部署不能改变业务边界：

- 人才服务区仍只展示发布快照。
- 公开门户不输出原始候选人数据。
- 对接接口仍为 stub。
- 公共简历库、受控找人才、联系解锁、真实支付、真实短信/微信/小程序/App、真实地图/视频/直播、ATS/RPA 仍不得上线。
- 生产/灰度环境不要使用 `.env.phase3.example` 中的占位密钥。
- 不要把 MinIO 本地默认账号密码用于公网环境。

## 11. 推荐提交流程

部署前建议先提交本地变更：

```bash
git status --short
git add <本轮文件>
git commit -m "docs: add server deployment guide"
git push origin main
```

服务器自动部署必须以 `git push` 后的远程分支为准。
