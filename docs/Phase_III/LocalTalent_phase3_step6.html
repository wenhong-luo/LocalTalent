1. Step1-Step5 继承结论

Step1 冻结三期目标为小范围真实试运营；Step2 收口 candidate 真实闭环；Step3 收口 company 招聘工作台；Step4 收口运营后台和门户运营化；Step5 收口审计、导出、风控与安全检查单。Step6 只做工程治理与上线演练，不扩展业务功能。

继承结论Step6 继承方式
candidate/company/operator/auditor 闭环已经规划完成。灰度演练必须覆盖四类角色的关键路径，而不是只测公开门户。
门户由真实状态和后台配置驱动。容量烟测、缓存、sitemap 和监控必须覆盖首页、找工作、找企业、人才服务区、招聘会、资讯等公开页。
Step5 已冻结安全检查单。Step6 必须把安全检查单接入上线前闸门、暂停条件和回滚流程。
风险池能力不得偷跑。灰度开关、Runbook、演示数据和故障处理都不得临时放开风险池能力。
三期不是公网正式运营。工程目标是灰度试运营可控、可暂停、可回滚，不是一次性生产大上线。

3. Step6 总目标

灰度试运营前，系统必须可启动、可灰度、可观测、可备份、可恢复、可暂停、可回滚、可退出。
  1. 本地、预发、灰度环境的配置差异清晰，敏感配置不进入仓库。
  2. feature flag 能控制三期新增能力的打开、暂停和回滚。
  3. 数据库和对象存储有备份、恢复、保留、清理和负责人。
  4. 日志脱敏、监控指标、告警门槛、故障排查路径清晰。
  5. 公共门户、私有中心、后台、导出和审计关键路径具备 smoke test。
  6. 灰度容量烟测覆盖公开读流量、登录态接口、导出 worker 和撤回下线传播。
  7. 上线、暂停、回滚 Runbook 可被新人照着执行。
  8. 灰度 KPI、扩大范围条件、停止条件和试运营退出条件明确。

4. 预发环境与灰度配置

环境目标允许能力禁止事项

production-ready四期正式上线准备压测、监控、客服、合规材料、供应商治理和生产发布流程。不在三期 Step6 中承诺完成。

4.1 配置口径

  • 本地和单机预发继续使用 Docker Compose 或单节点容器编排。
  • K8s 只作为无状态应用层的进阶选项：Next SSR/BFF、后端 API、worker。
  • MySQL、Redis、MinIO 不在三期同阶段强制迁移到 K8s。
  • 不在没有备份恢复演练和回滚演练前进入灰度。

flag 示例控制范围默认值回滚动作

5.1 灰度发布步骤

  1. 确认本轮范围、禁止事项和 feature flag 列表冻结。
  2. 在 staging 打开目标 flag，跑 smoke、边界、字段黑名单、IDOR/BOLA、导出和审计回归。
  3. 做备份恢复演练和应用回滚演练。
  4. 在 gray 环境对白名单用户打开 5% 或指定账号范围。
  5. 观察 5xx、P95、登录成功率、撤回下线传播、导出审批合规率、审计覆盖率。
  6. 无 P0/P1 问题后扩大到 20%、50%、100% 白名单。

5.2 暂停条件

  • 公开门户 API 5xx 持续超过 0.5%。
  • BOLA/IDOR、字段泄露、导出绕行、审计缺失等 P0/P1 安全问题。
  • 撤回下线传播超过最大 5 分钟仍未收敛。
  • 导出链接过期或二次发放规则失效。
  • 关键角色无法登录或身份分流错误。
  • 备份恢复失败或回滚演练失败。

8. 日志脱敏、监控告警与故障排查

8.1 日志脱敏规则

日志、错误信息、审计响应、open_api_log、埋点和调试输出不得包含手机号、邮箱、联系方式、简历正文、附件对象 key、证据、原始候选人 JSON、营业执照附件、审核材料、后台备注、报名名单、签到证据、password、secret、token、signature、nonce 明文。

8.2 最小监控指标

指标灰度关注点
应用健康/health、后端启动、Next SSR 启动、worker 存活。
错误率公开门户 5xx、私有中心 5xx、后台 5xx、导出失败率。
延迟公开页 P95、登录 P95、导出申请 P95、审计查询 P95。

10. 上线、暂停、回滚 Runbook

10.1 上线前检查

检查项要求
范围冻结本次灰度包含哪些 flag、页面、接口和用户范围。
禁止事项确认风险池能力未混入灰度主线。
测试通过后端、前端、边界脚本、字段黑名单、三期专项脚本通过。
备份完成数据库和对象存储备份完成并记录位置。
恢复演练staging 恢复演练通过。
回滚演练flag 关闭、应用回滚、roll-forward 修复路径演练通过。
监控告警健康、5xx、延迟、导出、审计、撤回下线告警可用。
人员排班产品、研发、运营、值守和决策人明确。

10.2 暂停步骤

  1. 停止扩大灰度名单。
  2. 关闭相关 feature flag。
  3. 暂停高风险写动作，例如导出申请、职位发布、推荐位调整。
  4. 保留只读入口，避免用户完全失联。
  5. 记录事件、trace、影响范围、临时处置和负责人。
  6. 完成根因确认后再决定恢复、回滚或 roll-forward。

10.3 回滚步骤

  1. 确认问题级别和影响范围。
  2. 先关闭相关 feature flag。
  3. 若仍未恢复，回滚应用版本。
  4. 数据库问题优先新增向前修复迁移，不直接修改旧迁移。
  5. 若涉及数据泄露或权限问题，立即冻结相关入口并保留审计证据。
  6. 回滚后执行 smoke test 和关键路径验证。
  7. 复盘记录必须写入故障记录模板。

11. 灰度 KPI 与试运营退出条件

KPI建议门槛说明

公开页 P95< 500ms按服务端公开接口口径。
登录成功率> 99%覆盖本地账号或 OIDC 灰度入口。
撤回下线传播中位 < 60 秒，最慢 < 5 分钟三期最关键合规指标之一。

真实灰度闭环数至少 20 位候选人、10 家企业、3 名运营跑通避免三期变成“更完整的演示”。

11.1 退出试运营条件

  1. candidate、company、operator、auditor 四类角色均完成真实闭环。
  2. 公开门户真实状态驱动稳定，SEO 与 sitemap 不含私有或撤回对象。
  3. 审计、导出、撤回、IDOR/BOLA、字段黑名单回归稳定。
  4. 备份恢复、回滚演练、告警联通和容量烟测都有记录。
  5. 风险池能力仍未偷跑。
  6. 产品、技术、运营共同确认可进入四期正式上线治理。
