# LocalTalent 三期领导演示路线

> 本文面向领导汇报和客户看样场景：用 10 分钟讲清楚 LocalTalent 当前做到什么、怎么演示、哪些能力可以灰度试运营、哪些能力仍然不能越界。三期当前定位是 **小范围真实试运营候选版本**，不是公网正式上线，也不是商业化完整版。

## 1. 演示前结论

- 三期主线 `Prompt 25-31` 已收口，可作为小 demo 给领导看。
- 整站 UI 精致化 `Prompt UI-1` 至 `Prompt UI-8` 已收口，公开门户、求职者中心、招聘者后台和运营后台视觉密度已统一。
- 已按 `docs/Phase_III/三期灰度试运营使用说明.md` 做过页面复核。
- `./scripts/smoke_phase3` 已通过。
- 以下页面访问均返回 `200`：`/`、`/jobs`、`/companies`、`/portal/talent-service-area`、`/job-fairs`、`/articles/policies`、`/hr-tools`、`/auth/login`、`/candidate/center`、`/company`、`/admin`。
- 演示时建议只说“可进入灰度复核/小范围试运营候选”，不要说“已经公网正式上线”。

## 2. 演示前准备

### 2.1 启动环境

```bash
docker compose -f infra/docker-compose.yml up -d
```

启动后端：

```bash
cd backend
./mvnw spring-boot:run
```

启动前端：

```bash
cd frontend
npm run dev
```

默认地址：

| 服务 | 地址 |
| --- | --- |
| 前端 | `http://localhost:3000` |
| 后端健康检查 | `http://localhost:8080/health` |

### 2.2 导入演示数据

```bash
./scripts/seed_phase2_demo_data
```

说明口径：

- 三期没有把演示数据放进生产迁移。
- 演示数据只用于本地和灰度看样。
- 真实环境数据进入前仍要走灰度 Runbook 和合规检查。

### 2.3 开启三期演示能力

演示求职者、企业、运营三期能力前，后端启动环境建议开启：

```bash
export LOCALTALENT_PHASE3_CANDIDATE_CLOSURE=true
export LOCALTALENT_PHASE3_COMPANY_WORKBENCH=true
export LOCALTALENT_PHASE3_OPERATOR_PORTAL_OPS=true
```

说明口径：

- 三期新增能力默认由 feature flag 控制。
- 灰度时可以逐步打开、暂停、关闭。
- flag 不能绕过 RBAC、数据域、字段级权限、审计或导出审批。

## 4. 10 分钟演示路线

演示时可以先点明一条 UI 口径：公开门户已经统一导航高亮和左右广告留白，招聘者后台和求职者中心也已收敛为更精致的小控件、高密度卡片与可演示的私有中心布局。

### 4.1 开场 30 秒

建议话术：

```text
LocalTalent 现在已经从一二期的可演示人才网，推进到三期小范围灰度试运营形态。
今天演示的不是公网正式上线，也不是商业化完整版，而是验证真实求职者、企业和运营人员能在合规边界内完成核心闭环。
```

必须强调三条边界：

- 人才服务区只展示发布快照，不是公共简历库。
- 原始候选人数据不会出现在公开门户。
- 短信、微信、小程序、App、地图、视频、直播、支付、ATS/RPA 仍是风险池或占位。

### 4.2 门户 3 分钟

演示页面：

| 页面 | 地址 | 讲什么 |
| --- | --- | --- |
| 首页 | `/` | 对齐 `docs/page/网站首页` 的高保真首屏、职位分类墙、运营右栏、通栏广告位和公开内容模块 |
| 找工作 | `/jobs` | 职位筛选、职位卡片、详情入口 |
| 找企业 | `/companies` | 已认证企业公开列表、企业主页入口 |
| 人才服务区 | `/portal/talent-service-area` | 发布快照展示区，不是简历库 |
| 招聘会/政策/工具箱 | `/job-fairs`、`/articles/policies`、`/hr-tools` | 招聘会、就业政策、HR 工具箱 |
| 补充频道 | `/jobs/famous`、`/daily-jobs`、`/campus` | 二期补齐的低风险公开频道 |

讲解重点：

- 门户已经接近完整地方人才网形态。
- 首页已从“骨架门户”升级为可演示首页：右侧不再是登录表单，而是公告/招聘会/安全入口；下方补齐广告位、热招职位、名企展示、招聘会、资讯公告、HR 工具箱和友情链接。
- 首页、找工作、找企业、人才服务区等频道已统一广告留白与导航高亮，找企业不再和首页/找工作割裂。
- 公开页面只显示白名单字段。
- 推荐位和公开内容在三期开始转向运营配置和真实状态驱动。
- 高风险入口只占位，不偷跑真实外部能力。

建议串讲：

```text
这是给公众访问的人才网门户。首页、找工作、找企业、招聘会、政策资讯和工具箱已经形成完整门户结构。
这里重点不是“页面好看”而已，而是公开侧字段被严格裁剪，人才服务区只看发布快照，不能看到完整简历或联系方式。
```

### 4.3 求职者 2 分钟

演示页面：

- `/auth/login`
- `/candidate/center`

讲解重点：

- 求职者可以维护简历、预览、查看投递、收藏、订阅、站内通知、隐私与发布状态。
- 求职者首次完善简历、会员中心、期望职位/地区选择器、附件上传和 AI 建议卡片已统一为更精致的私有中心视觉。
- 同意/撤回由服务端控制。
- 撤回后人才服务区和公开推荐位不可见。
- 简历原始字段只在本人私有域使用，不进入公开门户、搜索、缓存、日志或 sitemap。
- 站内通知是当前默认渠道，未接真实短信/微信/App。

建议串讲：

```text
这是求职者登录后的私有中心。三期重点是让求职者真正能维护资料、看投递、看通知和管理发布状态。
但这些都属于本人私有域，公开门户仍只读取发布快照白名单。撤回之后，公开侧会联动不可见。
```

### 4.4 企业 2 分钟

演示页面：

- `/auth/login`
- `/company`

讲解重点：

- 企业可以维护资料、提交认证、创建/编辑职位、送审、下线。
- 招聘者后台 Shell、会员首页、职位管理、发布职位长表单、企业资料、Logo、企业风采和受控搜索简历均已统一为更接近专业人才网后台的紧凑风格。
- 投递池只处理本企业职位收到的投递，不是找人才或简历库。
- 投递详情只展示裁剪后的候选人摘要。
- 不显示未授权手机号、邮箱、简历正文、附件 key 或证据。
- 导出只能走审批链，不能直接下载或绕过字段级权限。

建议串讲：

```text
这是企业侧招聘工作台。企业可以维护资料、提交认证、管理职位和处理投递。
这里的投递池不是公共简历库，只能处理本企业职位产生的投递，跨企业访问会被服务端阻断。
导出也不能直接下载，必须走审批和字段裁剪。
```

### 4.5 运营和安全 3 分钟

演示页面：

- `/admin`

重点展示：

- 运营首页
- 推荐位配置
- 风险审核
- 审计中心入口

讲解重点：

- `operator` 可以审核、配置推荐位、处理风险，所有写动作进入 `audit_log`。
- `auditor` 只读，不能写。
- 推荐位只能引用合法公开对象：已审核职位、认证企业、已发布内容、有效发布快照。
- 审计 trace 可以串联 `audit_log`、`field_access_log`、`open_api_log`，响应脱敏。
- 导出仍走 `export_apply -> operator 审批 -> 字段裁剪 -> 短期下载 -> 审计`。
- 三期已经补了灰度工程脚本：`smoke_phase3`、`load_phase3_public`、备份恢复演练、灰度暂停/回滚检查单。

建议串讲：

```text
运营后台是三期能否试运营的关键。我们不是只做了前台页面，也补了审核、推荐位、风险处理和审计入口。
operator 可以操作，但 auditor 只能只读。所有关键写动作都有 trace 和 audit_log，敏感字段访问进入 field_access_log。
```

## 6. 备用问答

### 6.1 领导问：现在能不能上线？

建议回答：

```text
可以作为小范围灰度试运营候选版本，不建议直接公网正式运营。
正式公网运营还需要四期继续补生产级监控、客服工单、压测、合规材料、备份策略、真实运营流程和上线治理。
```

### 6.2 领导问：是不是已经有简历库？

建议回答：

```text
不是。LocalTalent 当前坚持合规边界，公开侧只有人才服务区发布快照。
公共简历库、联系解锁、受控找人才都必须走风险准入，不能直接上线。
```

### 6.3 领导问：能不能接短信、微信、支付？

建议回答：

```text
当前不接。三期只做站内闭环和灰度治理。
真实短信、微信、小程序、App、支付、地图、视频、直播、ATS/RPA 都在风险池，后续单独评审。
```

### 6.4 领导问：和二期 demo 相比，三期新增了什么？

建议回答：

```text
二期重点是高保真公开门户，三期重点是让真实角色能试运营。
求职者能维护简历和管理投递，企业能处理职位和投递，运营能审核、配置和查审计。
同时补了灰度脚本、备份恢复、smoke/load、暂停回滚检查单。
```

### 6.5 领导问：最大风险是什么？

建议回答：

```text
最大风险不是页面，而是真实数据进入后能否持续守住权限、字段裁剪、撤回联动和审计。
所以三期把 IDOR/BOLA、字段黑名单、导出审批、审计 trace、灰度回滚都放进验收闸门。
```

## 7. 演示收尾 30 秒

建议话术：

```text
今天看到的是三期灰度试运营候选版本。
它已经能支撑领导看样和内部灰度复核，但我们仍然严格区分灰度试运营和公网正式上线。
下一步建议先做一次真实灰度演练，验证少量求职者、企业和运营人员的完整闭环，再决定是否进入四期正式上线治理。
```
