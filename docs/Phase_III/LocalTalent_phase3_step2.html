一、Step1 继承结论

Step1 已经冻结三期目标：把一二期的可演示系统推进为 小范围真实试运营。Step2 只承接其中的 candidate 闭环，不处理企业工作台、运营后台、商业化或风险池能力。

Step1 结论Step2 继承方式
三期目标是灰度试运营，不是公网正式运营。Step2 只要求 candidate 闭环真实可用，暂不追求大规模流量和完整客服体系。
人才服务区仅展示发布快照。简历编辑可处理本人原始字段，但公开展示仍只能来自 candidate_publish_snapshot。
状态必须由服务端返回。前端不得推断 resume、application、favorite、subscription、notification、consent、publish 状态。
三期默认通知渠道是站内通知。收藏、订阅、投递状态、面试邀约只进入站内通知，不接真实短信、真实微信、小程序或 App。
风险池能力不得偷跑。公共简历库、受控找人才、联系解锁、AI 处理原始候选人数据不进入 Step2。

二、一二期求职者能力基线

一二期已经完成了一部分求职者能力，三期 Step2 不应重复造轮子。

能力已有基线Step2 增量
注册登录已有本地账号/JWT 与登录注册 UI。三期后续按总规划接入认证与身份映射；Step2 只定义 candidate 业务闭环。
求职者中心 overview已有 GET /api/candidate/center/overview 只读联调。扩展为真实中心首页，汇总简历、投递、收藏、订阅、通知与发布状态。
同意/撤回已有 consent、control profile、publish snapshot、撤回下线链路。强化便捷撤回、发布状态说明、撤回后公开不可见回归。
人才服务区已有发布快照列表、高保真筛选、字段黑名单。继续只读发布快照，不新增公共简历库或查看简历能力。
投递与面试已有投递、企业投递池、面试签到基础闭环。candidate 侧补投递记录、面试/签到状态回流和站内通知。
公开门户入口二期已有找工作、职位详情、登录入口。candidate 从职位详情进入投递、收藏、订阅后回流求职者中心。
Step2 允许规划“简历编辑使用本人原始字段”，但必须同时写清：这些字段不得进入公开门户、搜索索引、日志、缓存、sitemap 或导出旁路。

三、Step2 总目标

candidate 能完成：维护简历 → 预览确认 → 投递职位 → 查看投递/面试状态 → 收藏职位 → 订阅搜索 → 接收站内通知 → 同意/撤回发布。
  1. 登录后进入求职者中心，看到真实的简历完成度、发布状态、投递状态、收藏/订阅数量和未读通知。
  2. 编辑本人简历并预览，保存后状态由服务端返回。
  3. 从职位详情发起投递，之后在投递记录中看到状态回流。
  4. 收藏职位、取消收藏、创建搜索订阅、取消订阅，动作具备幂等和本人数据域限制。
  5. 收到站内通知，能查看并标记已读。
  6. 便捷同意或撤回发布，撤回后人才服务区、搜索索引、缓存、sitemap 和公开入口联动不可见。

四、求职者中心信息架构

模块目标能力状态与边界
中心首页汇总简历、投递、面试、收藏、订阅、通知、发布状态。只展示本人摘要；状态来自服务端。
简历编辑编辑基础资料、教育经历、工作经历、技能、求职意向、附件占位。可使用本人原始字段；不得进入公开层。
简历预览candidate 预览本人简历与公开快照差异。预览页必须区分“本人可见字段”和“公开快照字段”。
投递记录查看投递职位、企业、状态、面试邀约、签到状态。只能访问本人 application；覆盖 IDOR/BOLA。
职位收藏收藏、取消收藏、查看收藏列表。不触发企业通知，不变成联系入口。
搜索/职位订阅保存关键词、城市、职类等低风险条件。默认只生成站内通知，不做短信/微信/App 推送。
站内通知查看系统通知、投递状态、面试邀约、订阅提醒。不接真实短信、真实微信、小程序、App。
隐私与发布状态查看同意状态、发布快照状态、撤回入口。撤回必须便捷，且公开可见性联动下线。

求职者中心可以更像“工作台”，但绝不能变成“公开简历库的后台入口”。candidate 自己能看见的，不代表游客、企业或运营都能看见。

五、状态机与服务端口径

Step2 的核心纪律是：状态以服务端为准。

状态域服务端应返回的状态口径前端禁止行为
resume草稿、待完善、已完善、不可发布、可发布、更新时间。禁止按字段数量自行推断最终可发布状态。
application已投递、已查看、邀约面试、已签到、已拒绝、已撤回或关闭。禁止前端按时间或按钮点击推断投递阶段。
favorite已收藏、未收藏、收藏总数。禁止只用 localStorage 作为真实状态。
subscription已订阅、暂停、已取消、最近触发时间。禁止直接触发短信、微信、小程序、App。
notification未读、已读、归档、通知类型。禁止把通知当作外部消息发送记录。
consent已同意、已撤回、未同意、版本、更新时间。禁止前端推断 consent_status。
publish已发布、已下线、不可发布、生成中、生成失败。禁止把原始简历当公开发布内容。

任何写动作成功后，页面应重新读取服务端状态，而不是用写接口返回值在前端拼一个“看起来对”的状态。

六、字段与权限边界

本人私有域

candidate 在本人私有中心内，可以维护和查看本人求职必要字段，例如基础资料、教育经历、工作经历、技能、求职意向、附件占位和投递记录。

  • 必须 JWT candidate 登录。
  • 必须强制当前 candidate 只能访问本人数据。
  • 所有含对象 ID 的接口必须覆盖 IDOR/BOLA。
  • 非 candidate、未登录、跨 candidate 访问必须返回统一无权限响应与 trace。

公开域

公开门户、人才服务区、搜索索引、日志、缓存、sitemap、analytics 只能使用公开白名单。

  • 不得包含手机号、邮箱、身份证、联系方式。
  • 不得包含简历正文、附件对象 key、证据。
  • 不得包含原始 JSON、聊天记录、通知明细、投递备注、面试评价。
  • 不得包含审核材料、运营备注、风控结论。
人才服务区公开字段继续限定为发布快照白名单：snapshot_id/display_name_masked/city_code/category_code/skills_summary/experience_years/updated_at。

Step2 不新增企业找人才、联系解锁或公共简历库能力。企业仍只能通过合法投递链路看到与本企业职位相关的投递信息。

七、审计、幂等与撤回联动

动作要求
简历保存写接口需要幂等或防重复提交；记录更新时间和 trace。
简历发布状态变化写 audit_log；若涉及字段读取，按字段级权限写 field_access_log。
同意幂等；写 consent 历史、control profile、audit_log；生成发布快照。
撤回幂等；写 consent 历史、control profile、audit_log；下线发布快照。
投递继承一期投递审计和重复投递冲突处理。
收藏/取消收藏幂等；仅影响本人私有列表。
订阅/取消订阅幂等；仅生成站内通知，不外发短信/微信/App。
通知已读幂等；只影响本人通知状态。

撤回联动必须覆盖 candidate_control_profile 当前状态、candidate_consent 历史记录、candidate_publish_snapshot 下线、人才服务区不可见、搜索索引/缓存/sitemap/公开推荐位失效，以及审计日志按 trace 串联。

三期灰度口径建议沿用总规划：撤回下线传播中位小于 60 秒，最慢小于 5 分钟；这是灰度试运营的信任底线。

九、Step2 结论与下一步

  • 三期求职者闭环应优先让 candidate 真实可用，而不是继续堆公开页面。
  • 简历编辑与预览属于本人私有域，可以处理本人原始字段，但不得进入公开域。
  • 收藏、订阅、通知只做站内闭环；真实短信、真实微信、小程序、App 继续暂缓。
  • 同意/撤回和发布快照仍是人才服务区的合规核心。
  • 所有状态必须由服务端返回，前端不得推断。
  • 所有对象 ID 必须覆盖 IDOR/BOLA。
