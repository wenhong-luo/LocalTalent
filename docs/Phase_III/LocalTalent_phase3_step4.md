# LocalTalent Phase 3 Step4：运营后台生产化与门户运营化

## 开发前必读卡

- **文件定位**：三期第四份 Step 文档；用于把“运营后台演示能力”和“二期高保真门户”设计成后续蓝图和 Prompt 可继承的真实运营后台与门户运营体系。
- **输入资料**：`00_PHASE_III_BOUNDARY.md`、`LocalTalent_phase3_step1.md`、`LocalTalent_phase3_step2.md`、`LocalTalent_phase3_step3.md`、`LocalTalent_三期总规划.md`、Phase I/II 运营审核、CMS/event、审计中心、公开门户相关基线。
- **本轮目标**：冻结 operator/auditor 在三期灰度试运营中必须具备的审核、配置、发布、下线、推荐、风险处理、审计查询和门户运营化能力。
- **禁止越界点**：不写代码，不新增迁移，不生成三期蓝图或提示词，不实现会员商业化、真实支付、真实短信、真实微信、小程序、App、真实地图、视频、直播、ATS/RPA 或风险池能力。
- **输出边界**：只定义运营后台生产化与门户运营化的产品范围、状态口径、权限边界、审计要求和验收关注点。
- **下一步影响**：本文件稳定后，下一份建议文档是 `LocalTalent_phase3_step5.md：审计、导出、风控与安全收口`。

**统一硬边界提醒**：运营后台可以管理和审核，但不得绕过服务端权限、数据域、字段级权限和审计链；公开门户只能展示已审核、已发布、在线且合规的公开白名单字段；人才服务区仍只展示发布快照。

## 1. Step1-Step3 继承结论

Step1 已经冻结三期目标是 **小范围真实试运营**；Step2 已经收口 candidate 真实闭环；Step3 已经收口 company 招聘工作台。Step4 只承接运营后台和门户运营化，不处理商业化、外部能力或风险池专项。

| 继承结论 | Step4 继承方式 |
| --- | --- |
| 三期目标是灰度试运营，不是公网正式运营。 | Step4 只要求运营后台能支撑小范围真实运营，不追求完整商业运营平台。 |
| 求职者中心是本人私有域。 | 运营后台不得把候选人原始字段暴露到公开门户或推荐位。 |
| 企业中心是本企业私有域。 | 运营审核可改变企业/职位状态，但不能替企业绕过认证、审核或数据域。 |
| 公开门户必须真实状态驱动。 | 首页、找工作、找企业、招聘会、资讯、HR 工具箱等必须只展示合法公开对象。 |
| operator/auditor 分权必须强制。 | operator 可在授权范围内写；auditor 只读，所有写接口必须拒绝 auditor。 |
| 风险池能力不得偷跑。 | 推荐位、内容活动、风险审核不得包装成联系解锁、会员权益或外部同步。 |

Step4 的关键问题是：真实运营人员能不能安全地审核、配置和运营门户，同时每个动作可追踪、可回滚、可解释。

## 2. 一二期运营能力基线

一二期已经具备运营后台的基础能力，三期 Step4 不应重复造轮子。

| 能力 | 已有基线 | Step4 增量 |
| --- | --- | --- |
| 企业审核 | 已有企业审核队列、详情、通过、驳回和审计。 | 补生产化列表筛选、驳回补正链路、风险提示和审核效率视图。 |
| 职位审核 | 已有职位审核队列、详情、通过、驳回和可见性联动。 | 补职位内容风险提示、批量审核边界、审核后门户状态解释。 |
| CMS/event | 已有内容和活动最小 CRUD、软删除、operator/auditor 权限。 | 补内容频道、招聘会、活动公开展示配置和发布/下线状态治理。 |
| 导出审批 | 已有 export_apply、审批、生成、MinIO 短期下载和审计链。 | 运营后台只强化审批视图、风险提示和审计跳转，不绕过审批。 |
| 审计中心 | 已有 audit_log、field_access_log、open_api_log 查询和脱敏。 | 补运营视角的 trace 聚合、筛选、异常入口和只读审计 UI。 |

| 二期公开门户 | 已有首页、找工作、找企业、人才服务区、招聘会、资讯、HR 工具箱、更多服务。 | 从静态演示和低风险占位转为后台配置与真实状态驱动。 |

Step4 允许运营后台管理公开展示配置，但公开门户最终展示必须继续由服务端可见性规则和字段白名单收口。

## 3. Step4 总目标

三期运营后台生产化与门户运营化的目标是：

```text
operator 能完成：审核企业/职位 → 管理内容/活动/招聘会 → 配置推荐位 → 处理风险 → 查询审计 → 驱动公开门户真实展示。
auditor 能完成：只读查看审核、风险、审计和运营记录，不得写入任何业务状态。
```

达到灰度试运营口径时，运营侧至少应能做到：

1. 运营首页看到待审核企业、待审核职位、待处理内容、待处理活动、导出审批、风险提示和最近审计摘要。
2. 审核企业和职位，填写驳回原因，审核结果联动公开门户可见性。
3. 管理就业政策、职场资讯、公告、HR 工具箱、帮助中心等内容，并支持发布、下线、软删除。
4. 管理招聘会和活动公开展示配置，不暴露报名名单、签到证据或候选人明细。
5. 配置首页和频道推荐位，但推荐对象必须是已审核、已发布、在线且合规的公开对象。
6. 处理内容、企业、职位、导出、审计异常等风险项，并记录处理结论。
7. 通过审计中心 UI 查询 trace、审核记录、字段访问记录和开放接口日志，且响应脱敏。

## 4. 运营后台信息架构

Step4 冻结运营后台的信息架构，不直接展开成接口或数据库 schema。

| 模块 | 目标能力 | 状态与边界 |
| --- | --- | --- |
| 运营首页 | 汇总待审核、待发布、待处理风险、导出审批、审计异常和门户健康摘要。 | operator/auditor 均可读；写入口按权限显示，服务端强制。 |
| 企业审核 | 查看企业资料、认证材料、历史审核、通过或驳回。 | 不公开营业执照附件、审核材料、后台备注；写 audit_log。 |
| 职位审核 | 查看职位内容、企业认证状态、风险提示、通过或驳回。 | 未认证企业职位不得通过上线；写 audit_log。 |
| 内容管理 | 管理就业政策、资讯、公告、HR 工具箱、帮助中心。 | 发布/下线/软删除均写审计；公开输出做字段白名单和安全文本处理。 |
| 活动/招聘会管理 | 管理现场招聘会、网络招聘会占位、校园活动、普通活动。 | 不公开报名名单、签到证据、参会个人明细。 |
| 推荐位配置 | 配置首页、频道、企业、职位、招聘会、内容推荐。 | 推荐对象必须状态合法；不得推荐撤回快照、下线职位、未认证企业。 |
| 风险审核 | 处理内容风险、企业资料风险、职位风险、导出风险、审计异常风险。 | 风险池能力不能直接升级为开发功能；处理动作写 audit_log。 |
| 审计中心 | 查询 audit_log、field_access_log、open_api_log 和 trace 聚合。 | 只读、脱敏、可按 trace 串联；敏感字段读取写 field_access_log。 |
| 系统配置占位 | 展示灰度配置、门户开关、环境提示、Runbook 链接。 | 不提供真实支付、短信、微信、地图、直播等外部能力开关。 |

运营后台可以提升管理效率，但不能成为绕过服务端规则的“超级后门”。

## 5. 门户运营化设计

二期已经实现高保真门户；三期 Step4 的目标是让这些页面由真实状态和后台配置驱动。

| 门户区域 | 运营化目标 | 可见性规则 |
| --- | --- | --- |
| 首页 | 推荐位、公告、招聘会、推荐企业、热招职位由后台配置或服务端规则驱动。 | 只展示已审核、已发布、在线对象。 |
| 找工作 | 职位列表、名企直聘、急聘/今日招聘从真实职位状态读取。 | 企业认证通过、职位审核通过、职位在线。 |
| 找企业 | 企业列表和企业主页从企业认证状态和公开资料读取。 | 企业认证通过，且只展示公开白名单字段。 |
| 人才服务区 | 继续只读 `candidate_publish_snapshot`，可按低风险筛选展示。 | 候选人同意、可发布、快照有效；撤回后不可见。 |
| 招聘会 | 招聘会列表/详情由活动状态驱动。 | 活动 `status=1` 且公开展示，不含报名名单或签到证据。 |
| 政策资讯 | 就业政策、职场资讯、公告、HR 工具箱由 CMS 状态驱动。 | 内容已发布、未软删除，正文安全输出。 |
| 更多服务 | 低风险页面稳定路由；高风险入口保留占位或风险池文案。 | 禁止真实地图、视频、直播、短信、微信、小程序、App。 |
| sitemap/SEO | 只收录公开页面和合法公开对象。 | 私有中心、后台、审计、风险页全部 noindex。 |

推荐位配置必须具备状态校验。如果配置对象后续被驳回、下线、撤回或软删除，门户应自动隐藏或降级，不得继续展示陈旧对象。

## 6. 状态机与服务端口径

Step4 的核心纪律是：**运营状态和门户展示状态以服务端返回为准**。

| 状态域 | 服务端应返回的状态口径 | 前端禁止行为 |
| --- | --- | --- |
| company_review | 待审核、通过、驳回、补正中、冻结。 | 禁止前端根据资料完整度自行判定认证通过。 |
| job_review | 待审核、通过、驳回、下线、风险待处理。 | 禁止未审核或驳回职位公开可见。 |
| cms_content | 草稿、待发布、已发布、已下线、软删除。 | 禁止直接渲染未经安全处理的 HTML。 |
| activity_event | 草稿、待发布、已发布、已下线、软删除、已结束。 | 禁止公开报名名单、签到证据或参会个人明细。 |
| recommendation | 待生效、生效中、已下线、目标失效、风险拦截。 | 禁止推荐无效、未审核、已撤回或高风险对象。 |
| risk_review | 待处理、处理中、已通过、已驳回、已转风险池。 | 禁止风险池能力进入主线开发。 |
| audit_center | 可查询、无结果、脱敏输出、权限不足。 | 禁止审计接口返回敏感明文。 |

任何写动作成功后，运营页面应重新读取服务端状态；前端按钮隐藏只是体验，不能替代服务端权限控制。

## 7. 权限、字段与审计边界

### 7.1 operator 与 auditor

| 角色 | 权限口径 |
| --- | --- |
| operator | 可在授权范围内执行审核、发布、下线、软删除、推荐位调整、风险处理、导出审批等写动作。 |
| auditor | 只读查看审核、风险、审计和运营记录；不得调用任何写接口。 |

要求：

- 前端可隐藏写按钮，但服务端必须强制 RBAC。
- auditor 写接口必须统一返回无权限响应和 trace。
- operator 写动作必须写 `audit_log`。
- 敏感字段读取必须写 `field_access_log`。

### 7.2 公开字段边界

运营后台可以看到更多管理字段，但公开门户只能展示公开白名单。

禁止进入公开门户、sitemap、缓存、搜索索引、埋点和推荐位摘要的字段：

- 手机号、邮箱、身份证、联系方式。
- 简历正文、附件对象 key、证据、原始候选人 JSON。
- 营业执照附件、认证材料、审核材料、后台备注。
- 报名名单、签到证据、参会个人明细。
- 审计详情、字段访问记录、风控结论。
- 开放接口 secret、token、签名、nonce 明文。

### 7.3 审计口径

以下动作必须写 `audit_log`：

- 企业审核通过、驳回、补正处理。
- 职位审核通过、驳回、下线。
- 内容发布、下线、软删除、恢复。
- 活动/招聘会发布、下线、软删除、恢复。
- 推荐位新增、调整、下线、目标失效处理。
- 风险审核通过、驳回、转风险池。
- 导出审批通过、驳回。
- 审计异常处理。

以下读取或输出必须考虑 `field_access_log`：

- 查看高敏候选人字段。
- 查看导出字段明细。
- 查看认证材料、审核材料或风险材料。
- 查看开放接口敏感摘要。

审计中心自身响应必须脱敏，不能因为“审计人员要查问题”就返回敏感明文。

## 8. 风险审核与风险池边界

Step4 可以规划风险审核入口，但不得把风险池能力升级为三期当前需求。

| 风险类型 | Step4 允许做 | Step4 禁止做 |
| --- | --- | --- |
| 内容风险 | 标记、驳回、下线、转人工复核。 | 自动发布高风险内容。 |
| 企业资料风险 | 驳回认证、要求补正、冻结展示。 | 公开营业执照附件或审核材料。 |
| 职位内容风险 | 驳回职位、下线职位、记录原因。 | 允许未审核职位公开可见。 |
| 导出风险 | 审批驳回、记录原因、跳转审计。 | 绕过 export_apply 直接导出。 |
| 审计异常风险 | 标记异常、记录处理、关联 trace。 | 删除或篡改审计日志。 |
| 风险池能力 | 记录评估状态、关联 Step6 准入规则。 | 直接上线受控找人才、联系解锁、支付、短信、微信、地图、直播、ATS/RPA。 |

风险审核结论必须可追踪、可解释、可回滚。任何无法回滚的风险动作不应进入三期灰度试运营。

## 9. 自动化验收关注点

Step4 后续拆 Prompt 时，每轮至少需要覆盖以下验收点。

| 验收点 | 关注内容 |
| --- | --- |
| auditor 写拒绝 | auditor 调用企业审核、职位审核、内容发布、活动发布、推荐位、风险处理写接口必须 403。 |
| operator 写审计 | operator 审核、发布、下线、软删除、推荐位调整、风险处理必须写 `audit_log`。 |
| 门户可见性 | 公开门户只展示已审核、已发布、在线、未撤回、未软删除对象。 |
| 推荐位合法性 | 推荐位不得引用撤回快照、下线职位、未认证企业、未发布内容或软删除活动。 |
| 审计脱敏 | 审计中心响应不含手机号、邮箱、简历正文、附件、证据、secret、token。 |
| 字段黑名单 | 公开 DOM、响应、sitemap、缓存、搜索索引不含原始候选人数据和审核材料。 |
| noindex | 私有中心、运营后台、审计中心、风险页默认 noindex。 |
| 风险池阻断 | 高风险入口只能是占位、禁用或风险池状态，不存在真实外部调用。 |

建议后续实现阶段保留或新增以下测试方向：

- `AdminReviewProductionIT`：企业/职位审核、驳回原因、审核审计、门户可见性。
- `AdminContentActivityProductionIT`：内容/活动发布、下线、软删除、公开不可见。
- `AdminRecommendationIT`：推荐位配置、目标状态校验、失效隐藏。
- `AdminRiskReviewIT`：风险审核、转风险池、审计记录。
- `AuditCenterUiAccessIT`：operator/auditor 只读查询、脱敏、trace 聚合。
- 前端 admin 状态测试：loading、empty、error、unauthorized、pending_review、published、offline、soft_deleted、risk_pool。

这些测试名称只是规划建议，不是本轮要创建的代码文件。

## 10. Step4 结论与下一步

Step4 结论：

- 三期运营后台应优先支撑真实试运营，而不是扩展完整商业化运营平台。
- operator 可写、auditor 只读是硬边界，必须由服务端强制。
- 公开门户运营化必须受状态机、字段白名单、推荐位合法性和 SEO/noindex 规则约束。
- 内容、活动、招聘会、推荐位、风险审核都必须可审计、可回滚、可解释。
- 审计中心是只读脱敏查询，不是敏感明文查看后门。
- 高风险能力仍停留在风险池，必须先走准入规则。

建议下一份文档：

```text
LocalTalent_phase3_step5.md：审计、导出、风控与安全收口
```

Step5 应聚焦安全收口，不建议直接跳到商业化或外部能力：

1. 审计 trace 串联和脱敏。
2. 导出审批、字段裁剪和下载链回归。
3. 撤回下线传播与缓存/索引失效。
4. IDOR/BOLA 与字段黑名单总验收。
5. 风险池准入与阻断脚本。
6. 灰度试运营安全检查单。
