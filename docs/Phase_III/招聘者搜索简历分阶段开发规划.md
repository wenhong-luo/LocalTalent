# 招聘者「搜索简历」分阶段开发规划

> 本文件用于规划招聘者会员中心「搜索简历」能力。它不是已完成实现记录，也不是公共简历库准入文件。后续若启动开发，必须一次只执行一个提示词，并继续遵守 Phase III 边界。

## 当前状态补记

- **Prompt 1：搜索简历边界重签与数据源设计已完成**。
- **Prompt 2：后端受控搜索接口与六个下拉筛选已完成**。
- **Prompt 3：搜索简历列表页高保真 UI 已完成**。
- **Prompt 4：安全简历详情抽屉与举报简历已完成**。
- 已登记 `phase3.company_resume_search=false`，默认关闭；当前 `/company` 的「搜索简历」已具备 company 私有列表页、安全详情抽屉和举报弹窗，但仍不开放联系方式、完整简历下载或联系解锁。
- 允许的低风险子集仅限：company 私有域内、已认证企业、读取候选人主动授权发布的 `candidate_publish_snapshot` 安全快照。
- Prompt 2 已新增 company 私有后端 API：`GET /api/company/workbench/resume-search`，只返回发布快照安全摘要，并写 `field_access_log`。
- Prompt 3 已将 `/company` 的「搜索简历」风险池占位替换为高保真列表页，筛选、分页、空态和占位按钮均只读取安全快照摘要。
- Prompt 4 已新增 company 私有 `GET /api/company/workbench/resume-search/{snapshotId}` 安全详情接口、`POST /api/company/workbench/resume-search/{snapshotId}/reports` 举报接口和 `V0027__create_company_resume_snapshot_report.sql` 举报表；详情读取写 `field_access_log`，举报写 `audit_log` 并可创建风险审核摘要任务。
- 本轮仍未实现联系方式展示、完整简历下载、联系解锁、真实短信/微信、小程序/App 或会员支付。
- 下一轮如继续开发，必须从 **Prompt 5：下载/联系/职聊/面试邀请的门禁占位** 开始。

## 1. 总体结论

建议拆成 **6 个提示词** 完成。

当前 `/company` 的「搜索简历」仍是风险池占位，三期边界也明确禁止公共简历库、联系解锁和原始候选人数据公开。因此不能直接照搬 74CMS 的「完整简历库 + 下载联系方式」形态，需要先把它定义为：

- company 私有域内的受控简历搜索。
- 仅搜索候选人主动授权发布的 `candidate_publish_snapshot` 安全快照。
- 不直接展示手机号、邮箱、微信、附件、简历正文、联系方式。
- 不做联系解锁、真实简历下载、真实短信/微信/小程序/App 或会员支付。

如果只做领导演示 MVP，可以压缩成 4 轮：后端安全搜索、前端列表、详情/举报、安全收口。但如果要对齐 `docs/page/招聘者会员/搜索简历/` 中的 74CMS 页面资料，并为后续可控扩展留空间，推荐按 6 轮推进。

## 3. 六轮提示词拆分

### Prompt 1：搜索简历边界重签与数据源设计

状态：**已完成边界、数据源和文档/配置设计，不做完整 UI。**

目标：把「搜索简历」从笼统风险池拆出低风险子集，但只做边界、数据源和文档/配置设计，不做完整 UI。

- 新增规划中的 feature flag：`phase3.company_resume_search=false`，默认关闭。
- 明确数据源只允许 `candidate_publish_snapshot`，不得把 `candidate_resume` 原始 JSON 直接给企业前端。
- 设计安全快照字段扩展，用于支持筛选：地区、职位类别、学历、经验、性别、行业、专业、工作性质、期望薪资、简历标签、更新时间。
- 更新三期边界与提示词记录：允许的是「受控发布快照搜索」，不是公共简历库、联系解锁或下载完整简历。
- 保持 `/company` 入口默认占位，直到后端和前端验收通过。

验收重点：

- `phase3.company_resume_search` 默认关闭。
- 文档明确「搜索简历」只能读取发布快照。
- 公开门户不增加搜索简历入口。
- 联系方式、完整简历、附件、原始 JSON 仍禁止输出。

Prompt 1 数据源设计：

| 类型 | 字段/口径 | 说明 |
| --- | --- | --- |
| 唯一数据源 | `candidate_publish_snapshot` | 企业侧搜索不得读取 `candidate_resume` 原始 JSON；快照构建仍由 candidate 同意/发布流程负责。 |
| 可筛选字段 | 地区、职位类别、学历、经验、性别、行业、专业、工作性质、期望薪资、简历标签、更新时间 | 仅允许使用快照内已脱敏、已规范化、候选人同意发布的摘要字段。 |
| 可返回摘要 | 快照 ID、脱敏姓名、年龄段、学历、经验、期望职位、期望地区、期望薪资、技能摘要、更新时间 | 后续 API 响应只能返回卡片级安全字段，不返回联系方式或完整简历正文。 |
| 必须隐藏 | 手机号、邮箱、微信、附件 object key、简历正文、证据、原始 JSON、审核材料 | 禁止出现在接口响应、DOM、日志、审计摘要、URL、sitemap 或公开门户。 |
| 撤回联动 | consent 撤回、快照下线、候选人禁用后不可见 | 搜索列表、详情、推荐或缓存都必须按服务端状态排除。 |

### Prompt 2：后端受控搜索接口与六个下拉筛选

状态：**已完成后端受控搜索接口、快照安全筛选字段和权限种子；尚未开放前端页面。**

目标：实现 company 私有受控搜索 API。

- 新增 `GET /api/company/workbench/resume-search`。
- 要求 company 登录、企业资料完整、企业认证通过、feature flag 开启。
- 支持顶部筛选：地区、职位类别、关键词。
- 支持学历、工作经验快捷筛选。
- 支持「其他筛选」六个下拉：性别、简历标签、行业类别、所学专业、工作性质、期望薪资。
- 响应只返回安全卡片字段：快照 ID、脱敏姓名、年龄段或出生年推导摘要、学历、经验、期望职位、期望地区、期望薪资、技能摘要、更新时间。
- 搜索行为写审计或字段访问日志，证明企业搜索行为可追踪。

验收重点：

- 非 company、未认证企业、资料未完善企业不能访问。
- 撤回或下线快照不可搜索。
- 响应不含手机号、邮箱、微信、附件 key、简历正文、证据、原始 JSON。
- 六个下拉筛选均走服务端参数校验。

Prompt 2 实现记录：

| 类型 | 完成内容 | 边界 |
| --- | --- | --- |
| Feature flag | 继续使用 `phase3.company_resume_search=false`，并要求同时开启 `phase3.company_workbench=true`。 | 默认关闭；未开启时返回 `FEATURE_DISABLED_403`。 |
| 权限 | 新增 `company.resume-search.read`，仅授予 `company_admin`、`recruiter`。 | candidate/operator/auditor/open_client 不授予。 |
| 数据源 | 新增 `V0026__extend_publish_snapshot_resume_search_fields.sql`，为 `candidate_publish_snapshot` 补安全筛选生成列与索引。 | 搜索接口只查 `candidate_publish_snapshot`，不读取 `candidate_resume` 原始 JSON。 |
| API | 新增 company 私有 `GET /api/company/workbench/resume-search`。 | 只允许已认证 company 访问；未认证、非 company、无权限均拒绝。 |
| 筛选 | 支持 `keyword`、`city_code`、`category_code`、`education_code`、`experience_min/max`、`gender`、`resume_tag`、`industry_code`、`major`、`work_nature`、`expected_salary_code`、`updated_within`、`page`、`size`、`sort`。 | keyword 仅匹配脱敏姓名、期望职位、期望地区、技能摘要、专业等安全字段。 |
| 响应 | 返回快照 ID、脱敏姓名、年龄段、学历、经验、期望职位、期望地区、期望薪资、技能摘要、更新时间等安全摘要。 | 不返回 candidate_id、手机号、邮箱、微信、附件、证据、完整简历正文或原始 JSON。 |
| 日志 | 对返回快照写 `field_access_log`，`access_type=COMPANY_RESUME_SEARCH`。 | 只记录 snapshot ID、trace 和访问摘要，不记录关键词原文或敏感字段值。 |

### Prompt 3：搜索简历列表页高保真 UI

状态：**已完成 `/company` 搜索简历高保真列表页；只展示发布快照安全摘要，高风险入口仍为占位。**

目标：将 `/company` 中「搜索简历：风险池占位」替换为高保真搜索列表页。

- 参考 `搜索简历.jpeg` 实现顶部搜索条、学历/经验筛选、六个下拉筛选、简历卡片列表、分页、空态。

- 简历卡片只展示发布快照字段。

### Prompt 4：安全简历详情抽屉与举报简历

状态：**已完成安全详情抽屉、举报弹窗、举报表、详情/举报后端接口与审计/字段访问记录。**

目标：实现安全详情抽屉和举报弹窗。

- 参考 `电子名字或者下载简历显示.jpeg` 做简历详情抽屉。
- 详情页仍只读取发布快照安全字段。
- 联系方式区域显示：`请先通过合规流程申请查看联系方式`，不直接展示。
- 增加「举报简历」弹窗，参考 `点击举报.jpeg`：举报原因下拉、备注、提交。
- 举报可新增私有表或复用风险审核任务，但只存摘要，不存敏感明文。
- 导出、打印、下载简历、聊一聊先保留禁用或申请入口，不直接开放。

Prompt 4 实现记录：

| 类型 | 完成内容 | 边界 |
| --- | --- | --- |
| 详情接口 | 新增 company 私有 `GET /api/company/workbench/resume-search/{snapshotId}`。 | 只返回发布快照安全详情字段；不返回 `candidate_id`、手机号、邮箱、微信、附件 key、证据、完整简历正文或原始 JSON。 |
| 举报接口 | 新增 company 私有 `POST /api/company/workbench/resume-search/{snapshotId}/reports`，原因枚举固定为 `false_information`、`inappropriate_content`、`duplicate_snapshot`、`wrong_category`、`privacy_concern`、`other`。 | 举报只保存摘要，要求 `X-Trace-Id` 与 `X-Idempotency-Key`，不保存联系方式、附件、证据或原始简历。 |
| 数据表 | 新增 `V0027__create_company_resume_snapshot_report.sql`，保存举报 company、举报人、snapshot、原因、备注摘要、状态和时间戳。 | 不新增完整简历表，不读取 `candidate_resume` 原始 JSON。 |
| 风控联动 | 举报成功写 `audit_log`，详情读取写 `field_access_log`，并可创建 `risk_review` 摘要任务。 | 风险审核只记录摘要，不开放联系方式或下载能力。 |
| 前端 | `/company` 搜索简历页新增右侧详情抽屉和举报弹窗。 | `画像分析`、`下载简历`、`聊一聊` 继续占位，不触发真实能力。 |

验收重点：

- 详情抽屉不展示手机号、邮箱、附件、证据、完整简历正文或原始 JSON。
- 举报写审计，只记录快照 ID、原因、备注摘要和 trace。
- 企业 A 不能读取企业 B 的举报记录。
- candidate/operator/auditor/visitor 不能访问 company 简历搜索详情接口。

### Prompt 5：下载/联系/职聊/面试邀请的门禁占位

目标：把高风险入口做成明确门禁或申请入口，避免误认为已开放。

- 「下载简历」「聊一聊」「面试邀请」统一展示受控弹窗。
- 本轮不展示联系方式、不生成简历文件、不发短信微信。
- 可实现「访问申请」或「后续开放」弹窗，说明需要审批、套餐或风险准入。
- 面试邀请如接入，只能选择本企业已审核职位并生成站内流程，不接真实短信、微信、小程序或 App。
- 保持搜索简历不是联系解锁，不是公共简历库，不是商业化套餐能力。

验收重点：

- 下载、联系、职聊、面试邀请不会泄露联系方式。
- 不出现真实支付、真实短信、真实微信、小程序、App。
- 不创建导出旁路，不生成完整简历文件。
- 高风险入口都有清晰禁用/申请文案。

### Prompt 6：安全收口与验收矩阵

目标：为搜索简历建立可回归的安全证据。

- 覆盖 IDOR/BOLA：企业 A 不能查看企业 B 的搜索记录、举报记录、申请记录。
- 非 company 角色不能访问搜索简历接口。
- 候选人撤回同意后，搜索列表和详情立即不可见。
- 公开门户仍不出现搜索简历入口、联系方式、完整简历或原始候选人数据。
- 字段黑名单覆盖：手机号、邮箱、微信、附件 object key、简历正文、证据、原始 JSON、审核材料。
- 补前后端测试、边界脚本、三期文档收口记录。

验收重点：

- `check_phase3_boundary`、`check_phase3_ops`、`check_phase3_security_acceptance`、`check_portal_snapshot_fields`、`check_boundary` 通过。
- 后端搜索、详情、举报、申请入口均有鉴权、数据域、审计和 trace。
- 前端 DOM 字段黑名单通过。
- 三期文档明确：该能力是受控快照搜索，不是公共简历库。

## 5. 固定边界

- 只搜索 `candidate_publish_snapshot`。
- 不把 `candidate_resume` 原始 JSON 输出给企业前端。
- 不展示手机号、邮箱、微信、附件 key、证据、完整简历正文。
