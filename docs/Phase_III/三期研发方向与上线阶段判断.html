一、当前阶段判断

当前结论很清楚：一期完成“合规安全底座 + 招聘基础闭环”，二期完成“高保真人才网公开门户 + 演示收口”。三期不应继续堆更多静态高保真页面，也不应立刻做公共简历库、联系解锁、会员支付、真实外部能力。

一期解决能不能安全地做；
二期解决像不像一个人才网；
三期要解决能不能让真实用户用起来。

二、三期主要开发内容

1. 真实用户闭环生产化

补齐求职者注册登录后的真实使用能力：简历编辑、简历预览、隐私与发布状态、投递记录、职位收藏、搜索/职位订阅、面试状态、站内通知、同意/撤回状态展示与便捷操作。

状态必须由后端返回，撤回后人才服务区和所有公开可见入口必须联动不可见。

2. 企业招聘工作台生产化

补企业资料维护、认证材料提交、职位创建/编辑/提交审核/下线、投递池筛选、投递详情、企业备注、阶段流转、面试邀约、导出申请入口。

企业只能访问和操作本企业数据，未认证企业不得上线职位。

3. 运营后台生产化

补企业审核、职位审核、内容管理、活动/招聘会管理、风险审核、推荐位配置、审计中心查询 UI 和运营首页 overview。

operator 可写，auditor 只读；所有审核、下线、软删除、推荐位调整、风险处理都必须写审计。

4. 公开门户从演示数据转真实运营数据

首页推荐、找工作、找企业、人才服务区、招聘会、政策资讯、HR 工具箱从“演示可看”变成“后台可配置、前台可索引、状态可控、SEO 可运营”。

5. 账号与安全体验补齐

补正式登录体验、注册成功分流、找回密码或重置密码、会话过期提示、无权限态、错误提示 trace_id 和验证码占位治理。

6. 生产化工程收口

补预发配置、生产环境变量说明、数据库备份恢复演练、对象存储生命周期、日志脱敏检查、监控告警、上线/回滚 Runbook 和真实数据前验收清单。

三、不建议三期做的内容

  • 不做公共简历库：人才服务区仍只展示发布快照白名单。
  • 不做受控找人才和联系解锁：必须先走 Phase II Step6 风险准入规则。
  • 不做真实支付和会员商业化：套餐、权益、支付、发票、退款应放到商业化专项期。
  • 不接真实短信、微信、小程序、App、地图、视频、直播：三期可保留入口或占位，但不接供应商。
  • 不接真实 ATS/RPA/受控候选池同步：对接接口继续 stub，真实同步单独立项评审。

五、推荐阶段划分

三期：真实业务试运营闭环

让求职者、企业、运营人员真的能用。重点是简历编辑、职位发布、投递处理、面试邀约、内容运营、站内通知、后台审核和生产化配置。

四期：正式上线治理与运营稳定性

从试运营走向公网正式使用。重点是监控告警、备份恢复、性能压测、SEO 运营、权限复核、日志脱敏、客服/问题处理、合规材料、上线演练和回滚演练。

五期：风险池能力专项评估

选择性进入商业化和外部能力。只有通过 Step6 准入后，才考虑受控找人才、联系解锁、会员权益、真实支付、短信/微信、小程序/App、地图/视频/直播、ATS/RPA 等。

  1. 三期边界与试运营蓝图。

七、Assumptions

  • 默认“上线使用”按“真实求职者和企业可小范围试运营”理解，因此建议三期后可进入灰度试运营。
  • 默认不为了上线牺牲一期/二期硬边界：人才服务区只展示发布快照，对接接口仅 stub，原始候选人数据不得公开，权限/字段级权限/审计必须服务端收口。
  • 如果目标改为“三期后直接公网正式上线”，三期范围需要显著扩大，并加入压测、监控、备份、合规审查、客服运营和正式发布演练。
  • 如果目标改为“快速商业化”，则必须先完成 Step6 风险准入，不能把会员、支付、联系解锁直接塞进三期主线。
