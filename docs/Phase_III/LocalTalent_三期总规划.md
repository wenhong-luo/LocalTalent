## 执行摘要

结论先行：你这份三期草案的方向是对的，核心目标也对——不是继续堆静态页面，也不是直接冲商业化，而是把一二期从“可演示”推进到“可小范围灰度试运营”。一期已经锁定了不可打破的底线：当前只建设地方人才服务平台；人才服务区只能展示发布快照；不得公开原始候选人数据；ATS 与受控候选池只能保留对接 stub 与兼容位；RBAC、数据域、字段级权限、同意/撤回、审计必须在服务端收口。

三期草案与《三期研发方向与上线阶段判断》对“三期要解决什么”的判断一致：目标是让真实求职者、企业和运营人员在受控范围内跑通闭环，并继续明确排除公共简历库、受控找人才、联系解锁、会员支付、真实短信/微信/小程序/App、真实地图/视频/直播、真实 ATS/受控候选池同步。

真正的问题不在“大方向冲突”，而在“工程默认项不够细”：三期草案还没有把认证体系默认方案、灰度容量目标、导出频控、Compose→K8s 迁移路径、站内通知默认渠道、门户真实数据驱动的字段白名单与搜索索引边界、回滚条件和阶段闸门写成可以直接投喂 Codex 的执行口径。因此，本报告给出的不是概念建议，而是一份落库为 `docs/Phase_III/LocalTalent_三期总规划.md` 的三期作战手册。

**推荐总策略**  
三期建议拆成 **7 个阶段**、约 **10–11 周**推进：先做三期基线和认证，再做求职者闭环、企业工作台、运营后台与门户运营化，最后做审计/导出/风控收口和灰度试运营演练。这样最符合你目前的一二期沉淀，也最不容易让 Codex 偷跑到风险池能力。

**一句话边界**  
三期可以做“真实试运营闭环”，但**不能**借试运营名义，绕过一期/二期已经冻结的公开数据边界、字段级权限、导出审批、审计留痕和风险池准入规则。

## 草案对照与必改项

结论是：**未发现与你一期 Step1–Step4、二期红线相冲突的“直接硬冲突”**；但有若干处必须补充或收紧，否则三期文档还不足以直接指导开发与上线判断。

| 草案条目 | 结论 | 原因 | 建议修改 |
| --- | --- | --- | --- |

| “广告位” | 需改写 | 二期允许的是静态运营广告位/推荐位，不是商业投放、结算、外部广告网络接入。 | 统一改成：**运营位/推荐位**；若保留“广告位”一词，必须同时注明“不做广告售卖与结算”。 |

## 三期总规划

### 目标、范围、排除项与成功口径

| 维度 | 三期规划 |
| --- | --- |
| 总目标 | 把一二期的“可演示系统”推进为“可让真实求职者、企业、运营人员小范围灰度试运营”的业务系统。 |
| 核心范围 | 求职者真实闭环、企业工作台生产化、运营后台生产化、公开门户真实数据驱动、认证与身份映射、工程治理与灰度上线演练。 |
| 明确不做 | 公共简历库、受控找人才、联系方式解锁、会员商业化/真实支付、真实短信/微信/小程序/App、真实地图/视频/直播、真实 ATS/RPA/受控候选池同步、AI 原始候选人数据处理。 |
| 继承原则 | 一期的 RBAC + 数据域 + 字段级权限 + 导出审批 + 审计 + 对接 stub 全部继承；二期的 Next/SSR 公开门户与 noindex/SEO 口径全部继承，不回退为纯演示页。 |
| 阶段成功定义 | 本阶段代码能跑、测试通、没越界；且本阶段新增对象级授权、字段级权限、审计留痕、导出审批、撤回下线回归全部通过。 |
| 三期完成定义 | 真实候选人可编辑简历、投递、收藏、订阅、查看状态并撤回同意；真实企业可认证、发职位、处理投递、邀约面试、申请导出；operator 可审核与配置；auditor 只读；公开门户由真实审核状态和后台配置驱动；灰度运行与回滚手册齐全。 |

### P3-1 认证与身份映射

- **主控说明：**OIDC 只负责认证和基础 claims，授权仍留在 LocalTalent 服务端执行；不可把角色判断外包给 IdP。OIDC 作为预发/灰度默认入口，本地账号 + JWT 只保留给 dev/test 与应急白名单。OIDC 是建立在 OAuth 2.0 之上的身份层；Authorization Code Flow 避免把令牌直接暴露给用户代理；OAuth 安全最佳实践要求公有客户端必须使用 PKCE，机密客户端也推荐使用。
- **本轮目标：**接入 OIDC 登录、回调、内部用户映射、角色映射、登出、失败态与 fallback 策略；本地 JWT 登录保留但默认为关闭。
- **禁止事项：**禁止做 SCIM、企业目录同步、社交登录、短信/微信登录；禁止把 operator / auditor 权限交给前端或 IdP claim 直接决定。
- **输出要求：**OIDC 配置、回调路由、内部账号映射表设计说明、角色映射规则、环境变量与测试用例。
- **验收要求：**预发模式下 OIDC 正常登录并映射到内部账号；dev 模式下本地 JWT 可用；prod/grey 模式下本地登录默认关闭；日志不出现 token、secret、明文 claims。

### P3-3 企业招聘工作台生产化

- **主控说明：**企业中心属于 company 私有域；企业公开主页与企业中心必须彻底隔离；对象 ID 一律做 BOLA/IDOR 测试。OWASP 把对象级授权缺失与对象属性级授权缺失都列为 API 高风险。
- **本轮目标：**企业资料维护、认证材料提交、职位创建/编辑/送审/下线、投递池、阶段流转、面试邀约、导出申请。

## 默认技术与治理建议

### 认证与授权默认值

| 场景 | 默认建议 | 说明 |
| --- | --- | --- |
| 本地开发 | 本地账号 + JWT | 与你一期现状保持一致，便于本地自测与 Codex 迭代。 |
| 预发 / 灰度 | OIDC 作为主认证入口；Authorization Code + PKCE | 认证走 OIDC，内部平台再映射到 candidate/company/operator/auditor 本地角色与数据域。 |
| 生产应急 | 本地登录默认关闭，仅 break-glass 白名单启用 | 避免双入口长期并存带来审计与权限混乱。 |
| 授权归属 | 仍由 LocalTalent 后端执行 RBAC + 数据域 + 字段级权限 | IdP 只负责“你是谁”，平台后端负责“你能看什么、能做什么”。 |

推荐这样做的原因很直接：OIDC 本身是 OAuth 2.0 之上的身份层，Code Flow 能减少令牌暴露面；而你的一期/二期已经把对象级访问、字段级权限、导出审批和审计归于服务端，三期不应该把这些判断分散到前端或外部 IdP。

### 部署与运维默认值

| 层级 | 默认建议 | 原因 |
| --- | --- | --- |
| 开发 / 本地验证 | Docker Compose | 启动快、依赖清晰，适合多轮 Codex 开发和本地演练。 |
| 预发 / 单机灰度 | 继续 Compose 或单节点容器编排 | 在三期只追求灰度试运营时，先保证稳定和回滚简单。 |
| 三期末进阶 | K8s 仅承接前端 SSR/BFF、后端 API、worker 等无状态工作负载 | 让应用层先具备 HPA / Ingress / 滚动发布能力，降低把状态层一起迁移的风险。 |
| MySQL / Redis / MinIO | 优先外置或保持现状；三期不强制同阶段重构 | 灰度试运营阶段，状态层迁移风险通常大于收益。 |

Compose 的价值在于用单个 YAML 管理多容器应用并保证环境一致；Kubernetes 的 HPA 则适合在观测到 CPU/内存等资源压力时自动横向扩容工作负载。把二者放在“先后分层”的关系里，而不是同阶段强行替换，是三期最稳妥的默认路径。

### 灰度容量默认目标

下表是**三期灰度试运营**的工程默认值，不是现有文档已经锁死的硬指标。它们服务的是“真实但小范围”的试运营，而不是公网正式运营。

| 指标 | 建议目标 | 口径 | 理由 |
| --- | --- | --- | --- |
| 公开门户总读流量 | 峰值 120 QPS | 首页、找工作、找企业、人才服务区、招聘会、资讯等公开页总和 | 足以覆盖小范围真实试运营与基本市场投放，不逼迫你提前进入四期大规模容量治理。 |
| 私有中心 / 后台混合流量 | 峰值 25 QPS | candidate / company / operator / auditor 的登录后接口总和 | 更贴近灰度试运营的真实写操作密度。 |
| 认证相关流量 | 峰值 12 QPS | 登录、回调、token 刷新等 | 灰度名单有限，认证峰值不会像公网大促那样尖锐。 |
| 导出申请提交 | 峰值 2 QPS | 申请入口而非下载 | 导出本来就必须审批，写峰值应低。 |
| 导出频次 | 每企业默认 3 次/日，全站 30 次/日 | 超过则触发人工复核或限流 | 灰度期用配额与审批一起压住误用和批量抓取。 |
| 导出并发 | 2 个 worker | 按导出任务并发，不按下载链接并发 | 灰度期优先稳定、可追溯，不追大吞吐。 |
| 活跃发布快照规模 | 1 万条 | `candidate_publish_snapshot` 当前对外可见快照 | 足够覆盖小范围地方人才网试运营，同时便于索引与撤回下线联动控制。 |
| 历史快照保留规模 | 5 万条 | 含下线/撤回历史版本 | 既保留追溯能力，又不把三期变成大规模归档系统。 |
| 撤回下线传播 SLA | 中位 < 60 秒，最慢 < 5 分钟 | 撤回后从公开门户、搜索索引、缓存与 sitemap 联动下线 | 这是三期最关键的合规与信任指标之一。 |

### 数据与隐私护栏

一期和二期最核心的经验，三期不能丢：**公开层、搜索层、缓存层、日志层、导出层、对接层都不能成为原始候选人数据的旁路出口**。其中对象级授权、属性级授权和“把敏感字段先下发再在前端隐藏”的做法，都是明确高风险。

- 人才服务区、公开搜索、sitemap、缓存与 analytics 只读 **发布快照白名单**。
- 手机号、邮箱、简历正文、附件对象 key、同意证据、原始 JSON、聊天记录、审核材料、营业执照附件不得进入公开层。
- 高敏字段查看与导出必须写 `field_access_log`；审批、通过、驳回、下载、撤回、下线必须写 `audit_log`；trace 要能串起来。
- 搜索索引中不放原始简历文本；缓存值中不放 S3/S4 高敏字段；日志中不放 token、secret、证据明文。
- 撤回触发后，快照、搜索索引、缓存和公开页必须联动下线。

### 导出审批与审计流默认值

| 环节 | 默认动作 | 审计要求 |
| --- | --- | --- |
| 申请 | 继续走 `export_apply` | 记录申请人、对象范围、模板、理由、时间 |
| 审批 | 由有权 operator 审批；auditor 只读 | 记录审批结论、驳回原因、审批人 |
| 生成 | 后台 worker 生成导出文件 | 记录生成结果、trace\_id、对象数量、敏感模板校验结果 |
| 下载 | 一次性、短时预签名链接；建议 15–30 分钟 TTL | 记录链接发放、下载确认、过期/撤销 |
| 高敏字段查看 | 必须经过字段级权限判定 | 写 `field_access_log` |

这也是为什么三期不应该把“导出可用”理解成“导出放宽”。三期应该做的是把导出审批、频控、下载链接生命周期和审计留痕做得更像生产系统，而不是更随意。

## 上线判断与 KPI

### 何时进入下一阶段 / 何时允许上线

1. **范围冻结通过：**当前阶段的需求清单、禁止事项、feature flag 和上线对象已经固定，没有把风险池能力混入主线。
2. **边界守门通过：**`check_boundary` / `check_phase3_boundary` / 快照字段白名单扫描全部通过。
3. **认证与授权通过：**OIDC 或本地 dev 登录工作正常；跨对象、跨企业、跨角色访问全部被后端拒绝；auditor 仍只读。
4. **关键闭环通过：**当前阶段对应的真实业务闭环已跑通，而且没有依赖人工改库或临时脚本才能完成。
5. **导出与审计通过：**导出审批、字段访问留痕、trace 串联和日志脱敏全部通过，且没有“看得到但不留痕”的高敏字段读取。
6. **门户与快照边界通过：**人才服务区、公开搜索、sitemap、缓存、埋点都没有原始候选人数据旁路。
7. **工程演练通过：**备份恢复、回滚、灰度开关、容量烟测、告警联通都跑过一遍。
8. **复盘结果通过：**本阶段 P0/P1 问题已清零或明确降级，不把已知阻断问题带入下一阶段。

## 风险清单

| 风险 | 等级 | 表现 | 缓解动作 |
| --- | --- | --- | --- |
| 风险池能力偷跑 | 高 | Codex 顺手把联系解锁、公共简历库、短信/微信之类做出来。 | 阶段脚本强扫关键字；每阶段执行提示词重复“禁止事项”；未过边界扫描不得合并。 |
| OIDC 接入拖期 | 中高 | 回调、角色映射、登出与本地 fallback 处理不清。 | 把身份认证阶段单独拆出来；先完成 OIDC 与本地 JWT 双轨，再进入业务闭环阶段。 |
| 公开层泄露原始候选人字段 | 高 | 搜索、缓存、埋点或推荐位不小心带出原始数据。 | 快照白名单测试、DOM 黑名单测试、索引白名单、日志脱敏扫描一起上。 |
| 企业工作台出现 BOLA/IDOR | 高 | 跨企业访问投递、面试、导出对象。 | 对象级授权测试变成阶段必测项；所有“按 ID 查对象”的接口统一进授权中间层。 |
