# LocalTalent Phase 3 Step2：求职者真实闭环生产化

## 开发前必读卡

- **文件定位**：三期第二份 Step 文档；用于把“求职者中心演示能力”设计成后续蓝图和 Prompt 可继承的真实试运营闭环。
- **输入资料**：`00_PHASE_III_BOUNDARY.md`、`LocalTalent_phase3_step1.md`、`LocalTalent_三期总规划.md`、`三期研发方向与上线阶段判断.md`、Phase I/II 求职者相关基线。
- **本轮目标**：冻结 candidate 在三期灰度试运营中必须具备的简历、投递、收藏、订阅、通知、同意/撤回与发布状态能力。
- **禁止越界点**：不写代码，不新增迁移，不生成三期蓝图或提示词，不实现公共简历库、受控找人才、联系解锁、真实短信、真实微信、小程序、App 或任何外部推送能力。
- **输出边界**：只定义求职者真实闭环的产品范围、状态口径、字段边界、审计要求和验收关注点。
- **下一步影响**：本文件稳定后，下一份建议文档是 `LocalTalent_phase3_step3.md：企业招聘工作台生产化`。

**统一硬边界提醒**：人才服务区仅展示发布快照；求职者中心是 candidate 本人私有域；不得公开原始候选人数据；同意、撤回、数据域、字段级权限、审计和 trace 必须在服务端收口。

## 1. Step1 继承结论

Step1 已经冻结三期目标：把一二期的可演示系统推进为 **小范围真实试运营**。Step2 只承接其中的 candidate 闭环，不处理企业工作台、运营后台、商业化或风险池能力。

| Step1 结论 | Step2 继承方式 |
| --- | --- |
| 三期目标是灰度试运营，不是公网正式运营。 | Step2 只要求 candidate 闭环真实可用，暂不追求大规模流量和完整客服体系。 |
| 人才服务区仅展示发布快照。 | 简历编辑可处理本人原始字段，但公开展示仍只能来自 `candidate_publish_snapshot`。 |
| 状态必须由服务端返回。 | 前端不得推断 resume、application、favorite、subscription、notification、consent、publish 状态。 |
| 三期默认通知渠道是站内通知。 | 收藏、订阅、投递状态、面试邀约只进入站内通知，不接真实短信、真实微信、小程序或 App。 |
| 风险池能力不得偷跑。 | 公共简历库、受控找人才、联系解锁、AI 处理原始候选人数据不进入 Step2。 |

Step2 的关键问题不是“页面有没有”，而是 candidate 是否能在受控权限下完成真实求职动作，并且撤回同意后所有公开可见入口联动下线。

## 2. 一二期求职者能力基线

一二期已经完成了一部分求职者能力，三期 Step2 不应重复造轮子。

| 能力 | 已有基线 | Step2 增量 |
| --- | --- | --- |
| 注册登录 | 已有本地账号/JWT 与登录注册 UI。 | 三期后续按总规划接入认证与身份映射；Step2 只定义 candidate 业务闭环。 |
| 求职者中心 overview | 已有 `GET /api/candidate/center/overview` 只读联调。 | 扩展为真实中心首页，汇总简历、投递、收藏、订阅、通知与发布状态。 |
| 同意/撤回 | 已有 consent、control profile、publish snapshot、撤回下线链路。 | 强化便捷撤回、发布状态说明、撤回后公开不可见回归。 |
| 人才服务区 | 已有发布快照列表、高保真筛选、字段黑名单。 | 继续只读发布快照，不新增公共简历库或查看简历能力。 |
| 投递与面试 | 已有投递、企业投递池、面试签到基础闭环。 | candidate 侧补投递记录、面试/签到状态回流和站内通知。 |
| 公开门户入口 | 二期已有找工作、职位详情、登录入口。 | candidate 从职位详情进入投递、收藏、订阅后回流求职者中心。 |

Step2 允许在文档中规划“简历编辑使用本人原始字段”，但必须同时写清：这些字段不得进入公开门户、搜索索引、日志、缓存、sitemap 或导出旁路。

## 3. Step2 总目标

三期求职者真实闭环的目标是：

```text
candidate 能完成：维护简历 → 预览确认 → 投递职位 → 查看投递/面试状态 → 收藏职位 → 订阅搜索 → 接收站内通知 → 同意/撤回发布。
```

达到灰度试运营口径时，candidate 至少应能做到：

1. 登录后进入求职者中心，看到真实的简历完成度、发布状态、投递状态、收藏/订阅数量和未读通知。
2. 编辑本人简历并预览，保存后状态由服务端返回。
3. 从职位详情发起投递，之后在投递记录中看到状态回流。
4. 收藏职位、取消收藏、创建搜索订阅、取消订阅，动作具备幂等和本人数据域限制。
5. 收到站内通知，能查看并标记已读。
6. 便捷同意或撤回发布，撤回后人才服务区、搜索索引、缓存、sitemap 和公开入口联动不可见。

## 4. 求职者中心信息架构

Step2 冻结求职者中心的信息架构，不直接展开成接口或数据库 schema。

| 模块 | 目标能力 | 状态与边界 |
| --- | --- | --- |
| 中心首页 | 汇总简历、投递、面试、收藏、订阅、通知、发布状态。 | 只展示本人摘要；状态来自服务端。 |
| 简历编辑 | 编辑基础资料、教育经历、工作经历、技能、求职意向、附件占位。 | 可使用本人原始字段；不得进入公开层。 |
| 简历预览 | candidate 预览本人简历与公开快照差异。 | 预览页必须区分“本人可见字段”和“公开快照字段”。 |
| 投递记录 | 查看投递职位、企业、状态、面试邀约、签到状态。 | 只能访问本人 application；覆盖 IDOR/BOLA。 |
| 职位收藏 | 收藏、取消收藏、查看收藏列表。 | 不触发企业通知，不变成联系入口。 |
| 搜索/职位订阅 | 保存关键词、城市、职类等低风险条件。 | 默认只生成站内通知，不做短信/微信/App 推送。 |
| 站内通知 | 查看系统通知、投递状态、面试邀约、订阅提醒。 | 不接真实短信、真实微信、小程序、App。 |
| 隐私与发布状态 | 查看同意状态、发布快照状态、撤回入口。 | 撤回必须便捷，且公开可见性联动下线。 |

求职者中心可以更像“工作台”，但绝不能变成“公开简历库的后台入口”。candidate 自己能看见的，不代表游客、企业或运营都能看见。

## 5. 状态机与服务端口径

Step2 的核心纪律是：**状态以服务端为准**。

| 状态域 | 服务端应返回的状态口径 | 前端禁止行为 |
| --- | --- | --- |
| resume | 草稿、待完善、已完善、不可发布、可发布、更新时间。 | 禁止按字段数量自行推断最终可发布状态。 |
| application | 已投递、已查看、邀约面试、已签到、已拒绝、已撤回或关闭。 | 禁止前端按时间或按钮点击推断投递阶段。 |
| favorite | 已收藏、未收藏、收藏总数。 | 禁止只用 localStorage 作为真实状态。 |
| subscription | 已订阅、暂停、已取消、最近触发时间。 | 禁止直接触发短信、微信、小程序、App。 |
| notification | 未读、已读、归档、通知类型。 | 禁止把通知当作外部消息发送记录。 |
| consent | 已同意、已撤回、未同意、版本、更新时间。 | 禁止前端推断 consent_status。 |
| publish | 已发布、已下线、不可发布、生成中、生成失败。 | 禁止把原始简历当公开发布内容。 |

任何写动作成功后，页面应重新读取服务端状态，而不是用写接口返回值在前端拼一个“看起来对”的状态。

## 6. 字段与权限边界

### 6.1 本人私有域

candidate 在本人私有中心内，可以维护和查看本人求职必要字段，例如基础资料、教育经历、工作经历、技能、求职意向、附件占位和投递记录。

要求：

- 必须 JWT candidate 登录。
- 必须强制当前 candidate 只能访问本人数据。
- 所有含对象 ID 的简历、投递、收藏、订阅、通知接口必须覆盖 IDOR/BOLA。
- 非 candidate、未登录、跨 candidate 访问必须返回统一无权限响应与 trace。

### 6.2 公开域

公开门户、人才服务区、搜索索引、日志、缓存、sitemap、analytics 只能使用公开白名单。

禁止进入公开域的字段和内容：

- 手机号、邮箱、身份证、联系方式。
- 简历正文、附件对象 key、证据。
- `base_profile_json`、`education_json`、`experience_json`、`skills_json` 原始 JSON。
- 聊天记录、通知明细、投递备注、面试评价。
- 审核材料、运营备注、风控结论。

人才服务区公开字段继续限定为发布快照白名单：`snapshot_id/display_name_masked/city_code/category_code/skills_summary/experience_years/updated_at`。

### 6.3 企业与运营可见性

Step2 不新增企业找人才、联系解锁或公共简历库能力。企业仍只能通过合法投递链路看到与本企业职位相关的投递信息；运营如需查看风险或审计数据，必须由后续运营后台 Step 设计并继承脱敏与审计。

## 7. 审计、幂等与撤回联动

Step2 后续实现时，以下动作必须具备 trace、幂等或审计口径。

| 动作 | 要求 |
| --- | --- |
| 简历保存 | 写接口需要幂等或防重复提交；记录更新时间和 trace。 |
| 简历发布状态变化 | 写 `audit_log`；若涉及字段读取，按字段级权限写 `field_access_log`。 |
| 同意 | 幂等；写 consent 历史、control profile、audit_log；生成发布快照。 |
| 撤回 | 幂等；写 consent 历史、control profile、audit_log；下线发布快照。 |
| 投递 | 继承一期投递审计和重复投递冲突处理。 |
| 收藏/取消收藏 | 幂等；仅影响本人私有列表。 |
| 订阅/取消订阅 | 幂等；仅生成站内通知，不外发短信/微信/App。 |
| 通知已读 | 幂等；只影响本人通知状态。 |

撤回联动必须覆盖：

1. `candidate_control_profile` 当前状态。
2. `candidate_consent` 历史记录。
3. `candidate_publish_snapshot` 下线。
4. 人才服务区不可见。
5. 搜索索引、缓存、sitemap、公开推荐位移除或失效。
6. 审计日志可按 trace 串联。

三期灰度口径建议沿用总规划：撤回下线传播中位小于 60 秒，最慢小于 5 分钟；这是灰度试运营的信任底线。

## 8. 自动化验收关注点

Step2 后续拆 Prompt 时，每轮至少需要覆盖以下验收点。

| 验收点 | 关注内容 |
| --- | --- |
| 字段黑名单 | 公开 DOM、响应、搜索索引、日志、缓存、sitemap 不含原始候选人数据字段。 |
| 撤回不可见 | 同意后可见、撤回后人才服务区不可见。 |
| 本人数据域 | candidate 只能访问本人简历、投递、收藏、订阅、通知。 |
| 非 candidate 无权限 | company/operator/auditor/未登录访问 candidate 私有接口均被拒绝。 |
| IDOR/BOLA | 所有对象 ID 接口覆盖跨用户阻断。 |
| 状态服务端驱动 | 前端不推断 consent、publish、application、resume 状态。 |
| 站内通知不外发 | 不存在真实短信、真实微信、小程序、App 调用。 |
| 审计与 trace | 同意、撤回、发布状态变化、敏感字段访问可追踪。 |

建议后续实现阶段保留或新增以下测试方向：

- `CandidateResumeFlowIT`：简历保存、预览、本人访问、字段黑名单。
- `CandidateApplicationStatusIT`：投递记录、面试状态、跨 candidate 阻断。
- `CandidateFavoriteSubscriptionIT`：收藏、订阅、幂等、站内通知。
- `CandidateConsentPublishRegressionIT`：同意、撤回、快照上下线、公开不可见。
- 前端 candidate 中心状态测试：loading、empty、error、unauthorized、retrying、consented、revoked、not_publishable。

这些测试名称只是规划建议，不是本轮要创建的代码文件。

## 9. Step2 结论与下一步

Step2 结论：

- 三期求职者闭环应优先让 candidate 真实可用，而不是继续堆公开页面。
- 简历编辑与预览属于本人私有域，可以处理本人原始字段，但不得进入公开域。
- 收藏、订阅、通知只做站内闭环；真实短信、真实微信、小程序、App 继续暂缓。
- 同意/撤回和发布快照仍是人才服务区的合规核心。
- 所有状态必须由服务端返回，前端不得推断。
- 所有对象 ID 必须覆盖 IDOR/BOLA。

建议下一份文档：

```text
LocalTalent_phase3_step3.md：企业招聘工作台生产化
```

Step3 应聚焦企业中心，不建议直接跳到运营后台、门户运营化、商业化或风险池能力：

1. 企业资料维护。
2. 企业认证材料提交。
3. 职位创建、编辑、送审、下线。
4. 投递池筛选、详情、备注、阶段流转。
5. 面试邀约。
6. 导出申请入口。
7. 跨企业数据域与 IDOR/BOLA 阻断。
