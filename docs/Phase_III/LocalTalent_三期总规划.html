执行摘要

结论先行：你这份三期草案的方向是对的，核心目标也对——不是继续堆静态页面，也不是直接冲商业化，而是把一二期从“可演示”推进到“可小范围灰度试运营”。一期已经锁定了不可打破的底线：当前只建设地方人才服务平台；人才服务区只能展示发布快照；不得公开原始候选人数据；ATS 与受控候选池只能保留对接 stub 与兼容位；RBAC、数据域、字段级权限、同意/撤回、审计必须在服务端收口。fileciteturn0file13 fileciteturn0file14 fileciteturn0file8 fileciteturn0file9

三期草案与《三期研发方向与上线阶段判断》对“三期要解决什么”的判断一致：目标是让真实求职者、企业和运营人员在受控范围内跑通闭环，并继续明确排除公共简历库、受控找人才、联系解锁、会员支付、真实短信/微信/小程序/App、真实地图/视频/直播、真实 ATS/受控候选池同步。fileciteturn0file0 fileciteturn0file1

真正的问题不在“大方向冲突”，而在“工程默认项不够细”：三期草案还没有把认证体系默认方案、灰度容量目标、导出频控、Compose→K8s 迁移路径、站内通知默认渠道、门户真实数据驱动的字段白名单与搜索索引边界、回滚条件和阶段闸门写成可以直接投喂 Codex 的执行口径。因此，本报告给出的不是概念建议，而是一份落库为 docs/Phase_III/LocalTalent_三期总规划.html 的三期作战手册。

推荐总策略
三期建议拆成 7 个阶段、约 10–11 周推进：先做三期基线和认证，再做求职者闭环、企业工作台、运营后台与门户运营化，最后做审计/导出/风控收口和灰度试运营演练。这样最符合你目前的一二期沉淀，也最不容易让 Codex 偷跑到风险池能力。
一句话边界

一期和二期最核心的经验，三期不能丢：公开层、搜索层、缓存层、日志层、导出层、对接层都不能成为原始候选人数据的旁路出口。其中对象级授权、属性级授权和“把敏感字段先下发再在前端隐藏”的做法，都是明确高风险。fileciteturn0file14 fileciteturn0file9 citeturn1search5turn1search8turn1search1

上线判断与 KPI

何时进入下一阶段 / 何时允许上线

  1. 范围冻结通过：当前阶段的需求清单、禁止事项、feature flag 和上线对象已经固定，没有把风险池能力混入主线。
  2. 边界守门通过：check_boundary / check_phase3_boundary / 快照字段白名单扫描全部通过。
  3. 认证与授权通过：OIDC 或本地 dev 登录工作正常；跨对象、跨企业、跨角色访问全部被后端拒绝；auditor 仍只读。
  4. 关键闭环通过：当前阶段对应的真实业务闭环已跑通，而且没有依赖人工改库或临时脚本才能完成。
  5. 导出与审计通过：导出审批、字段访问留痕、trace 串联和日志脱敏全部通过，且没有“看得到但不留痕”的高敏字段读取。
  6. 门户与快照边界通过：人才服务区、公开搜索、sitemap、缓存、埋点都没有原始候选人数据旁路。
  7. 工程演练通过：备份恢复、回滚、灰度开关、容量烟测、告警联通都跑过一遍。
  8. 复盘结果通过：本阶段 P0/P1 问题已清零或明确降级，不把已知阻断问题带入下一阶段。
