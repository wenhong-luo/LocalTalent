# LocalTalent 三期研发方向与上线阶段判断

## 开发前必读卡

- **文件定位**：三期规划起点文件，用于判断一二期完成后，下一阶段应做什么，以及距离真实上线还差几期。
- **事实基线**：一期已完成合规安全底座和招聘基础闭环；二期已完成高保真人才网公开门户和演示收口。
- **三期目标**：把“一二期可演示系统”推进为“可让真实求职者、企业、运营人员小范围试用的业务系统”。
- **禁止越界点**：三期不直接做公共简历库、受控找人才、联系解锁、会员支付、真实短信/微信/小程序/App、真实地图/视频/直播、真实 ATS/RPA 或受控候选池同步。
- **上线判断口径**：内部演示、灰度试运营、公网正式运营、完整商业化人才网是四个不同目标，不能混为一谈。

**统一硬边界提醒**：人才服务区仅展示发布快照；对接接口仅 stub；不得输出原始候选人数据；同意、撤回、数据域、字段级权限必须在服务端收口。

## 一、当前阶段判断

研读 Phase I 与 Phase II 文档后，当前结论如下：

- 一期完成了 **合规安全底座 + 招聘基础闭环**。
- 二期完成了 **高保真人才网公开门户 + 演示收口**。
- 三期不应该继续堆更多静态高保真页面。
- 三期也不应该立刻做公共简历库、联系解锁、会员支付、真实短信/微信/地图/直播/ATS 等高风险能力。
- 三期核心应是让系统从“能演示”进入“能小范围真实试运营”。

换句话说：

```text
一期解决能不能安全地做；
二期解决像不像一个人才网；
三期要解决能不能让真实用户用起来。
```

## 二、三期主要开发内容

### 1. 真实用户闭环生产化

三期应补齐求职者注册登录后的真实使用能力：

- 简历编辑。
- 简历预览。
- 隐私与发布状态。
- 投递记录。
- 职位收藏。
- 搜索/职位订阅。
- 面试状态。
- 站内通知。
- 同意/撤回状态展示与便捷操作。

关键约束：

- 状态仍必须由后端返回，前端不得推断。
- 撤回后人才服务区和所有公开可见入口必须联动不可见。
- 求职者中心是本人私有域，不得把原始候选人数据带入公开门户。

### 2. 企业招聘工作台生产化

三期应把企业后台从“可看演示”推进为“可真实招聘”：

- 企业资料维护。
- 企业认证材料提交。
- 职位创建、编辑、提交审核、下线。
- 投递池筛选。
- 投递详情。
- 企业备注。
- 阶段流转。
- 面试邀约。
- 导出申请入口。

关键约束：

- 企业只能访问和操作本企业数据。
- 未认证企业不得上线职位。
- 企业投递池不是公共简历库。
- 导出仍必须走审批、字段级权限、MinIO 短期下载和审计链。

### 3. 运营后台生产化

三期应让运营人员能支撑真实内容和审核工作：

- 企业审核。
- 职位审核。
- 内容管理。
- 活动/招聘会管理。
- 风险审核。
- 推荐位配置。
- 审计中心查询 UI。
- 运营首页 overview。

关键约束：

- `operator` 可写，`auditor` 只读。
- 所有审核、下线、软删除、推荐位调整、风险处理都必须写审计。
- 审计中心响应必须脱敏。

### 4. 公开门户从演示数据转真实运营数据

二期公开门户已经像人才网，三期要让这些页面可运营：

- 首页推荐从静态/演示数据转为运营配置。
- 找工作使用真实职位发布和审核状态。
- 找企业使用真实企业认证和公开资料。
- 人才服务区继续只读发布快照。
- 招聘会、政策资讯、HR 工具箱由后台内容配置驱动。
- SEO 页面状态、上下线状态、空态和错误态可控。

关键约束：

- 公开门户只展示公开字段白名单。
- 下线、撤回、驳回、未审核内容不得公开可见。
- 私有中心与后台页面默认 noindex。

### 5. 账号与安全体验补齐

三期可以保留一期本地账号/JWT 底座，但需要补真实用户体验：

- 正式登录体验。
- 注册成功后的清晰分流。
- 找回密码或重置密码流程。
- 会话过期提示。
- 无权限态。
- 错误提示带 `trace_id`。
- 验证码占位治理。

关键约束：

- 真实短信、微信、小程序、App 仍不接。
- 前端只做体验，不能绕过服务端权限。

### 6. 生产化工程收口

三期应补齐进入灰度试运营前的工程治理：

- 预发环境配置。
- 生产环境变量说明。
- 数据库备份恢复演练。
- MinIO / 对象存储生命周期策略。
- 日志脱敏检查。
- 监控告警。
- 上线/回滚 Runbook。
- 真实数据导入前验收清单。
- 灰度试运营问题处理流程。

## 三、不建议三期做的内容

以下能力不建议纳入三期主线：

- **公共简历库**：人才服务区仍只展示发布快照白名单。
- **受控找人才和联系解锁**：必须先走 Phase II Step6 风险准入规则。
- **真实支付和会员商业化**：套餐、权益、支付、发票、退款应放到商业化专项期。
- **真实短信、微信、小程序、App、地图、视频、直播**：三期可保留入口或占位，但不接供应商。
- **真实 ATS/RPA/受控候选池同步**：对接接口继续 stub，真实同步单独立项评审。
- **AI 处理原始候选人数据**：必须等字段裁剪、脱敏、供应商审查和人工复核方案成熟后再评估。

## 四、还需要几期才能上线使用

这里必须区分“上线使用”的含义。

| 目标 | 当前判断 | 还需要几期 |
| --- | --- | --- |
| 内部演示 / 客户看样 | 二期已经基本可用，只需保持演示数据、文档和部署稳定。 | 0 期，当前即可演示。 |
| 小范围试运营，允许真实求职者和企业入驻 | 三期完成真实求职者中心、企业工作台、运营后台配置、通知和生产化 Runbook 后，可以灰度试运营。 | 1 期。 |
| 公网正式运营，可长期承接真实业务 | 需要三期试运营闭环，再做四期生产安全、运营治理、性能压测、监控告警、合规材料、备份和客服工单。 | 2 期。 |
| 接近 74cms 的完整商业化能力 | 需要三期试运营、四期正式上线治理、五期风险池专项与商业化能力评估。 | 3 期以上。 |

当前推荐口径：

```text
三期后可进入灰度试运营；
四期后再考虑公网正式运营；
五期以后再评估商业化和高风险外部能力。
```

## 五、推荐阶段划分

### 三期：真实业务试运营闭环

目标是让求职者、企业、运营人员真的能用，而不是只看 demo。

重点：

- 简历编辑。
- 职位发布。
- 投递处理。
- 面试邀约。
- 内容运营。
- 站内通知。
- 后台审核。
- 生产化配置。

建议验收标准：

- 真实 candidate 能完成注册、简历维护、投递、查看状态、撤回同意。
- 真实 company 能完成注册、认证、职位发布、处理投递、邀约面试。
- operator 能审核企业、职位、内容、活动，并查看审计。
- auditor 只能只读。
- 所有公开页面仍通过字段白名单和发布状态检查。

### 四期：正式上线治理与运营稳定性

目标是从试运营走向公网正式使用。

重点：

- 监控告警。
- 备份恢复。
- 性能压测。
- SEO 运营。
- 权限复核。
- 日志脱敏。
- 客服/问题处理。
- 合规材料。
- 上线演练。
- 回滚演练。

建议验收标准：

- 有预发环境和生产环境配置。
- 有数据库与对象存储备份恢复演练记录。
- 有压测报告和容量评估。
- 有上线/回滚 Runbook。
- 有真实数据前的安全合规检查单。

### 五期：风险池能力专项评估

目标是选择性进入商业化和外部能力。

只有通过 Step6 准入后，才考虑：

- 受控找人才。
- 联系解锁。
- 会员权益。
- 真实支付。
- 短信/微信/小程序/App。
- 地图/视频/直播。
- ATS/RPA。
- 受控候选池同步。
- AI 推荐或模型处理。

建议验收标准：

- 每个风险能力都有业务目的、数据来源、合法性依据、字段白名单、同意/撤回方案、服务端权限、审计、测试和回滚方案。
- 未通过准入的风险能力不得进入开发。

## 六、三期建议拆分方向

后续正式规划三期时，建议不要一次写成超大 Prompt，而是像一期和二期一样拆成逐轮闭环。

建议三期 Prompt 方向：

1. **三期边界与试运营蓝图**：冻结三期目标、上线口径、试运营边界和禁区。
2. **求职者简历编辑与预览**：简历表单、附件占位、隐私与发布状态。
3. **求职者收藏、订阅与站内通知**：职位收藏、搜索订阅、站内消息。
4. **企业资料与认证材料完善**：企业资料、认证材料、公开主页联动。
5. **职位发布与编辑闭环**：职位创建、编辑、提交审核、下线。
6. **投递池工作台增强**：筛选、备注、阶段流转、拒绝/邀约。
7. **面试邀约与站内通知闭环**：企业邀约、求职者查看、通知留痕。
8. **运营内容与推荐位配置**：首页推荐、频道内容、招聘会配置。
9. **审计中心 UI 与风险审核**：审计查询、风险处理、operator/auditor 分权。
10. **三期试运营演示数据、Runbook 与总闸门**：真实试运营前收口。

这些只是方向，不是最终 Prompt。正式进入三期前，仍应先写 `docs/Phase_III/00_PHASE_III_BOUNDARY.md`、三期蓝图和三期研发提示词。

## 七、Assumptions

- 默认“上线使用”按“真实求职者和企业可小范围试运营”理解，因此建议三期后可进入灰度试运营。
- 默认不为了上线牺牲一期/二期硬边界：人才服务区只展示发布快照，对接接口仅 stub，原始候选人数据不得公开，权限/字段级权限/审计必须服务端收口。
- 如果目标改为“三期后直接公网正式上线”，三期范围需要显著扩大，并加入压测、监控、备份、合规审查、客服运营和正式发布演练。
- 如果目标改为“快速商业化”，则必须先完成 Step6 风险准入，不能把会员、支付、联系解锁直接塞进三期主线。
