# LocalTalent Phase 3 Step6：灰度试运营工程治理与上线演练

## 开发前必读卡

- **文件定位**：三期第六份 Step 文档；用于把 Step1-Step5 已规划的业务闭环、安全收口和运营边界，落到灰度试运营前必须具备的工程治理、上线演练和退出条件。
- **输入资料**：`00_PHASE_III_BOUNDARY.md`、`LocalTalent_phase3_step1.md` 至 `LocalTalent_phase3_step5.md`、`LocalTalent_三期总规划.md`、`三期研发方向与上线阶段判断.md`、Phase I/II Runbook、Testing、CI、MinIO、导出、审计和边界脚本基线。
- **本轮目标**：冻结三期灰度试运营工程治理口径，包括预发环境、灰度配置、feature flag、备份恢复、对象存储生命周期、日志脱敏、监控告警、容量烟测、上线/暂停/回滚 Runbook、灰度 KPI 和试运营退出条件。
- **禁止越界点**：不写代码，不新增迁移，不生成三期蓝图或提示词，不创建部署脚本，不把灰度试运营写成公网正式上线，不引入真实短信、真实微信、小程序、App、真实地图、视频、直播、真实 ATS/RPA 或风险池能力。
- **输出边界**：只定义后续实现阶段必须满足的工程治理、演练材料、验收门槛和阻断规则；K8s、监控、压测、备份恢复脚本均只作为后续实现输出物规划。
- **下一步影响**：本文件稳定后，Step1-Step6 即可作为三期蓝图和 Prompt 拆解的上游依据；下一份建议文档是 `三期研发实施蓝图与任务拆解.md/html`。

**统一硬边界提醒**：三期仍是小范围真实试运营；人才服务区仅展示发布快照；原始候选人数据不得公开；导出必须审批；审计必须脱敏；对接接口继续 stub；风险池能力不得绕过准入。

## 1. Step1-Step5 继承结论

Step1 冻结三期目标为 **小范围真实试运营**；Step2 收口 candidate 真实闭环；Step3 收口 company 招聘工作台；Step4 收口运营后台和门户运营化；Step5 收口审计、导出、风控与安全检查单。Step6 只做工程治理与上线演练，不扩展业务功能。

| 继承结论 | Step6 继承方式 |
| --- | --- |
| candidate/company/operator/auditor 闭环已经规划完成。 | 灰度演练必须覆盖四类角色的关键路径，而不是只测公开门户。 |
| 门户由真实状态和后台配置驱动。 | 容量烟测、缓存、sitemap 和监控必须覆盖首页、找工作、找企业、人才服务区、招聘会、资讯等公开页。 |
| Step5 已冻结安全检查单。 | Step6 必须把安全检查单接入上线前闸门、暂停条件和回滚流程。 |
| 风险池能力不得偷跑。 | 灰度开关、Runbook、演示数据和故障处理都不得临时放开风险池能力。 |
| 三期不是公网正式运营。 | 工程目标是灰度试运营可控、可暂停、可回滚，不是一次性生产大上线。 |

Step6 的关键问题是：当真实候选人、真实企业和真实运营人员开始小范围使用时，系统是否能被稳定运行、快速定位、及时暂停、安全回滚，并用证据证明边界仍然有效。

## 2. 一二期与三期工程治理基线

一二期已经完成合规底座和高保真门户，三期 Step1-Step5 已完成试运营闭环设计。Step6 不重做这些能力，而是把工程治理补成灰度前的统一闸门。

| 能力 | 已有基线 | Step6 收口口径 |
| --- | --- | --- |
| 本地依赖 | Docker Compose 可拉起 MySQL、Redis、MinIO。 | 继续作为开发和单机预发基座，灰度前必须有明确环境变量和启动顺序。 |
| CI/闸门 | 已有 `run_all`、边界脚本、字段黑名单脚本、Phase II planning/demo 闸门。 | 三期后续应补 `check_phase3_boundary`、`check_phase3_ops` 或同等闸门。 |
| Next/SSR 门户 | 二期完成高保真公开门户和 SEO 基础。 | 灰度烟测必须覆盖 SSR 构建、公开页面可访问、sitemap 不含私有和撤回对象。 |
| 导出与 MinIO | 已有审批、异步生成、短期预签名下载和审计链。 | 灰度前补对象生命周期、导出文件保留、过期清理和异常告警口径。 |
| 审计与 trace | 已有 audit、field access、open api 三类日志查询与脱敏。 | 灰度运行必须能按 trace 串联关键路径，并有异常审计告警入口。 |
| 备份与回滚 | Phase I/II Runbook 已有发布、回滚、备份、监控口径。 | Step6 要求真实演练备份恢复、应用回滚、灰度暂停和 roll-forward 修复。 |

三期工程治理的核心不是“工具堆满”，而是让最小灰度系统具备可操作、可观测、可恢复、可证明。

## 3. Step6 总目标

Step6 的目标可以概括为：

```text
灰度试运营前，系统必须可启动、可灰度、可观测、可备份、可恢复、可暂停、可回滚、可退出。
```

达到 Step6 口径时，至少应能做到：

1. 本地、预发、灰度环境的配置差异清晰，敏感配置不进入仓库。
2. feature flag 能控制三期新增能力的打开、暂停和回滚。
3. 数据库和对象存储有备份、恢复、保留、清理和负责人。
4. 日志脱敏、监控指标、告警门槛、故障排查路径清晰。
5. 公共门户、私有中心、后台、导出和审计关键路径具备 smoke test。
6. 灰度容量烟测覆盖公开读流量、登录态接口、导出 worker 和撤回下线传播。
7. 上线、暂停、回滚 Runbook 可被新人照着执行。
8. 灰度 KPI、扩大范围条件、停止条件和试运营退出条件明确。

## 4. 预发环境与灰度配置

三期默认环境分层如下。

| 环境 | 目标 | 允许能力 | 禁止事项 |
| --- | --- | --- | --- |
| local | 开发与自测 | Compose、本地账号/JWT、演示数据、低风险脚本。 | 不接真实外部供应商，不存放生产数据。 |
| staging | 集成与演练 | 与灰度接近的配置、预发域名、脱敏样本数据、备份恢复演练。 | 不用生产明文数据，不绕过审批和审计。 |
| gray | 小范围真实试运营 | 真实候选人、企业、运营人员按白名单进入；feature flag 分批开启。 | 不直接变成公网正式运营，不开放风险池能力。 |
| production-ready | 四期正式上线准备 | 压测、监控、客服、合规材料、供应商治理和生产发布流程。 | 不在三期 Step6 中承诺完成。 |

### 4.1 配置口径

后续实现阶段应准备以下配置材料：

- `.env.phase3.example`：只保留示例值，不保存真实密钥。
- 环境变量清单：数据库、Redis、MinIO/S3、JWT/OIDC、Next API base URL、日志级别、feature flag、导出 TTL、导出频控。
- 灰度账号白名单：candidate、company、operator、auditor 的试运营名单。
- 数据域隔离说明：灰度数据不得混入生产未知来源数据。
- 开关默认值：三期新增功能默认关闭，逐项打开。

### 4.2 Compose 与 K8s 边界

三期默认：

- 本地和单机预发继续使用 Docker Compose 或单节点容器编排。
- K8s 只作为无状态应用层的进阶选项：Next SSR/BFF、后端 API、worker。
- MySQL、Redis、MinIO 不在三期同阶段强制迁移到 K8s。
- 不在没有备份恢复演练和回滚演练前进入灰度。

这样做的目的很朴素：先把真实业务闭环跑稳，再谈更复杂的基础设施编排。别让基础设施在灰度第一天扮演“惊喜嘉宾”。

## 5. feature flag、灰度发布、暂停与回滚

三期新增能力必须能被开关控制，且每个开关都要有负责人、默认值、影响范围和回滚方式。

| flag 示例 | 控制范围 | 默认值 | 回滚动作 |
| --- | --- | --- | --- |
| `phase3.candidate.resume_edit` | 求职者简历编辑与预览。 | 关闭 | 关闭入口，保留只读 overview。 |
| `phase3.candidate.favorite_subscription` | 收藏、订阅、站内通知入口。 | 关闭 | 关闭入口，保留投递记录。 |
| `phase3.company.job_manage` | 企业职位创建、编辑、送审、下线。 | 关闭 | 关闭写入口，保留职位只读列表。 |
| `phase3.company.application_workspace` | 投递池阶段流转和面试邀约。 | 关闭 | 冻结写动作，保留只读投递池。 |
| `phase3.admin.portal_ops` | 推荐位、内容、活动、风险审核生产化。 | 关闭 | 回退到二期静态/只读展示。 |
| `phase3.export.quota` | 导出频控、配额、风险提示。 | 开启 | 不建议关闭；异常时暂停导出申请。 |

### 5.1 灰度发布步骤

1. 确认本轮范围、禁止事项和 feature flag 列表冻结。
2. 在 staging 打开目标 flag，跑 smoke、边界、字段黑名单、IDOR/BOLA、导出和审计回归。
3. 做备份恢复演练和应用回滚演练。
4. 在 gray 环境对白名单用户打开 5% 或指定账号范围。
5. 观察 5xx、P95、登录成功率、撤回下线传播、导出审批合规率、审计覆盖率。
6. 无 P0/P1 问题后扩大到 20%、50%、100% 白名单。
7. 每次扩大范围前都记录负责人、时间、观察窗口和回滚点。

### 5.2 暂停条件

出现以下任一情况，应暂停扩大灰度，必要时关闭相关 flag：

- 公开门户 API 5xx 持续超过 0.5%。
- BOLA/IDOR、字段泄露、导出绕行、审计缺失等 P0/P1 安全问题。
- 撤回下线传播超过最大 5 分钟仍未收敛。
- 导出链接过期或二次发放规则失效。
- 关键角色无法登录或身份分流错误。
- 备份恢复失败或回滚演练失败。

### 5.3 回滚原则

- 应用代码可回滚到上一版本。
- 数据库结构优先 roll-forward 修复，不直接回滚已上线结构。
- feature flag 是第一回滚手段；应用版本回滚是第二手段；数据修复必须保留审计记录。
- 对外部能力仍为 stub 或占位，不应出现供应商级回滚复杂度。

## 6. 数据库备份恢复演练

灰度前必须证明数据库可备份、可恢复、可验证。

| 项目 | Step6 口径 |
| --- | --- |
| 备份范围 | MySQL 主库、Flyway schema history、关键业务表、审计日志表。 |
| 备份频率 | 灰度期至少每日全量备份；关键上线窗口前手动备份。 |
| 恢复演练 | staging 环境至少完成一次从备份恢复到可启动应用的演练。 |
| 校验方式 | 表数量、核心记录数、Flyway 版本、登录、门户、导出、审计查询 smoke test。 |
| 保留周期 | 灰度期建议至少保留 7 天，正式运营前再按合规要求扩展。 |
| 负责人 | 明确备份执行人、恢复执行人、复核人和失败升级路径。 |

### 6.1 数据库恢复后必须验证的路径

- `/health` 可访问。
- 首页、找工作、找企业、人才服务区可访问。
- candidate 登录、company 登录、operator 登录可用。
- 同意/撤回状态与发布快照一致。
- 企业职位可见性与审核状态一致。
- 导出申请、审批、生成和审计记录可查询。
- `audit_log`、`field_access_log`、`open_api_log` 未丢失关键 trace。

## 7. 对象存储生命周期与导出文件保留策略

对象存储是导出链路和附件治理的关键点。三期仍以 MinIO/S3 兼容策略为默认口径。

| 对象类型 | 保留策略 | 风险控制 |
| --- | --- | --- |
| 导出文件 | 短期保留，建议 7 天内清理；下载链接 15-30 分钟 TTL。 | 未审批不可生成；过期不可下载；文件 key 不公开。 |
| 简历附件 | 仅本人私有域和授权流程可访问。 | 不进入公开门户、搜索、缓存或日志。 |
| 审核材料 | 仅 operator 授权查看，auditor 只读脱敏视图。 | 不公开营业执照附件、证据或后台材料。 |
| 演示素材 | 本地演示可用占位图。 | 不混入真实敏感材料。 |

### 7.1 导出文件治理

- 继续使用 `export_apply` 审批链。
- 下载链接只能短期有效。
- 文件生成、链接发放、下载确认和过期拒绝均需可审计。
- 导出对象 key 不得进入公开 DOM、日志、open_api_log 或审计响应明文。
- 清理任务失败应有告警或人工检查入口。

## 8. 日志脱敏、监控告警与故障排查

三期灰度需要最小可观测性，不追求一次性搭建完整平台。

### 8.1 日志脱敏规则

日志、错误信息、审计响应、open_api_log、埋点和调试输出不得包含：

- 手机号、邮箱、身份证、联系方式。
- 简历正文、附件对象 key、证据、原始候选人 JSON。
- 营业执照附件、认证材料、审核材料、后台备注。
- 报名名单、签到证据、参会个人明细。
- password、secret、token、signature、nonce 明文。

### 8.2 最小监控指标

| 指标 | 灰度关注点 |
| --- | --- |
| 应用健康 | `/health`、后端启动、Next SSR 启动、worker 存活。 |
| 错误率 | 公开门户 5xx、私有中心 5xx、后台 5xx、导出失败率。 |
| 延迟 | 公开页 P95、登录 P95、导出申请 P95、审计查询 P95。 |
| 安全事件 | AUTHZ_403 异常飙升、IDOR 拦截、字段黑名单扫描失败。 |
| 导出链路 | 申请数、审批数、生成成功/失败、下载链接发放、过期拒绝。 |
| 撤回下线 | 撤回动作数量、传播耗时、超时未下线对象。 |
| 对接 stub | 签名错误、重放、幂等冲突、重试队列积压。 |

### 8.3 最小告警门槛

- 公开门户 API 5xx 连续 5 分钟超过 0.5%。
- 登录失败率异常上升或 OIDC 回调失败。
- 导出生成失败连续出现或导出 worker 队列积压。
- 审计写入失败、字段访问日志写入失败。
- 撤回下线传播超过 5 分钟。
- 边界脚本或字段黑名单扫描失败。

告警不是为了吓人，是为了别让凌晨三点的你成为唯一监控系统。

## 9. 容量烟测与关键路径压测

三期容量目标是灰度试运营，不是正式公网大流量。

| 指标 | 建议目标 | 口径 |
| --- | --- | --- |
| 公开门户总读流量 | 峰值 120 QPS | 首页、找工作、找企业、人才服务区、招聘会、资讯等公开页总和。 |
| 私有中心/后台混合流量 | 峰值 25 QPS | candidate/company/operator/auditor 登录后接口总和。 |
| 认证相关流量 | 峰值 12 QPS | 登录、回调、token 刷新等。 |
| 导出申请提交 | 峰值 2 QPS | 申请入口，不含下载链接访问。 |
| 导出并发 | 2 个 worker | 按导出任务并发控制。 |
| 活跃发布快照规模 | 1 万条 | 当前对外可见 `candidate_publish_snapshot`。 |
| 历史快照保留规模 | 5 万条 | 含下线和撤回历史版本。 |
| 撤回下线传播 SLA | 中位 < 60 秒，最慢 < 5 分钟 | 从撤回到公开门户、搜索、缓存、sitemap 不可见。 |

### 9.1 Smoke test 覆盖路径

- 首页、找工作、职位详情、找企业、企业主页。
- 人才服务区列表、筛选、空态。
- 招聘会、政策资讯、HR 工具箱、更多服务。
- `/auth/login`、`/auth/register`。
- `/candidate/center` overview。
- `/company` dashboard。
- `/admin` dashboard。
- 导出申请、审批状态查询、下载链接发放拒绝/成功场景。
- 审计 trace 查询。

### 9.2 压测禁止事项

- 禁止用真实候选人敏感字段做压测样本。
- 禁止绕过登录、审批、字段级权限做“方便压测”。
- 禁止把压测数据写入公网可见入口。
- 禁止压测真实短信、微信、小程序、App、地图、视频、直播或 ATS/RPA。

## 10. 上线、暂停、回滚 Runbook

后续实现阶段应生成可执行 Runbook。Step6 先冻结 Runbook 内容要求。

### 10.1 上线前检查

| 检查项 | 要求 |
| --- | --- |
| 范围冻结 | 本次灰度包含哪些 flag、页面、接口和用户范围。 |
| 禁止事项确认 | 风险池能力未混入灰度主线。 |
| 测试通过 | 后端、前端、边界脚本、字段黑名单、三期专项脚本通过。 |
| 备份完成 | 数据库和对象存储备份完成并记录位置。 |
| 恢复演练 | staging 恢复演练通过。 |
| 回滚演练 | flag 关闭、应用回滚、roll-forward 修复路径演练通过。 |
| 监控告警 | 健康、5xx、延迟、导出、审计、撤回下线告警可用。 |
| 人员排班 | 产品、研发、运营、值守和决策人明确。 |

### 10.2 暂停步骤

1. 停止扩大灰度名单。
2. 关闭相关 feature flag。
3. 暂停高风险写动作，例如导出申请、职位发布、推荐位调整。
4. 保留只读入口，避免用户完全失联。
5. 记录事件、trace、影响范围、临时处置和负责人。
6. 完成根因确认后再决定恢复、回滚或 roll-forward。

### 10.3 回滚步骤

1. 确认问题级别和影响范围。
2. 先关闭相关 feature flag。
3. 若仍未恢复，回滚应用版本。
4. 数据库问题优先新增向前修复迁移，不直接修改旧迁移。
5. 若涉及数据泄露或权限问题，立即冻结相关入口并保留审计证据。
6. 回滚后执行 smoke test 和关键路径验证。
7. 复盘记录必须写入故障记录模板。

## 11. 灰度 KPI 与试运营退出条件

### 11.1 灰度 KPI

| KPI | 建议门槛 | 说明 |
| --- | --- | --- |
| 公开门户 API 5xx | < 0.5% | 持续超标则暂停扩大灰度。 |
| 公开页 P95 | < 500ms | 按服务端公开接口口径。 |
| 登录成功率 | > 99% | 覆盖本地账号或 OIDC 灰度入口。 |
| 撤回下线传播 | 中位 < 60 秒，最慢 < 5 分钟 | 三期最关键合规指标之一。 |
| 导出审批合规率 | 100% | 所有下载必须关联审批单和生成记录。 |
| 审计覆盖率 | 100% | 审批、导出、撤回、高敏字段查看可追踪。 |
| 越权缺陷 | P0 = 0，P1 = 0 | BOLA、IDOR、字段泄露均为阻断级。 |
| 真实灰度闭环数 | 至少 20 位候选人、10 家企业、3 名运营跑通 | 避免三期变成“更完整的演示”。 |

### 11.2 扩大灰度条件

- 当前灰度范围连续一个观察窗口内无 P0/P1 问题。
- 5xx、延迟、登录成功率、撤回传播、导出审批和审计覆盖达标。
- 运营人员能处理审核、风险、内容、活动和导出审批。
- candidate/company 真实闭环不依赖人工改库或临时脚本。
- 暂停和回滚演练可执行。

### 11.3 退出试运营条件

灰度试运营可以判定完成，当且仅当：

1. candidate、company、operator、auditor 四类角色均完成真实闭环。
2. 公开门户真实状态驱动稳定，SEO 与 sitemap 不含私有或撤回对象。
3. 审计、导出、撤回、IDOR/BOLA、字段黑名单回归稳定。
4. 备份恢复、回滚演练、告警联通和容量烟测都有记录。
5. 风险池能力仍未偷跑。
6. 产品、技术、运营共同确认可进入四期正式上线治理。

如果以上条件未满足，三期应继续灰度修复，而不是为了“阶段完成感”提前宣布公网正式运营。

## 13. Step6 结论与下一步

Step6 结论：

- 三期 Step 文档已从产品目标、求职者闭环、企业工作台、运营后台、安全收口，推进到工程治理和上线演练。
