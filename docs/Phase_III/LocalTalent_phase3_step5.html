一、Step1-Step4 继承结论

Step1 已经冻结三期目标为 小范围真实试运营；Step2 收口 candidate 真实闭环；Step3 收口 company 工作台；Step4 收口运营后台和门户运营化。Step5 只做安全闭环，不新增业务范围。

继承结论Step5 继承方式
candidate 可维护简历、投递、收藏、订阅、通知、同意/撤回。所有 candidate 私有对象必须覆盖本人数据域、字段黑名单和撤回下线回归。
company 可认证、发职位、处理投递、面试邀约、申请导出。所有 company 对象必须覆盖跨企业 IDOR/BOLA、导出审批和投递池字段裁剪。
operator 可审核、配置、发布、风险处理。所有 operator 写动作必须写 audit_log，并可按 trace 串联。
auditor 只读。auditor 写接口必须被后端拒绝，审计中心响应必须脱敏。
门户由真实状态和后台配置驱动。公开门户、推荐位、sitemap、缓存、搜索索引只能展示合法公开对象。
风险池能力不得偷跑。Step5 必须明确阻断规则和扫描/验收口径。

二、一二期安全能力基线

能力已有基线Step5 收口口径
同意/撤回已有 consent、control profile、publish snapshot、撤回下线。三期所有公开入口、推荐位、缓存、搜索索引和 sitemap 必须继承撤回不可见。
发布快照人才服务区唯一公开数据源为 candidate_publish_snapshot。公开字段继续限定白名单，禁止回退到原始简历或候选人表。
字段级权限已有 FieldPolicyEngine、field_access_log、字段裁剪测试。导出、投递池、审计、风险审核、运营查看均必须明确字段裁剪或脱敏。
导出审批已有 export_apply、operator 审批、异步生成、MinIO 短期下载。三期不得新增绕过审批的下载、报表、缓存或运营导出路径。
审计中心已有 audit、field access、open api 三类日志查询和脱敏。三期关键动作必须能按 trace 聚合，审计响应不得泄露敏感明文。
对接接口已有签名、幂等、trace、open_api_log、重试队列 stub。三期继续 stub，不接真实 ATS/RPA/受控候选池。
边界脚本已有一期/二期字段黑名单、规划闸门、demo 验收脚本。三期应新增或继承边界脚本，阻断风险池和敏感字段泄露。
Step5 不是“补更多功能”，而是把安全证据准备好，让灰度试运营能被信任。

三、Step5 总目标

灰度前确保：可追踪、可裁剪、可撤回、可阻断、可回滚、可证明。
  1. 核心业务动作都能通过 trace_id 串联审计。
  2. 导出仍必须经过申请、审批、生成、字段裁剪、短期下载和审计。
  3. 候选人撤回后，公开快照、门户、推荐位、搜索、缓存、sitemap 全部不可见或失效。
  4. candidate、company、operator、auditor、visitor 的对象访问边界清晰且可测试。
  5. 公开页面、日志、缓存、索引、导出、审计响应均不泄露原始候选人数据和审核材料。
  6. 风险池能力被文档、脚本、测试和代码审查共同阻断。
  7. 进入 P3-6 前有一张明确的灰度试运营安全检查单。

四、审计 trace 串联

三期灰度试运营必须能回答“谁在什么时候对什么对象做了什么，为什么，产生了什么影响”。

日志类型记录对象Step5 收口要求
audit_log关键业务写动作、审核、发布、下线、撤回、风险处理、导出审批。必须包含 trace_id、操作者、角色、动作、业务对象、前后状态或摘要。
field_access_log敏感字段读取、导出字段读取、高风险材料查看。必须包含 trace_id、字段名、访问类型、操作者、角色、业务对象。
open_api_log对接接口 stub 调用、签名、幂等、重试摘要。只能记录摘要、hash、耗时、状态和 trace，不记录敏感明文。

必须串联的场景

  • candidate 同意、发布快照、撤回、快照下线。
  • candidate 简历保存、发布状态变化、订阅变更、通知读取。
  • company 认证提交、职位送审、职位下线、投递阶段流转、面试邀约、导出申请。
  • operator 企业审核、职位审核、内容发布、活动发布、推荐位调整、风险处理、导出审批。
  • openapi stub 请求、重放拦截、幂等冲突、重试入队。
审计中心是“可追溯视图”，不是敏感明文查看后门。

五、导出审批与字段裁剪回归

环节必须满足
申请企业只能为本企业数据提交 export_apply；申请写 audit_log。
审批operator 审批通过或驳回；auditor 不得审批；驳回原因入审计。
生成审批通过后异步生成；生成成功/失败写 audit_log。
字段裁剪导出字段必须经过字段级权限；无权字段脱敏或不导出；读取写 field_access_log。
存储文件写入对象存储，延续 MinIO / S3 兼容策略。
下载只发放短期下载链接；未审批、驳回、未生成、二次发放或过期必须拒绝。
追踪申请、审批、生成、下载链接发放、字段访问必须可按 trace 串联。

禁止绕行

后台“报表”直接下载候选人明细。
运营备注或审计详情输出候选人字段。
搜索索引、缓存、日志、埋点携带候选人原始字段。
以“调试接口”“灰度工具”“临时脚本”导出生产数据。
将导出文件长久在线公开。

六、撤回下线传播

撤回同意是三期信任底线。撤回后必须停止继续发布，且公开侧应尽快不可见。

传播目标Step5 收口要求
candidate_control_profile当前状态更新为撤回或不可发布。
candidate_consent保留撤回历史和原因。
candidate_publish_snapshot快照下线或不可公开查询。
人才服务区列表、筛选、详情或推荐入口均不可见。
首页/推荐位若推荐位引用该快照，必须自动隐藏或失效。
搜索索引不得继续返回撤回候选人快照。
缓存必须失效或刷新，不得继续输出旧快照。
sitemap不得收录撤回后的候选人公开入口。
审计撤回动作和传播结果可按 trace 查询。
三期灰度口径建议沿用总规划：撤回下线传播中位小于 60 秒，最慢小于 5 分钟。无法满足时，必须在灰度说明中标注风险和人工处理流程。

七、IDOR/BOLA 与字段黑名单总验收

7.1 对象级授权

身份必须阻断
visitor访问 candidate/company/admin 私有接口。
candidate A访问 candidate B 的简历、投递、收藏、订阅、通知、同意状态。
company A访问 company B 的资料、职位、投递池、面试、导出申请。
operator绕过业务规则直接公开未审核对象。
auditor调用任何写接口。
openapi client绕过签名、nonce、时间窗、幂等或传输敏感原文。

7.2 字段黑名单

以下字段不得出现在公开响应、公开 DOM、sitemap、缓存、搜索索引、日志、埋点、open_api_log、审计响应和未授权导出中：

mobile / email / phone / contact
resume_body / attachment_object_key / evidence
password_hash / 原始候选人 JSON
营业执照附件、审核材料、后台备注
报名名单、签到证据
secret / token / password / signature / nonce 明文
字段黑名单不得通过“前端隐藏”实现，必须由服务端裁剪、脱敏或不返回。

八、风险池准入与阻断

风险能力Step5 结论
公共简历库禁止；人才服务区仍只展示发布快照。
受控找人才暂缓；必须先走风险准入。
联系解锁暂缓；涉及联系方式、高敏字段、权益和审计。
会员商业化/真实支付暂缓；涉及资金流、对账、退款、发票和风控。
真实短信/微信/小程序/App暂缓；三期默认站内通知。
真实地图/视频/直播暂缓；只允许占位或风险池入口。
真实 ATS/RPA/受控候选池同步暂缓；对接接口继续 stub。
AI 处理原始候选人数据暂缓；需最小必要、脱敏、供应商评审和人工复核。

禁止伪装上线

不得把风险能力伪装成前端隐藏按钮、运营备注、临时报表、搜索索引、缓存字段、活动报名、会员权益占位、对接 stub 的“测试数据”或管理后台高级筛选。

十一、Step5 结论与下一步

  • 三期灰度前必须能证明系统可追踪、可裁剪、可撤回、可阻断、可回滚、可证明。
  • 审计中心必须脱敏，trace 必须能串联核心业务动作。
  • 导出必须继续继承审批、字段级权限、短期下载和审计链。
  • 撤回下线必须覆盖人才服务区、推荐位、搜索、缓存和 sitemap。
  • IDOR/BOLA、字段黑名单和 auditor 只读必须成为总闸门。
  • 风险池能力不得借灰度、报表、推荐位、缓存或运营工具偷跑。
