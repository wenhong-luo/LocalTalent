# LocalTalent Phase-2 Step 3：求职者与企业转化闭环

## 开发前必读卡

- **文件定位**：二期第三份规划文档，只冻结公开门户到求职者中心、企业中心的转化路径、状态机、边界与验收关注点。
- **实现约束**：本文件是规划输入，不是开发 Prompt；不新增代码、不新增接口 schema、不新增数据库迁移、不生成二期蓝图或提示词。
- **禁止越界点**：禁止公共简历库；禁止受控找人才和联系解锁提前展开；禁止真实短信、微信、小程序、App、地图、视频、直播、支付、ATS/RPA 或受控候选池同步。
- **下游影响**：为后续求职者中心、企业中心体验增强提供产品口径；后续 Step4 聚焦“求职者中心与企业中心能力细化”。
- **自动化验收关注点**：后续实现必须继续守住人才服务区、发布快照、原始候选人数据、同意、撤回、对接接口、数据域、字段级权限。

**统一硬边界提醒**：人才服务区仅展示发布快照；对接接口仅 stub；不得输出原始候选人数据；同意、撤回、数据域、字段级权限必须在服务端收口。

Step3 目标不是增加更多频道，而是把公开门户的关键入口接到已有中心能力：

| 转化对象 | 起点 | 目标中心 | 二期 Step3 要冻结的闭环 |
| --- | --- | --- | --- |
| 求职者 | 首页、找工作、职位详情、招聘会、就业政策、人才服务区 | 求职者中心 | 登录/注册、简历状态确认、投递、收藏、订阅、投递与面试状态回流。 |
| 企业 | 首页、找企业、职位详情、招聘会、管理中心 | 企业中心 | 招聘者注册/登录、企业认证、职位发布、提交审核、投递池处理。 |
| 运营/审计 | 管理中心 | 运营后台 | operator 审核和配置；auditor 只读；前端守卫只是体验，权限仍由后端强制。 |

## 4. 求职者转化闭环

### 4.1 入口与路径

求职者转化入口来自：

- 首页搜索和推荐职位。
- 找工作列表。
- 职位详情。
- 招聘会/活动详情。
- 就业政策/资讯/HR 工具箱的求职引导。
- 顶部“求职者登录/注册”。
- 管理中心身份分流。

首批闭环路径：

```text
公开职位列表/详情
  -> 未登录提示登录或注册
  -> 求职者登录/注册
  -> 求职者中心 overview
  -> 简历摘要与发布状态确认
  -> 投递职位
  -> 投递记录/面试状态回流
```

增强闭环路径：

```text
职位列表/详情
  -> 收藏职位
  -> 搜索订阅或职位订阅
  -> 求职者中心查看收藏/订阅
  -> 后续站内通知提醒
```

### 4.2 求职者页面状态

| 状态 | 触发条件 | 页面表达 |
| --- | --- | --- |
| 未登录 | 无 token 或 token 失效 | 展示登录/注册入口，保留返回原页面能力。 |
| 已登录但非 candidate | company/operator/auditor 访问求职者动作 | 展示无权限，提示切换求职者账号。 |
| 简历未完善 | overview 返回简历完成度不足或无简历摘要 | 提示先完善简历，不在前端推断字段。 |
| 可投递 | 后端确认职位在线、企业认证通过、candidate 身份有效 | 展示投递入口。 |
| 已投递 | 后端返回该职位已有投递 | 展示已投递和投递详情入口，不重复投递。 |
| 面试中/已签到 | 投递状态或签到摘要由后端返回 | 展示面试邀请、签到结果和后续状态。 |
| 已同意/已撤回/不可发布 | 由 overview 的发布状态返回 | 只展示状态，不在前端推断 consent_status。 |
| 失败/重试中 | 网络或服务端错误 | 复用 StateView 的 error/retrying。 |

### 4.3 投递口径

投递是高价值动作，后续实现必须遵守：

- 只能 candidate 对在线且企业认证通过的职位投递。
- 投递身份必须来自 JWT，不信任前端传入 candidate_id。
- 重复投递必须由服务端识别并返回稳定状态。
- 投递后企业只能在本企业投递池看到。
- 投递动作必须写 `audit_log`。
- 任何跨候选人、跨职位或跨企业对象 ID 必须有 IDOR/BOLA 阻断测试。

### 4.4 收藏与订阅口径

收藏/订阅在 Step3 只作为规划入口，不定义表结构和接口细节。

后续实现必须满足：

- 收藏职位、收藏企业、订阅搜索条件必须登录。
- 收藏/订阅写接口必须幂等。
- 收藏/订阅属于本人私有数据，不进入公开门户。
- 搜索订阅可触发站内通知，但真实短信、微信、小程序、App 推送不在本阶段接入。
- 收藏/订阅的删除或取消也应可审计。

### 4.5 求职者数据边界

求职者中心可展示本人私有摘要，但公开门户不得展示原始候选人数据。

允许在求职者中心展示：

- 本人简历完成度、更新时间、技能摘要。
- 本人投递数量、最近投递职位、投递状态。
- 本人面试/签到状态摘要。
- 本人同意/撤回/不可发布状态。
- 本人收藏/订阅摘要。

禁止在公开门户或人才服务区展示手机号、邮箱、简历正文、附件对象 key、同意证据、撤回证据、原始 JSON、聊天记录或内部备注。

## 5. 企业转化闭环

### 5.1 入口与路径

企业转化入口来自：

- 首页“企业招人才”入口。
- 找企业页面和企业主页的“我是企业/发布职位”入口。
- 职位详情中的企业信息区域。
- 招聘会页面的企业参会/发布职位入口。
- 顶部“招聘者登录/注册”。
- 管理中心身份分流。

首批闭环路径：

```text
公开门户企业入口
  -> 招聘者注册/登录
  -> 企业中心
  -> 企业认证
  -> 认证通过
  -> 发布职位/提交审核
  -> 运营审核通过
  -> 门户职位可见
  -> 投递池处理
```

认证未通过路径：

```text
招聘者注册/登录
  -> 企业中心
  -> 企业认证待审核/驳回
  -> 展示原因和补充入口
  -> 未通过前不得上线职位
```

### 5.2 企业页面状态

| 状态 | 触发条件 | 页面表达 |
| --- | --- | --- |
| 未登录 | 无 token 或 token 失效 | 展示招聘者登录/注册入口。 |
| 已登录但非 company | candidate/operator/auditor 访问企业动作 | 展示无权限，提示切换企业账号。 |
| 未认证/待审核 | 企业未提交或待审核 | 可以引导认证，不允许上线职位。 |
| 已驳回 | 企业认证驳回 | 展示驳回原因和重新提交入口。 |
| 已通过 | 企业认证通过 | 展示职位发布、职位列表、投递池入口。 |
| 职位待审核 | 职位提交后未通过 | 门户不可见，只在企业中心展示状态。 |
| 职位在线 | 企业认证通过且职位审核通过 | 门户可见。 |
| 职位下线 | 企业主动下线或运营处理 | 门户不可见。 |
| 投递池空 | 本企业暂无投递 | 展示空态和优化职位建议。 |
| 失败/重试中 | 网络或服务端错误 | 复用 StateView 的 error/retrying。 |

### 5.3 职位发布与审核口径

职位发布闭环必须继承一期状态机：

- 企业可以创建和提交待审核职位。
- 企业未认证通过时，不得让职位上线。
- 企业不能绕过运营审核直接把职位置为在线。
- 职位只有 `企业认证通过 + 职位审核通过 + 职位状态在线` 时，才允许公开门户展示。
- 职位审核、上下线必须写 `audit_log`。
- 企业只能操作本企业职位，跨企业访问必须返回统一错误。

### 5.4 投递池口径

企业投递池只服务本企业招聘闭环，不是候选人公开库。

投递池允许展示：

- application_id。
- job_id、职位标题。
- 投递状态、投递时间。
- 候选人发布快照摘要或受字段级权限裁剪后的最小流程摘要。
- 面试邀约和签到状态摘要。

投递池禁止展示：

- 非本企业职位产生的投递。
- 手机号、邮箱、简历正文、附件对象 key、证据等未授权原始候选人数据。
- 已撤回同意后继续展示为可联系人才。
- 通过前端隐藏代替服务端字段裁剪。

### 5.5 企业数据边界

企业中心是受控后台，不是公开门户。后续实现必须：

- 所有企业操作都从 JWT 和服务端数据域获取 company_id。
- 不信任前端传入 company_id。
- 企业认证、职位发布、职位审核、投递池、导出申请均有权限校验。
- 关键写动作写 `audit_log`。
- 敏感字段访问写 `field_access_log`。

## 6. 管理中心身份分流

| 身份 | 目标页面 | 说明 |
| --- | --- | --- |
| 未登录 | 登录/注册选择页 | 提供求职者和招聘者入口；真实短信/微信扫码仅占位。 |
| candidate | `/candidate/center` | 求职者中心，展示本人私有摘要、投递、签到、同意/撤回状态。 |
| company | `/company` | 企业中心，展示企业认证、职位状态、投递池、导出入口。 |
| operator | `/admin` | 运营后台，可执行授权范围内审核和配置。 |
| auditor | `/admin` | 运营后台只读视角，不允许写接口。 |

前端守卫只做体验层判断。所有关键动作必须由后端 `@RequirePermission`、数据域和字段级权限强制。

## 7. 状态机与空态

| 状态 | 页面域 | 设计要求 |
| --- | --- | --- |
| loading | 全部 | 首屏加载或切换状态。 |
| empty | 职位、投递、收藏、订阅、投递池 | 提供下一步引导，但不伪造数据。 |
| unauthorized | 求职者中心、企业中心、运营后台 | 明确提示登录或切换身份。 |
| forbidden | 企业/运营动作 | 后端拒绝时展示无权限，不隐藏错误事实。 |
| error | 全部 | 展示 trace_id 或错误提示，允许重试。 |
| retrying | 全部 | 用户触发重试后展示进行中。 |
| submitted | 投递、认证、职位提交 | 显示“已提交/待审核”，不前端推断最终状态。 |
| revoked/offline | 同意撤回、职位下线 | 显示已下线，不继续引导公开展示。 |

所有状态必须以接口返回为准。前端可以做展示兜底，但不得推断 `consent_status`、`publishable_flag`、企业认证结果、职位审核结果、投递归属或权限结果。

## 8. SEO 与 noindex 策略

| 页面 | SEO 策略 | 原因 |
| --- | --- | --- |
| 首页、找工作、职位详情、找企业、企业主页、招聘会、资讯 | index | 公开门户核心流量入口。 |
| 人才服务区列表 | index | 只展示发布快照白名单摘要。 |
| 登录/注册页 | noindex | 转化页，不需要搜索收录。 |
| 求职者中心 | noindex | 私有页面。 |
| 企业中心 | noindex | 私有后台。 |
| 运营后台 | noindex | 内部后台。 |
| 投递成功/认证提交/职位提交结果页 | noindex | 状态页，不面向搜索。 |
| 收藏/订阅列表 | noindex | 本人私有数据。 |

后续实现时，Next metadata、robots 和 sitemap 必须继续区分公开流量页与私有转化页。

## 9. 字段与权限边界

公开门户继续使用 Step2 字段白名单。

求职者中心必须禁止：

- candidate A 访问 candidate B 的中心。
- company 访问 candidate center overview。
- operator/auditor 通过前端路径读取候选人私有中心。
- 前端把本人原始字段复制到公开页面或人才服务区。

企业中心必须禁止：

- company A 查看 company B 的职位、投递池、导出申请。
- 企业未认证通过时上线职位。
- 企业绕过运营审核让职位公开。
- 投递池返回未授权联系方式或简历原文。

审计与日志：

- 企业认证提交与补充、职位创建、提交审核、审核通过/驳回、上线、下线、投递、收藏、订阅、面试邀约、导出申请、导出下载都必须审计。
- 敏感字段读取继续写 `field_access_log`，并通过 `trace_id` 与 `audit_log` 关联。

## 10. 自动化验收关注点

后续实现 Step3 相关能力时，至少要覆盖：

- 求职者未登录投递时进入登录/注册引导，登录后可回到原职位。
- candidate 投递在线职位成功，重复投递不产生重复记录。
- candidate 收藏/订阅只作用于本人，取消操作幂等。
- company 未认证通过时不能上线职位。
- company 只能查看本企业投递池，跨企业访问被阻断。
- operator/auditor 身份分流正确，auditor 写接口仍拒绝。
- 人才服务区仍只展示发布快照，撤回后不可见。
- 前端状态机覆盖 loading/empty/unauthorized/forbidden/error/retrying/submitted/offline。
- 私有页面和状态页 noindex，公开页面 index。

## 11. Step3 结论

Step3 冻结了二期从公开门户到求职者中心、企业中心的转化闭环：

- 求职者：公开职位列表/详情 -> 登录/注册 -> 求职者中心 -> 简历摘要/状态确认 -> 投递 -> 收藏/订阅占位 -> 投递与面试状态回流。
- 企业：公开门户入口 -> 招聘者注册/登录 -> 企业认证 -> 职位发布/提交审核 -> 运营审核 -> 门户职位可见 -> 投递池处理。
- 管理中心：按 candidate/company/operator/auditor 身份分流，前端只做体验，后端继续强制权限。

高保真门户优先补充：

- Step3 的转化闭环仍然有效，但实施顺序应承接 Prompt 16-21 的公开门户高保真页面。
- 求职者投递、收藏、订阅入口应嵌入找工作和职位详情页面，但深层私有中心增强不抢在首页、列表页和人才服务区改写之前。
- 登录/注册与管理中心入口应参考 demo 页面形态，但真实短信、微信扫码、小程序、App 登录仍只做禁用占位。

下一步应继续完成：

```text
LocalTalent_phase2_step4.md：求职者中心与企业中心能力细化
```

Step4 不建议立刻展开受控找人才、联系解锁、会员商业化、真实支付、真实短信/微信/小程序/App、真实地图/视频/直播、真实 ATS/RPA 或受控候选池同步。
