LocalTalent Phase-2 Step 3：求职者与企业转化闭环

统一硬边界提醒：人才服务区仅展示发布快照；对接接口仅 stub；不得输出原始候选人数据；同意、撤回、数据域、字段级权限必须在服务端收口。

开发前必读卡

  • 文件定位：冻结公开门户到求职者中心、企业中心的转化路径、状态机、边界与验收关注点。
  • 资料来源：边界文件、两份新 MD、Step1/Step2、本地 demo、一期文档；不再引用旧版资料文件。
  • 禁止越界点：禁止公共简历库、受控找人才和联系解锁提前展开；禁止真实外部能力。

1. Step1/Step2 继承结论

Step1 冻结门户导航、页面域和“简历库不得照搬为公共简历库”的边界。Step2 冻结公开门户、搜索体系、SEO/Next SSR 策略和字段白名单。

Step3 只回答一个问题：公开门户中的游客，如何安全、清晰地转化为求职者或企业用户？

2. Step3 总目标

转化对象起点目标中心闭环
求职者首页、找工作、职位详情、招聘会、就业政策、人才服务区求职者中心登录/注册、简历状态确认、投递、收藏、订阅、投递与面试状态回流。
企业首页、找企业、职位详情、招聘会、管理中心企业中心招聘者注册/登录、企业认证、职位发布、提交审核、投递池处理。
运营/审计管理中心运营后台operator 审核和配置；auditor 只读；权限由后端强制。

3. 求职者转化闭环

公开职位列表/详情
  -> 未登录提示登录或注册
  -> 求职者登录/注册
  -> 求职者中心 overview
  -> 简历摘要与发布状态确认
  -> 投递职位
  -> 投递记录/面试状态回流

增强路径：职位收藏、搜索订阅或职位订阅、求职者中心查看收藏/订阅、后续站内通知提醒。

状态页面表达
未登录展示登录/注册入口，保留返回原页面能力。
简历未完善提示先完善简历，不在前端推断字段。
可投递/已投递由后端确认职位在线、企业认证通过、candidate 身份和投递状态。
面试中/已签到投递状态或签到摘要由后端返回。
已同意/已撤回/不可发布由 overview 的发布状态返回，前端不得推断。

4. 企业转化闭环

公开门户企业入口
  -> 招聘者注册/登录
  -> 企业中心
  -> 企业认证
  -> 认证通过
  -> 发布职位/提交审核
  -> 运营审核通过
  -> 门户职位可见
  -> 投递池处理
状态页面表达
未登录展示招聘者登录/注册入口。
未认证/待审核引导认证，不允许上线职位。
已驳回展示驳回原因和重新提交入口。
已通过展示职位发布、职位列表、投递池入口。
职位在线/下线公开可见性完全由服务端状态决定。
投递池空展示空态和优化职位建议。

企业未认证通过不得上线职位；企业不能绕过运营审核；跨企业投递池必须服务端阻断。

5. 管理中心身份分流

身份目标页面说明
candidate/candidate/center展示本人私有摘要、投递、签到、同意/撤回状态。
company/company展示企业认证、职位状态、投递池、导出入口。
operator/admin可执行授权范围内审核和配置。
auditor/admin只读视角，不允许写接口。

6. 状态机与空态

loadingemptyunauthorizedforbiddenerrorretryingsubmittedrevoked/offline

所有状态必须以接口返回为准。前端可以做展示兜底，但不得推断 consent_status、publishable_flag、企业认证结果、职位审核结果、投递归属或权限结果。

7. 字段与权限边界

  • 公开门户继续使用 Step2 字段白名单。
  • candidate 只能访问本人中心和本人投递、收藏、订阅。
  • company 只能访问本企业职位、投递池、导出申请。
  • 投递池不是公共简历库，不得返回未授权联系方式或简历原文。
  • 关键写动作写 audit_log，敏感字段读取写 field_access_log。

8. Step3 结论

Step3 冻结了从公开门户到求职者中心、企业中心的转化闭环。

高保真门户优先补充：Step3 的转化闭环仍然有效，但实施顺序应承接 Prompt 16-21 的公开门户高保真页面；求职者投递、收藏、订阅入口应嵌入找工作和职位详情页面，但深层私有中心增强不抢在首页、列表页和人才服务区改写之前。

登录/注册与管理中心入口应参考 demo 页面形态，但真实短信、微信扫码、小程序、App 登录仍只做禁用占位。
