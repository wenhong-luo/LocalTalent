LocalTalent Phase-2 Step 6：风险池与高风险能力准入规则

统一硬边界提醒：人才服务区仅展示发布快照；对接接口仅 stub；不得输出原始候选人数据；同意、撤回、数据域、字段级权限必须在服务端收口。

开发前必读卡

  • 文件定位：冻结高风险能力从风险池进入开发的准入规则、评审材料、验收门槛和禁止绕行项。
  • 资料来源：边界文件、两份新 MD、Step1-Step5、本地 demo、一期文档。
  • 禁止越界点：禁止把风险池条目直接实现；禁止公共简历库；禁止真实 ATS/RPA、支付、短信、微信、小程序、App、地图、视频、直播。

1. Step6 总目标

目标说明
防止偷跑高风险能力不能因为 demo 有入口、业务想要或前端能隐藏，就直接进入开发。
明确准入高风险能力必须先有材料、方案、测试和验收门槛，才能进入后续 Prompt。
支持后续蓝图二期蓝图可以引用本文件，把低风险功能和高风险能力分开排期。
保持可追溯每个风险能力都必须能说明为什么暂缓、为什么准入、由谁负责、如何验收。

2. 风险池状态分级

状态含义是否可开发
risk_pool已识别风险，暂缓。否。
assessment正在评估业务价值、风险、合规和实现代价。否。
design_review已有方案草案，等待产品、安全、合规、技术评审。否。
stub_only只允许占位、文档、静态入口或 stub。仅限 stub。
approved_for_prompt已满足准入材料和评审要求，可进入后续 Prompt。是，但只能按 Prompt 范围做。
rejected_or_deferred已驳回或延期。否。

默认状态是 risk_pool。只有 approved_for_prompt 可以进入开发。

3. 高风险能力分组

公共简历库受控找人才联系解锁会员商业化/真实支付真实短信/微信/小程序/App真实地图/视频/直播真实 ATS/RPA/受控候选池同步AI/模型处理原始候选人数据政府/运营明细看板

这些能力不是永远不做，而是必须先经过准入流程。没有准入材料和验收门槛时，不能进入开发。

4. 准入流程

  1. 登记风险池：写明风险主题、业务诉求、触发场景和涉及数据。
  2. 业务必要性评估：证明该能力是否招聘闭环必须项，是否可替代或可做 stub。
  3. 数据地图评估：列出数据来源、流向、存储、缓存、搜索、日志、导出和外部系统。
  4. 合规与安全方案：说明合法性依据、同意/撤回、字段白名单、权限和审计。
  5. 设计评审：产品、技术、安全/合规共同确认边界。
  6. 测试设计：定义字段黑名单、撤回联动、IDOR/BOLA、审计、脱敏和回滚测试。
  7. Prompt 准入：通过评审后才能进入二期提示词中的明确 Prompt。
  8. 实施后验收：按 Prompt 验收，失败则回退到风险池。

5. 准入材料清单

材料必填内容
业务目的为什么要做，解决哪个招聘闭环问题，是否有低风险替代方案。
数据来源数据从哪里来，是否来自候选人本人、企业、运营、外部系统或第三方供应商。
合法性依据是否有单独告知、单独同意、合同依据、运营依据或政策依据。
同意/撤回方案同意范围、版本、撤回入口、撤回后的下线、删除或隐藏联动。
字段白名单允许展示、存储、导出、进入搜索的字段。
权限与审计RBAC、数据域、字段级权限、audit_log、field_access_log。
安全测试IDOR/BOLA、字段黑名单、脱敏、越权、幂等、重放和导出绕过测试。
旁路边界缓存、搜索、日志、报表、导出、推荐、AI、外部同步是否会带出数据。
回滚/下线方案出现风险时如何暂停、隐藏、撤回、删除、作废链接或停用外部能力。

6. 关键准入门槛

  • 公共简历库：禁止；只能改写为人才服务区发布快照公开，或企业受控找人才。
  • 受控找人才：企业认证、候选人同意、未撤回、字段级权限、默认不展示联系方式、全链路审计。
  • 联系解锁：必须有授权或审批依据、权益范围、有效期、访问次数和撤回后失效。
  • 会员商业化/真实支付：必须先建立权益模型，支付供应商需安全与合规评审。
  • 真实短信/微信/小程序/App：必须有模板、触达目的、退订、频控、隐私告知和供应商治理。
  • 真实地图/视频/直播：必须有定位、内容审核、版权、举报、下架和供应商评审方案。
  • 真实 ATS/RPA/受控候选池同步：必须明确数据来源合法性、最小字段、凭证安全和撤回联动。
  • AI/模型处理原始候选人数据：必须先字段裁剪、脱敏、最小必要、供应商审查和人工复核。
  • 政府/运营明细看板：默认只展示聚合统计；明细必须审批、脱敏、审计。

7. 上线前阻断规则

以下任一条件成立，能力不得上线：

没有字段白名单没有撤回联动没有服务端权限方案没有 IDOR/BOLA 测试没有字段黑名单测试没有审计日志方案没有旁路边界外部供应商未评审候选人原始数据无合法性依据

8. 禁止绕行项

不得把风险功能伪装成前端隐藏、运营备注、导出报表、搜索索引、缓存字段、调试日志、审计日志明文、外部同步、活动报名、会员权益、推荐位、AI 分析或管理员特权来提前实现。

如果某能力实质上会让未授权角色看到原始候选人数据、联系方式或敏感材料，就必须回到风险池准入流程。

9. 自动化验收关注点

  • 风险能力有明确 Prompt，且 Prompt 写明允许范围和禁止事项。
  • 公开响应字段白名单测试通过，人才服务区仍只展示发布快照。
  • 撤回后公开展示、受控摘要、联系解锁和导出均失效或不可见。
  • candidate/company/operator/auditor 数据域隔离测试通过。
  • 所有对象 ID 接口覆盖 IDOR/BOLA 阻断。
  • 敏感字段读取写 field_access_log，高风险写动作写 audit_log。
  • 日志、缓存、搜索、报表、导出不含未授权原始候选人数据。
  • 对接接口仍仅 stub，除非对应 Prompt 明确完成真实接入准入。

10. Step6 结论

Step6 冻结了二期风险池与高风险能力准入规则。默认状态是 risk_pool，只有 approved_for_prompt 可以进入开发。没有字段白名单、撤回联动、服务端权限、审计和测试，就不得开发。

高保真门户优先补充：demo 中的地图、视频、直播、短信、微信、小程序、App、自由职业、求职登记、快速招聘联系方式、会员权益和支付仍属于风险池或占位能力。

高保真页面可以展示这些入口的视觉占位，但不得因为“像 demo”而绕过 Step6 准入。
