# LocalTalent Phase-2 Step 6：风险池与高风险能力准入规则

## 开发前必读卡

- **文件定位**：二期第六份规划文档，只冻结高风险能力从风险池进入开发的准入规则、评审材料、验收门槛和禁止绕行项。
- **实现约束**：本文件是规划输入，不是开发 Prompt；不新增代码、不新增接口 schema、不新增数据库迁移、不生成二期蓝图或提示词。
- **禁止越界点**：禁止把风险池条目直接实现；禁止公共简历库；禁止真实 ATS/RPA/受控候选池同步；禁止真实支付、真实短信、真实微信、小程序、App、真实地图、视频、直播接入。
- **下游影响**：为二期蓝图、二期提示词、Runbook、Testing、上线检查和后续高风险能力评审提供统一准入门槛。
- **自动化验收关注点**：后续实现必须继续守住人才服务区、发布快照、原始候选人数据、同意、撤回、对接接口、数据域、字段级权限。

**统一硬边界提醒**：人才服务区仅展示发布快照；对接接口仅 stub；不得输出原始候选人数据；同意、撤回、数据域、字段级权限必须在服务端收口。

## 2. Step1-Step5 继承结论

Step6 继承以下前置结论：

- 公开门户只能展示公开字段和发布快照白名单字段。
- 人才服务区唯一公开数据源仍是 `candidate_publish_snapshot`。
- 求职者中心、企业中心、运营后台都是受控私有域。
- `operator` 可执行授权范围内的运营写操作；`auditor` 只读。
- 所有对象 ID 接口必须防 IDOR/BOLA。
- 所有敏感字段读取必须走字段级权限并写 `field_access_log`。
- 所有高风险写动作必须写 `audit_log`。
- 对接接口仍仅 stub，不接真实 ATS/RPA 或受控候选池主库。

## 3. Step6 总目标

Step6 的目标是给风险池建立“闸门”。

| 目标 | 说明 |
| --- | --- |
| 防止偷跑 | 任何高风险能力不能因为 demo 有入口、业务想要或前端能隐藏，就直接进入开发。 |
| 明确准入 | 高风险能力必须先有材料、方案、测试和验收门槛，才能进入后续 Prompt。 |
| 支持后续蓝图 | 二期蓝图可以引用本文件，把低风险功能和高风险能力分开排期。 |
| 保持可追溯 | 每个风险能力都必须能说明为什么暂缓、为什么准入、由谁负责、如何验收。 |

Step6 不定义最终 API schema，不新增数据库表，不新增代码实现。

## 4. 风险池状态分级

风险池条目统一使用以下状态：

| 状态 | 含义 | 是否可开发 |
| --- | --- | --- |
| `risk_pool` | 已识别风险，暂缓。 | 否。 |
| `assessment` | 正在评估业务价值、风险、合规和实现代价。 | 否。 |
| `design_review` | 已有方案草案，等待产品、安全、合规、技术评审。 | 否。 |
| `stub_only` | 只允许占位、文档、静态入口或 stub，不处理真实敏感能力。 | 仅限 stub。 |
| `approved_for_prompt` | 已满足准入材料和评审要求，可进入后续 Prompt。 | 是，但只能按 Prompt 范围做。 |
| `rejected_or_deferred` | 已驳回或延期。 | 否。 |

状态规则：

- 默认状态是 `risk_pool`。
- 只有 `approved_for_prompt` 可以进入开发。
- `stub_only` 不得处理真实个人信息、真实支付、真实外部发送或真实同步。
- 状态变更必须留下评审记录，后续实现时应能写入 `audit_log` 或文档变更记录。

## 5. 高风险能力分组

以下能力统一纳入风险池：

| 分组 | 当前结论 | 风险核心 |
| --- | --- | --- |
| 公共简历库 | 禁止 | 会把人才服务区变成原始候选人数据公开层。 |
| 受控找人才 | 暂缓 | 涉及企业访问候选人摘要、同意状态、字段级权限和撤回联动。 |
| 联系解锁 | 暂缓 | 涉及手机号、邮箱、微信等高敏字段和权益/审批。 |
| 会员商业化/真实支付 | 暂缓 | 涉及资金、权益、退款、发票、风控和审计。 |
| 真实短信/微信/小程序/App | 暂缓 | 涉及外部供应商、模板、退订、频控和隐私告知。 |
| 真实地图/视频/直播 | 暂缓 | 涉及 SDK 合规、内容审核、版权、定位和成本。 |
| 真实 ATS/RPA/受控候选池同步 | 暂缓 | 涉及外部数据来源、凭证、同步责任和撤回删除联动。 |
| AI/模型处理原始候选人数据 | 暂缓 | 涉及数据外发、供应商处理、解释性和自动化决策风险。 |
| 政府/运营明细看板 | 暂缓 | 涉及候选人明细化展示、导出和跨角色数据泄露。 |

## 6. 准入流程

高风险能力进入开发前必须经过以下流程：

1. **登记风险池**：写明风险主题、业务诉求、触发场景和涉及数据。
2. **业务必要性评估**：证明该能力是招聘闭环必须项，还是可替代、可后置或可做 stub。
3. **数据地图评估**：列出数据来源、数据流向、存储位置、缓存、搜索、日志、导出和外部系统。
4. **合规与安全方案**：说明合法性依据、同意/撤回、字段白名单、权限和审计。
5. **设计评审**：产品、技术、安全/合规共同确认边界。
6. **测试设计**：定义字段黑名单、撤回联动、IDOR/BOLA、审计、脱敏和回滚测试。
7. **Prompt 准入**：只有通过评审后，才能在二期提示词里生成明确 Prompt。
8. **实施后验收**：按 Prompt 验收，失败则回退到风险池。

## 7. 准入材料清单

每个高风险能力至少要准备以下材料：

| 材料 | 必填内容 |
| --- | --- |
| 业务目的 | 为什么要做，解决哪个招聘闭环问题，是否有低风险替代方案。 |
| 数据来源 | 数据从哪里来，是否来自候选人本人、企业、运营、外部系统或第三方供应商。 |
| 合法性依据 | 是否有单独告知、单独同意、合同依据、运营依据或政策依据。 |
| 同意/撤回方案 | 同意范围、版本、撤回入口、撤回后的下线/删除/隐藏联动。 |
| 字段白名单 | 允许展示、允许存储、允许导出、允许进入搜索的字段。 |
| 禁止字段 | 手机号、邮箱、简历正文、附件对象 key、证据、原始 JSON、聊天记录等是否被排除。 |
| 数据域与字段级权限 | 哪些角色可访问，是否本人/本企业/运营/审计隔离。 |
| 审计日志 | 哪些动作写 `audit_log`，哪些字段读取写 `field_access_log`。 |
| 安全测试 | IDOR/BOLA、字段黑名单、脱敏、越权、幂等、重放和导出绕过测试。 |
| 旁路边界 | 缓存、搜索、日志、报表、导出、推荐、AI、外部同步是否会带出数据。 |
| 外部供应商 | 供应商名称、处理数据、合同、密钥、失败补偿和退出方案。 |
| 回滚/下线方案 | 出现风险时如何暂停、隐藏、撤回、删除、作废链接或停用外部能力。 |
| 负责人和证据 | 产品、技术、安全/合规负责人以及验收证据位置。 |

缺少任一关键材料时，能力不得进入开发。

## 8. 各高风险能力准入门槛

### 8.1 公共简历库

当前结论：禁止。

准入门槛：

- 不允许以“公共简历库”形态进入开发。
- 只能改写为人才服务区发布快照公开，或企业受控找人才。
- 不得公开手机号、邮箱、简历正文、附件对象 key、证据、原始 JSON。

### 8.2 受控找人才

当前结论：暂缓。

未来准入门槛：

- 企业必须认证通过。
- 候选人必须同意对应用途，且未撤回。
- 只查询发布快照或受字段级权限裁剪后的受控摘要。
- 服务端强制 RBAC、数据域、字段级权限。
- 联系方式默认不可见。
- 查询、查看、筛选、联系申请必须写审计。
- 必须覆盖跨企业、跨候选人、撤回后不可见和字段黑名单测试。

### 8.3 联系解锁

当前结论：暂缓。

未来准入门槛：

- 明确候选人授权或审批依据。
- 明确企业权益、次数、范围和有效期。
- 手机号、邮箱、微信等字段必须由字段级权限控制。
- 解锁、查看、导出、过期、撤回后失效必须写审计。
- 候选人撤回后，后续不得继续展示联系方式。

### 8.4 会员商业化与真实支付

当前结论：暂缓。

未来准入门槛：

- 先建立权益模型，不直接接真实支付。
- 明确套餐、权益、消费、退款、发票、对账、风控和审计。
- 支付供应商必须经过安全与合规评审。
- 不得把付费权益作为绕过字段级权限、导出审批或候选人授权的理由。

### 8.5 真实短信、微信、小程序、App

当前结论：暂缓。

未来准入门槛：

- 明确模板、触达目的、退订、频控、失败补偿和隐私告知。
- 不发送敏感明文。
- 不把短信/微信作为默认同意渠道。
- 外部供应商密钥和回调必须有签名、幂等、trace 和审计。

### 8.6 真实地图、视频、直播

当前结论：暂缓。

未来准入门槛：

- 地图必须说明定位数据是否采集、是否存储、是否可关闭。
- 视频/直播必须说明内容审核、版权、举报、下架和留痕。
- 不得上传候选人简历附件、证据或未经授权的视频材料。
- 供应商 SDK 必须经过安全和隐私评审。

### 8.7 真实 ATS/RPA/受控候选池同步

当前结论：暂缓。

未来准入门槛：

- 明确数据来源合法性和平台协议。
- 默认最小字段，不同步候选人原始主库。
- 凭证必须本地加密和最小权限。
- 同步必须可暂停、可回滚、可审计。
- 候选人撤回后必须能联动删除、隐藏或停止使用。
- 对接接口仍应先走 stub，不得直接外联主库。

### 8.8 AI/模型处理原始候选人数据

当前结论：暂缓。

未来准入门槛：

- 不把手机号、邮箱、简历正文、附件、证据、聊天记录直接送入模型。
- 必须先做字段裁剪、脱敏和最小必要。
- 供应商、数据留存、训练使用、人工复核和解释性必须明确。
- 不能用黑盒评分作为自动拒绝或排序的唯一依据。

### 8.9 政府/运营明细看板

当前结论：暂缓。

未来准入门槛：

- 默认只展示聚合统计。
- 不展示候选人明细、联系方式、简历正文或附件。
- 如确需明细，必须有数据域、审批、脱敏、导出控制和审计。

## 9. 上线前阻断规则

以下任一条件成立，能力不得上线：

- 没有字段白名单。
- 没有撤回联动。
- 没有服务端权限方案。
- 没有 IDOR/BOLA 测试。
- 没有字段黑名单测试。
- 没有审计日志方案。
- 没有日志、缓存、搜索、报表和导出旁路边界。
- 涉及外部供应商但没有安全/合规评审。
- 涉及支付但没有财务、退款、对账和风控方案。
- 涉及候选人原始数据但没有合法性依据。

## 10. 禁止绕行项

不得把风险功能伪装成以下形式提前实现：

- 前端隐藏按钮。
- 运营备注。
- 导出报表。
- 搜索索引。
- 缓存字段。
- 调试日志。
- 审计日志明文。
- 外部同步。
- 活动报名。
- 会员权益。
- 推荐位。
- AI 分析。
- 管理员特权。

如果某能力实质上会让未授权角色看到原始候选人数据、联系方式或敏感材料，就必须回到风险池准入流程。

## 11. 自动化验收关注点

后续实现任何从风险池准入的能力时，至少要覆盖：

- 风险能力有明确 Prompt，且 Prompt 写明允许范围和禁止事项。
- 公开响应字段白名单测试通过。
- 人才服务区仍只展示发布快照。
- 撤回后公开展示、受控摘要、联系解锁和导出均失效或不可见。
- candidate/company/operator/auditor 数据域隔离测试通过。
- 所有对象 ID 接口覆盖 IDOR/BOLA 阻断。
- 敏感字段读取写 `field_access_log`。
- 高风险写动作写 `audit_log`。
- 日志、缓存、搜索、报表、导出不含未授权原始候选人数据。
- auditor 写接口继续拒绝。
- 对接接口仍仅 stub，除非对应 Prompt 明确完成真实接入准入。

## 12. Step6 结论

Step6 冻结了二期风险池与高风险能力准入规则：

- 高风险能力不是不能做，但不能无边界地做。
- 默认状态是 `risk_pool`，只有 `approved_for_prompt` 可以进入开发。
- 任何涉及原始候选人数据、联系方式、外部同步、支付、外部触达、地图、视频、直播或 AI 的能力，都必须先准备准入材料。
- 没有字段白名单、撤回联动、服务端权限、审计和测试，就不得开发。
- 禁止通过前端隐藏、日志、缓存、导出、搜索、活动报名或会员权益绕开风险准入。

高保真门户优先补充：

- demo 中的地图、视频、直播、短信、微信、小程序、App、自由职业、求职登记、快速招聘联系方式、会员权益和支付仍属于风险池或占位能力。
- 高保真页面可以展示这些入口的视觉占位，但不得因为“像 demo”而绕过 Step6 准入。
- 任何从占位升级为真实能力的需求，都必须先进入 `assessment` 或 `design_review`，最终达到 `approved_for_prompt` 后才能开发。
