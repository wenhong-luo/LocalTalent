# 人才网总盘点与高风险深挖评审稿

## 开发前必读卡

- **文件定位**：二期规划的风险总盘点文件，用于帮助后续 Step、蓝图、提示词识别哪些能力可以做、哪些能力必须暂缓、哪些能力必须先经过合规与安全方案评审。
- **实现约束**：本文不是开发 Prompt，不把所有风险点直接升级成当前需求；当前轮次只做当前 Prompt 明确要求的内容。
- **禁止越界点**：禁止公共简历库；禁止真实 ATS/RPA/受控候选池同步；禁止真实支付、短信、微信、小程序、App、地图、视频、直播接入；禁止把原始候选人数据放入公开门户、日志、缓存、搜索、导出或报表旁路。
- **下游影响**：供 README、RUNBOOK、测试矩阵、审计说明、上线检查单、二期 Step 和 Prompt 使用。
- **自动化验收关注点**：人才服务区字段黑名单、撤回下线、IDOR/BOLA 阻断、字段级权限、导出审批、审计脱敏、对接接口 stub。

**统一硬边界提醒**：人才服务区仅展示发布快照；对接接口仅 stub；不得输出原始候选人数据；同意、撤回、数据域、字段级权限必须在服务端收口。

## 0. 本次修订说明

本文由旧版“公益平台、RPA 本地化、ATS 合法入口驱动 项目总盘点与高风险深挖”整理而来。旧稿的风险判断仍有价值，但需要和 LocalTalent 一期已完成基线、二期人才网目标重新对齐。

本次修订后的使用原则：

1. **功能可以规划，但风险功能不急着做**。
2. **高风险点先进入风险池和上线检查单**，不自动变成当前开发需求。
3. **如果某功能涉及候选人原始数据、联系方式、导出、外部同步、支付或消息触达，必须先有规避方案再开发**。
4. **若本文与一期硬边界、二期边界文件冲突，一律以硬边界为准**。

## 1. 项目目标与当前判断

LocalTalent 的长期目标，是在一期合规底座上，演进为地方人才网：

- 面向求职者：找工作、看企业、投递、面试、管理简历、关注状态。
- 面向企业：认证企业、发布职位、处理投递、邀约面试、申请导出。
- 面向运营：审核企业和职位、管理内容活动、审计追踪、风控治理。
- 面向公开门户：展示职位、企业、招聘会、政策资讯、HR 工具箱和人才服务区发布快照。

当前最重要的策略判断：

```text
先做招聘闭环和门户体验；
高风险候选人数据能力先不做实；
所有风险点纳入风险池、说明书和验收清单。
```

## 2. 一期已完成的风险控制基线

二期不是从零开始，以下能力已经在一期建立，后续不能绕开：

| 能力 | 一期状态 | 二期约束 |
| --- | --- | --- |
| 人才服务区 | 只展示 `candidate_publish_snapshot` 发布快照 | 不得改成公共简历库。 |
| 同意/撤回 | 同意生成快照，撤回下线快照 | 后续任何展示都必须联动撤回。 |
| 权限 | RBAC + 数据域 + 字段级权限 | 私有中心、导出、审核、对象 ID 都必须服务端强制。 |
| 审计 | `audit_log`、`field_access_log`、`open_api_log` | 敏感读取和高风险动作必须可追溯。 |
| 导出 | export_apply 审批 + MinIO 短期下载 | 不得绕过审批直接导出或下载。 |
| 对接接口 | openapi stub，签名/幂等/trace/重试 | 不得升级成真实 ATS 或候选池同步。 |
| 前端 | Next/SSR + 私有中心 noindex | 公开页面要 SEO，私有页面不收录。 |
| 工程化 | CI、Runbook、边界脚本、演示数据 | 二期新增能力要接入测试矩阵。 |

## 3. 能力分级：现在可做、后续可做、风险暂缓

### 3.1 当前可做

这些能力与招聘闭环强相关，且可以在现有边界下实现：

| 能力 | 可做原因 | 必须遵守 |
| --- | --- | --- |
| 首页/找工作/找企业 | 公开字段白名单明确 | 不展示内部联系人和审核材料。 |
| 人才服务区增强 | 已有发布快照与字段黑名单 | 只能读发布快照。 |
| 求职者中心增强 | 本人私有域 | 状态由服务端返回，不前端推断。 |
| 企业中心增强 | 本企业私有域 | company_id 来自 JWT 和服务端数据域。 |
| 职位收藏/搜索订阅 | 本人私有数据 | 幂等、可取消、可审计。 |
| 站内通知 | 不依赖外部供应商 | 不发送真实短信或微信。 |
| 招聘会/资讯/HR 工具箱公开展示 | 公开内容域 | 不展示候选人名单和敏感数据。 |
| 运营内容活动管理 | 已有 admin 基础 | operator 写、auditor 只读。 |

### 3.2 后续可做

这些能力可以规划，但需要等基础闭环稳定：

| 能力 | 暂缓理由 | 进入条件 |
| --- | --- | --- |
| 网络招聘会 | 活动、报名、展位和互动较复杂 | 活动数据模型和报名隐私边界稳定。 |
| 校园招聘 | 组织体系和专题页较重 | 招聘会基础能力完成。 |
| 快速招聘/今日招聘 | 运营专题属性强 | 职位搜索和推荐配置成熟。 |
| 规则推荐 | 需要解释性和可调权重 | 搜索、标签、行为数据稳定。 |
| 企业主页增强 | 媒体、相册、招聘动态 | 内容审核和公开字段策略稳定。 |

### 3.3 高风险暂缓

这些能力有业务价值，但当前不应开发。它们应进入风险池，等规避方案成熟后再评估：

| 高风险能力 | 当前结论 | 未来规避方向 |
| --- | --- | --- |
| 公共简历库 | 禁止 | 只能改为发布快照公开 + 企业受控检索。 |
| 受控找人才 | 暂缓 | 企业认证、候选人同意、隐私开关、权益/审批、字段级权限、审计。 |
| 联系方式解锁 | 暂缓 | 授权、权益、审批、访问次数、撤回联动、字段访问日志。 |
| 会员商业化 | 暂缓 | 权益模型、支付合规、退款、发票、审计。 |
| 真实支付 | 暂缓 | 支付供应商、资金流、风控、财务对账。 |
| 真实短信/微信/小程序/App | 暂缓 | 模板合规、退订、频控、隐私协议、供应商合同。 |
| 真实地图/视频/直播 | 暂缓 | SDK 合规、成本、内容审核、版权和隐私。 |
| 真实 ATS/RPA/受控候选池同步 | 暂缓 | 数据来源合法性、最小字段、同步责任、撤回删除联动。 |
| AI/模型分析候选人原始数据 | 暂缓 | 字段裁剪、脱敏、最小必要、供应商审查。 |

## 4. 关键风险总览

| 序号 | 高风险点 | 触发场景 | 直接后果 | 当前控制方向 |
| --- | --- | --- | --- | --- |
| 1 | 候选人原始数据进入可读云端 | 简历正文、联系方式、聊天记录进入云端候选池 | 个保、竞争、平台协议风险放大 | 公开层只展示发布快照，原始数据只在受控域使用。 |
| 2 | 公益/地方平台混入商业平台候选人个人信息 | 直接展示或调取商业平台候选人 | 平台属性和政府合作可信度受损 | 地方平台公开侧只展示职位、企业、活动、政策、发布快照。 |
| 3 | ATS/RPA 继续承接高风险入口 | 外部候选人数据默认同步到 ATS 或云端 | ATS 成为高风险数据汇聚器 | 对接接口仅 stub，真实同步暂停。 |
| 4 | 日志泄漏原始候选人数据 | 调试、异常、审计、埋点写入明文 | 旁路泄漏和审计失控 | 日志脱敏，敏感字段禁入。 |
| 5 | 缓存/搜索/报表重新串数 | 搜索索引或 BI 报表写入敏感字段 | 边界被旁路打穿 | 公开索引白名单，报表聚合化。 |
| 6 | 授权链路不清晰 | 投递、签到、入库没有明确同意 | 后续使用无法证明合法性 | 单独告知、单独同意、支持撤回。 |
| 7 | 凭证管理不当 | 平台账号、Token、Cookie 明文存储 | 企业账号风险和平台封禁 | 当前不做真实 RPA；未来必须本地加密和最小权限。 |
| 8 | 公共统计看板使用候选人明细 | 政府/运营看板读取候选人详情 | 明细化暴露扩大风险 | 只展示聚合统计，不展示明细。 |
| 9 | 模型调用使用原始候选人数据 | 把简历原文、联系方式送入模型 | 数据外发和供应商处理风险 | 暂缓，未来必须脱敏和供应商审查。 |
| 10 | 先做全功能后补边界 | 把同步、人才库、推荐、商业化一起上线 | 返工和合规阻塞 | 先边界、先闭环、后增强。 |

## 5. 风险深挖与处理口径

### 风险 1：候选人原始数据进入可读云端

- **优先级**：最高。
- **典型场景**：简历正文、手机号、邮箱、聊天记录、面试反馈被同步到云端候选池、日志或搜索索引。
- **当前处理**：公开门户和人才服务区只展示发布快照；私有中心只展示本人或本企业范围内的受控摘要。
- **现在不做**：真实受控候选池同步、公共简历库、候选人原始数据云端汇聚。
- **未来进入条件**：完成数据地图、来源合法性、撤回删除联动、字段级权限、审计和合规评审。

### 风险 2：地方公益平台混入商业候选人个人信息

- **优先级**：最高。
- **典型场景**：公开门户或政府合作页面展示商业平台候选人详情。
- **当前处理**：公开门户只展示职位、企业、活动、政策、资讯和发布快照摘要。
- **现在不做**：把商业平台候选人明细导入人才服务区。
- **未来进入条件**：候选人单独同意、来源隔离、字段白名单和撤回联动成熟。

### 风险 3：ATS/RPA 高风险入口

- **优先级**：最高。
- **典型场景**：RPA 抓取候选人后自动回传云端或 ATS。
- **当前处理**：一期对接接口仅 stub；二期规划仍保持 stub。
- **现在不做**：真实 ATS 主库同步、真实 RPA 任务执行、真实受控候选池连接器。
- **未来进入条件**：明确本地执行、最小回传、凭证加密、企业授权、平台协议和合规审查。

### 风险 4：日志、缓存、搜索、报表旁路泄漏

- **优先级**：高。
- **典型场景**：异常日志打印请求体，搜索索引存入手机号，报表导出候选人明细。
- **当前处理**：审计中心脱敏；人才服务区字段黑名单；导出审批和字段级权限。
- **现在不做**：包含明文字段的公开索引、公开报表、未审批导出。
- **未来进入条件**：每个旁路系统都有字段白名单和自动化测试。

### 风险 5：授权链路不清晰

- **优先级**：高。
- **典型场景**：签到或投递后默认入库，没有独立同意或撤回入口。
- **当前处理**：同意写 `candidate_consent`，当前状态写 `candidate_control_profile`，发布快照撤回后下线。
- **现在不做**：把签到成功等同于授权入库。
- **未来进入条件**：每个新用途都明确告知目的、范围、版本、撤回方式和审计证据。

### 风险 6：联系解锁与会员商业化

- **优先级**：高。
- **典型场景**：企业付费后直接查看手机号、邮箱、微信。
- **当前处理**：不做真实联系解锁和会员商业化。
- **现在不做**：套餐、支付、消费、权益扣减、批量联系方式导出。
- **未来进入条件**：权益模型、授权状态、字段级权限、审批或候选人授权、审计和支付合规全部明确。

### 风险 7：真实消息触达

- **优先级**：中高。
- **典型场景**：短信、微信、公众号、小程序、App 推送求职状态或营销消息。
- **当前处理**：只规划站内通知入口。
- **现在不做**：真实短信、微信、小程序、App 推送。
- **未来进入条件**：模板、退订、频控、隐私告知、供应商合同和失败补偿机制明确。

### 风险 8：地图、视频、直播

- **优先级**：中高。
- **典型场景**：地图定位、直播招聘、视频简历或企业视频宣传。
- **当前处理**：只保留“更多服务”占位。
- **现在不做**：真实地图 SDK、视频上传、直播间、实时互动。
- **未来进入条件**：供应商、内容审核、版权、成本、隐私和风控方案明确。

### 风险 9：模型/AI 使用原始候选人数据

- **优先级**：高。
- **典型场景**：把完整简历、联系方式、聊天记录发送给模型做推荐或评分。
- **当前处理**：不做黑盒 AI 决策；推荐先走规则和可解释配置。
- **现在不做**：模型直接处理原始候选人数据。
- **未来进入条件**：字段裁剪、脱敏、最小必要、供应商合规、人工复核和审计可追踪。

### 风险 10：先全功能后补边界

- **优先级**：最高。
- **典型场景**：为了快速接近 74cms 效果，先做简历库、会员、支付、短信、地图、直播，再回头补合规。
- **当前处理**：二期逐 Step 规划，后续逐 Prompt 实现。
- **现在不做**：跨 Step 偷跑。
- **未来进入条件**：蓝图和提示词明确允许，并且边界、测试和审计同步落地。

## 6. 上线前置清单

任何涉及候选人数据、高风险企业操作或外部能力的功能，进入开发前必须回答：

- 数据来源是否合法、可证明吗？
- 是否有单独告知和同意？
- 撤回后如何联动下线、删除或隐藏？
- 字段白名单是什么？
- 是否会进入公开页面、缓存、搜索、日志、报表或导出？
- 哪些角色可访问？
- 数据域如何限制本人/本企业/运营/审计？
- 是否需要幂等键？
- 哪些动作写 `audit_log`？
- 哪些字段读取写 `field_access_log`？
- 是否存在 IDOR/BOLA 风险？
- 是否需要外部法律或合规评审？

如果这些问题没有答案，该功能只能停留在文档占位或风险池，不进入开发。

## 9. 评审结论

当前建议结论：

1. **二期可以继续向完整人才网体验演进**，但先补公开门户、搜索、转化、求职者中心、企业中心、运营后台，而不是先做商业化和外部能力。
2. **高风险功能不是不能做，而是不能现在无边界地做**。这些功能需要进入风险池，等数据地图、合规方案、权限模型、审计方案和自动化测试齐备后再排期。
3. **LocalTalent 的差异化不是复制公共简历库**，而是在合规底座上做地方招聘闭环：职位公开、求职者投递、企业处理、同意/撤回、发布快照、导出审批、审计可追溯。
4. **后续所有 Step 和 Prompt 都必须区分“当前可做功能”和“未来评估功能”**，这样才能既接近 74cms 的完整感，又不把高风险能力提前引爆。

最终执行原则：

```text
功能可以规划完整；
高风险可以先不做；
以后要做时，先写规避方案，再进入 Prompt。
```
